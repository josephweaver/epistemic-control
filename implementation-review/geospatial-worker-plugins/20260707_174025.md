## Intended Design

The slice objective is to implement `crop_by_polygons`, a GDAL-backed operation that chops one or more rasters into smaller raster pieces using vector polygon features or their bounding boxes. The slice explicitly accepts a bbox-only first implementation if `cutline` is rejected clearly.

Explicit constraints from the OS:
- no pair counting, aggregation, polygonization, real CDL/Yan/Roy data, or implicit reprojection;
- feature count must be bounded;
- unsafe feature IDs must not become raw output paths;
- CRS mismatch must fail clearly unless reprojection is implemented;
- writes must stay under the declared artifact output directory;
- default tests must not require GDAL, while GDAL tests run in the GDAL path.

## Object / Transition Map

`CropByPolygonsRequest` transitions from JSON bytes to a validated in-memory request when the envelope has API version `goet.geospatial/v1alpha1`, kind `GeospatialOperationRequest`, operation `crop_by_polygons`, at least one raster, a polygon layer/id field, safe outputs, mode `bbox`, and `max_features > 0`, by `ParseCropByPolygonsRequest`, while preserving the invariant that `manifest_json` is inside `output_directory`.

Vector layer metadata transitions from external dataset state to bounded feature metadata when `ogrinfo -json -so` can read the declared layer and `featureCount <= max_features`, by `inspectCropVectorSummary`, while preserving the invariant that no crop files are written before the feature-count guard passes.

Each raster/polygon pair transitions from source raster plus feature bbox to a cropped GeoTIFF artifact when raster CRS EPSG matches vector EPSG, by `runGDALBBoxCrop`, while preserving the invariant that output paths are artifact-root relative paths under `output_directory`.

The crop artifact set transitions from individual crop files to a deterministic manifest when all crops have metadata, by `cropWriteJSONFile`, while recording source raster, feature ID, crop bounds, output path, pixel dimensions, and mode.

## Execution Review

The CLI first reads the request file and unmarshals only the `operation` field. This matters because `crop_by_polygons` uses `inputs.rasters` as an array and `inputs.polygons` as an object, which cannot safely unmarshal into the older `OperationRequest.Inputs map[string]InputSpec`.

For `crop_by_polygons`, `ExecuteCropByPolygons` parses the raw JSON and validates mode, outputs, feature limit, raster inputs, polygon layer, and id field. Unsupported `cutline` fails with a direct message. `mask_to_polygon=true` also fails in bbox mode.

Before writing crop files, the executor runs `ogrinfo -json -so` to read vector CRS and feature count. If the declared feature count exceeds `max_features`, execution stops before writing the manifest or crop rasters.

After the guard passes, the executor runs `ogrinfo -json -features`, extracts feature IDs from the configured id field, computes bboxes by walking GeoJSON coordinate arrays, and crops each sorted raster/feature pair with `gdalwarp -te`. It then reads output raster metadata through existing raster metadata collection and appends one `CropManifestPiece` per crop.

Output filenames are generated as `<output_directory>/<safe raster name>/<index>-<safe feature id>-<hash>/crop.tif`. Unsafe path characters are replaced before becoming path segments, and every generated artifact path is resolved through `cropArtifactPath`.

## Concrete Trace

Given one raster named `cdl`, one vector layer `regions`, and features `west/unsafe` and `northeast`, parsing normalizes `output_directory` from `cropped_rasters/` to `cropped_rasters`.

`inspectCropVectorSummary` reads the vector layer and obtains EPSG 5070 and feature count 2. The raster metadata read obtains EPSG 5070, so CRS validation passes.

The feature `west/unsafe` has bbox `[0,0,60,120]`. Its output path becomes a safe path like `cropped_rasters/cdl/001-west_unsafe-<hash>/crop.tif`, not a nested `west/unsafe` path. `gdalwarp -te 0 0 60 120` writes a 2 by 4 crop for the 4 by 4 fixture raster. The manifest records the raw feature ID, crop bounds, output path, dimensions, and mode `bbox`.

If the vector CRS were EPSG 4326 while the raster CRS were EPSG 5070, execution would stop with a CRS mismatch error before crop outputs are written.

## Risks And Maintenance Notes

The vector feature reader depends on `ogrinfo -json` output including GeoJSON-like `features` and `geometryFields`. This is validated against GDAL 3.8.4 in WSL, but older GDAL JSON variants may need compatibility work.

`bbox` mode uses feature coordinate bounds directly and requires matching EPSG codes. It does not inspect axis-order subtleties beyond the EPSG match. This is acceptable for this slice because reprojection and broader vector semantics are out of scope.

The feature-count guard uses `ogrinfo -so` before the feature read, so crop output explosion is bounded before writes. It still relies on the vector driver reporting feature count accurately.

## Final EC Summary

What to understand before moving on:
- `crop_by_polygons` owns a custom request parser because its JSON input shape differs from the generic operation contract.
- `bbox` is implemented; `cutline` and `mask_to_polygon` are rejected.
- CRS equality is enforced by EPSG match before any crop is written.
- Feature IDs may appear raw in the manifest, but output paths use sanitized hashed segments.
- Crop artifacts and manifest paths are constrained under `output_directory`.
