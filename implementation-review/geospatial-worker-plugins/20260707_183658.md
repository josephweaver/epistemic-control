## 1. Intended Design

The OS objective is to implement categorical zonal counts where raster values are aggregated by vector polygon features.

The OS concept decision requires this transition:

```text
vector polygons -> temporary aligned zone-id raster -> pair count(zone_id, raster_value)
```

Explicit constraints and acceptance criteria from the OS:

- tiny polygon/raster fixture produces known categorical counts;
- output rows are sorted by `polygon_id,raster_value`;
- per-polygon proportions sum to 1.0 within documented rounding;
- `all_touched` is explicit in request and metadata;
- nodata behavior is explicit and tested;
- CRS mismatch fails clearly unless reprojection is implemented;
- temporary zone rasters are under safe temp/artifact paths and cleaned up or declared;
- counting reuses or parallels chunked counting logic;
- default tests do not require GDAL; GDAL tests run in the GDAL path.

Out of scope remains continuous zonal statistics, weighted overlap fractions, topology repair, real CDL/Yan/Roy data, and Parquet output.

## 2. Object / Transition Map

System sentence:

```text
Polygon features transition from vector geometries with string polygon IDs
to sorted polygon/raster-value count rows
when the value raster and polygon layer share an EPSG CRS and the request declares all_touched and include_value_nodata,
by rasterizing polygons to a temporary uint16 zone raster and running the existing chunked pair counter,
while preserving artifact-root confinement, explicit nodata policy, deterministic row order, and six-decimal proportion formatting.
```

Persistent data objects:

- request JSON with `operation=aggregate_by_polygons`;
- output CSV `polygon_id,raster_value,count,proportion`;
- output metadata JSON;
- temporary zone GeoJSON and zone GeoTIFF under an artifact-root temp directory.

In-memory data objects:

- `AggregateByPolygonsRequest`;
- `aggregatePolygonFeature`;
- `RasterMetadata`;
- `PairCountRow`;
- `AggregatePolygonCountRow`;
- `AggregateByPolygonsMetadata`.

Config objects:

- `options.all_touched`;
- `options.include_value_nodata`;
- optional `options.chunk_rows`;
- optional `options.categorical`, rejected when false.

Preconditions:

- value raster path, band, and nodata are valid;
- polygon path, layer, and id field are valid;
- `all_touched` and `include_value_nodata` are present;
- value raster dtype is `UInt16`;
- value raster and polygon layer resolve to the same EPSG code;
- feature count fits the uint16 zone-id range.

Invalid transitions:

- missing explicit policy fails in `ParseAggregateByPolygonsRequest`;
- CRS mismatch fails in `executeAggregateByPolygonsGDAL`;
- unsafe output paths fail before execution;
- temporary path escape fails in `createAggregateTempDir`.

## 4. Execution Order Review

1. CLI dispatch

`cmd/goet-geospatial/main.go` recognizes `geospatial.OperationAggregateByPolygons` and calls `ExecuteAggregateByPolygons`. The request enters as raw JSON and the output artifact root is `filepath.Dir(responsePath)`.

2. Request parsing

`ParseAggregateByPolygonsRequest` decodes strict JSON, validates envelope fields, requires `value_raster` and `polygons`, validates artifact-relative outputs, rejects `categorical=false`, and requires explicit `all_touched` plus `include_value_nodata`.

3. Raster/vector evidence checks

`executeAggregateByPolygonsGDAL` reads value raster metadata, requires CRS evidence and EPSG, reads polygon layer summary, and rejects EPSG mismatch. No reprojection is attempted.

4. Zone raster creation

`readAggregatePolygonFeatures` assigns sequential uint16 zone IDs to features while retaining polygon IDs. `writeAggregateZoneVector` writes a temporary GeoJSON zone vector. `rasterizeAggregateZones` runs `gdal_rasterize` aligned to the value raster grid. The `-at` flag is added only when `all_touched=true`.

5. Chunked counting

The zone raster and value raster are passed to `executeRasterPairValueCountsImpl` as separate uint16 rasters with `RequireAlignedGrid=true` and the requested `ChunkRows`. This reuses the existing GDAL chunk loop in `pair_counts_gdal.go`.

6. Output construction

`aggregateRowsFromPairCounts` maps zone IDs back to polygon IDs, computes per-polygon proportions, and sorts by `polygon_id,raster_value`. `writeAggregatePolygonCountsCSV` writes six-decimal proportions. Metadata records inclusion policy, nodata policy, pair count metadata, source evidence, GDAL version, and temp artifact cleanup policy.

## 6. Concrete Trace

Example test input:

- 3x3 `UInt16` value raster with rows `1 2 9 / 1 2 9 / 0 2 9`;
- one polygon with `field_id=A001`;
- polygon CRS `EPSG:5070`;
- raster CRS `EPSG:5070`;
- `all_touched=true`;
- `include_value_nodata=false`;
- `chunk_rows=1`.

Trace:

1. Parser creates `AggregateByPolygonsRequest`.
2. GDAL metadata confirms raster EPSG 5070 and dtype `UInt16`.
3. OGR metadata confirms polygon EPSG 5070.
4. The feature receives zone ID 1.
5. `gdal_rasterize -at` creates a uint16 zone raster aligned to the 3x3 raster grid.
6. The existing pair counter reads one row at a time because `chunk_rows=1`.
7. Zone nodata skips 3 outside-polygon pixels.
8. Value nodata skips 1 inside-polygon pixel.
9. Pair rows become `(zone 1, value 1, count 2)` and `(zone 1, value 2, count 3)`.
10. The zone ID maps back to `A001`.
11. CSV rows are `A001,1,2,0.400000` and `A001,2,3,0.600000`.

If the polygon CRS is `EPSG:4326`, execution stops before rasterization with a CRS mismatch error and no counts CSV is written.

## 8. Risks

GeoJSON CRS handling:

- The implementation writes a temporary GeoJSON with a CRS object. WSL GDAL accepted this in tests.
- Risk: future GDAL versions may warn about or handle GeoJSON CRS metadata differently.
- Action: acceptable for this slice because the rasterized fixture tests passed under GDAL in WSL.

Overlapping polygon behavior:

- GDAL rasterization order determines the final zone ID where polygons overlap.
- The OS explicitly excludes topology repair and weighted overlap, so this is deferred.

Uint16 zone limit:

- Zone IDs use uint16 to reuse the pair counter.
- More than 65535 polygon features fail clearly.
- This matches reuse of current pair-count machinery but may need a wider field-id path later.

## 10. Final EC Summary

What I should understand before moving on:

- Polygon IDs are not counted directly. They are mapped to temporary uint16 zone IDs first.
- The zone raster is aligned to the value raster grid, then the existing raster pair counter does the heavy counting.
- `all_touched` changes rasterization, not pair counting.
- Value nodata is either skipped or counted based on `include_value_nodata`.
- CRS mismatch fails; there is no reprojection in this slice.
