## Intended Design

The slice objective is to define a stable external JSON request/result contract for GOET geospatial worker operations and add a standalone `goet-geospatial` CLI skeleton that validates the contract without real raster transformations or GDAL dependencies.

The slice acceptance criteria require:

- `goet-geospatial --request valid.json --response result.json` writes a deterministic result for a supported no-op/validate operation.
- Invalid `api_version`, invalid `kind`, missing/unsupported `operation`, and unsafe output artifact paths are rejected.
- Output artifact paths reject absolute paths, Windows drive paths, backslashes, empty paths, and `..` traversal.
- Input paths are accepted as worker-local paths and are not treated as controller-openable.
- `go test ./...` passes without GDAL.
- The geospatial README tracker and OS status are updated.

## Object / Transition Map

`OperationRequest` transitions from untrusted JSON bytes to a validated in-memory request when `api_version`, `kind`, `operation`, input fields, and output artifact paths satisfy the contract, by `json.Unmarshal` followed by `OperationRequest.Validate`, while preserving the invariant that output paths are safe slash-relative artifact paths and input paths remain opaque worker-local strings.

`OperationResult` transitions from absent response file to deterministic JSON result file when request validation succeeds, by `NewValidationResult` and `json.MarshalIndent` in `cmd/goet-geospatial.run`, while preserving the invariant that this slice does not perform raster reads, GDAL calls, artifact promotion, or controller integration.

Invalid request data transitions from untrusted JSON bytes to a returned CLI error when any validation precondition fails, by `OperationRequest.Validate`, while preserving the invariant that the response file is not written for invalid requests.

## Transition Review

Request parsing begins in `cmd/goet-geospatial.run`. The CLI requires `--request` and `--response`, reads the request file with `os.ReadFile`, unmarshals into `geospatial.OperationRequest`, and calls `request.Validate`.

Request validation is implemented in `internal/geospatial/validation.go`. It checks exact `api_version`, exact `kind`, and operation membership. For this slice, supported no-op operations are `validate` and `version`. It checks input names and non-empty input paths but does not reject absolute or Windows-style input paths. That matches the acceptance criterion that input paths are worker-local and not assumed controller-openable.

Output artifact path validation delegates to `internal/model.ValidateArtifactRelativePath`, the existing Data Assets validator. This preserves the shared invariant for slash-relative artifact paths and rejects empty, absolute, drive-qualified, backslash, dot, traversal, and unclean paths.

Result writing is implemented in `NewValidationResult` plus `json.MarshalIndent`. The result has stable struct field order and fixed empty `artifacts`, `summary`, and `warnings` values. No GDAL package, cgo dependency, or raster read path is introduced.

## Concrete Trace

Input file:

```json
{
  "api_version": "goet.geospatial/v1alpha1",
  "kind": "GeospatialOperationRequest",
  "operation": "validate",
  "inputs": {
    "field_raster": {
      "path": "/worker/cache/yanroy/tile_001/fields.tif",
      "band": 1,
      "nodata": 0
    }
  },
  "outputs": {
    "metadata_json": "metadata/result.json"
  },
  "options": {}
}
```

The CLI parses flags, reads the file, unmarshals the JSON, validates the envelope and output path, constructs `OperationResult`, and writes:

```json
{
  "api_version": "goet.geospatial/v1alpha1",
  "kind": "GeospatialOperationResult",
  "operation": "validate",
  "artifacts": [],
  "summary": {},
  "warnings": []
}
```

If `outputs.metadata_json` is changed to `../escape.json`, validation fails at the output path check and the CLI returns an error without writing the response file.

## Test Coverage

Covered:

- `TestOperationRequestValidateAcceptsWorkerLocalInputPaths` proves absolute Unix and Windows-style input paths are accepted as opaque worker-local strings.
- `TestOperationRequestValidateRejectsInvalidEnvelopeAndOperation` proves invalid API version, invalid kind, missing operation, and unsupported operation are rejected.
- `TestOperationRequestValidateRejectsUnsafeOutputPaths` proves empty, absolute, Windows drive, backslash, and traversal output paths are rejected.
- `TestRunWritesDeterministicValidationResult` proves the CLI writes the exact expected result JSON for a valid request.
- `TestRunRejectsInvalidRequest` proves invalid output paths fail before response writing.

Not covered:

- A `version` operation CLI example is not separately tested. It uses the same validation and result-writing path as `validate`, so this is a low-risk gap.
- Unknown JSON fields are not rejected. The OS does not require strict decoding, so this does not block the slice.

## Risks and Ambiguities

The operation enum currently supports only `validate` and `version`. The target-state examples use `raster_pair_value_counts`, but the OS explicitly allows this slice to implement only a `version` or `validate` no-op operation. Later slices should add real operation names when they implement behavior.

The output contract uses `map[string]string` for request outputs, matching the examples where output names map to artifact-relative paths. If later operations need structured output declarations, a later slice should version or extend this field deliberately.

The CLI overwrites an existing response file with `os.WriteFile`. The OS requires deterministic writing but does not specify overwrite policy. This can remain as-is unless a later worker integration slice defines atomic or fail-if-exists response semantics.

## What I Should Understand Before Moving On

- `OperationRequest` is the external JSON boundary for geospatial plugin calls.
- `OperationRequest.Validate` enforces envelope, operation, input, and output-path preconditions.
- Input paths are intentionally opaque worker-local strings.
- Output paths must reuse the Data Assets slash-relative artifact path invariant.
- `goet-geospatial` currently validates and writes a compact no-op result; it does not transform rasters.

## Questions I Should Answer

- Should the later real operation enum include `raster_pair_value_counts` only when the operation is implemented, or should future docs reserve enum names earlier?
- Should response writing eventually be atomic when integrated into worker execution?
- Should future request decoding reject unknown fields before this becomes a public compatibility promise?
