## Intended Design

Strategic Concept: `docs/concepts/geospatial-worker-plugins/README.md`

Operational Slice objective: implement the core categorical raster pair count:

`field_id raster + crop_id raster -> field_id,crop_id,count`

Explicit slice constraints and acceptance points:

- support `raster_pair_value_counts` in `goet-geospatial`,
- support separate-raster mode and stacked-raster mode,
- fail before counting when separate rasters are misaligned,
- resolve missing nodata from band metadata when available,
- skip field nodata,
- skip value nodata unless `include_value_nodata=true`,
- use chunked/windowed reads instead of loading the full raster,
- emit deterministic `field_id,crop_id,count` rows sorted ascending,
- use `uint64` counts and reject silent overflow,
- keep default tests GDAL-free and run GDAL tests behind `-tags gdal`.

## Transition Review

### 1. Request parsing

`ParseRasterPairValueCountsRequest` reads raw JSON rather than going through the older `OperationRequest.Validate` path.

Why:
- stacked-raster mode needs `field_band`, `value_band`, `field_nodata`, and `value_nodata`, which are not represented by the older `InputSpec`.

What it enforces:
- exact operation name,
- `counts_csv` and `metadata_json` outputs,
- `require_aligned_grid` may only be true,
- `chunk_rows > 0`,
- `field_dtype` and `value_dtype` are only `uint16`,
- inputs use either separate mode or stacked mode, not both.

Failure behavior:
- returns an error before any artifact path is created.

Evidence:
- `internal/geospatial/pair_counts.go:176-280`

### 2. GDAL metadata and preconditions

`executeRasterPairValueCountsGDAL` resolves band metadata before counting.

What it checks:
- source band exists,
- source band dtype is `UInt16`,
- nodata is taken from the request when present or the band metadata otherwise,
- separate rasters must match width, height, CRS, transform, and alignment through `GridsEqual`.

Failure behavior:
- misalignment or missing nodata fails before output files are written.

Evidence:
- `internal/geospatial/pair_counts_gdal.go:23-121`
- `internal/geospatial/pair_counts_gdal_test.go:149-243`

### 3. Chunked counting

The counting loop advances by `chunk_rows` and calls `gdal_translate -of XYZ -srcwin ... /vsistdout/` per band per window.

What object enters:
- raster band path, band number, window origin, window size.

What state leaves:
- one `[]uint16` chunk for the field band and one `[]uint16` chunk for the value band.

Invariant:
- only one chunk is resident per band at a time; the implementation never reads the full raster into memory.

Evidence:
- `internal/geospatial/pair_counts_gdal.go:46-69`
- `internal/geospatial/pair_counts_gdal.go:124-169`

### 4. Pair aggregation and deterministic output

`pairCountAccumulator.AddChunk` applies the slice semantics in order:

1. skip field nodata,
2. skip value nodata unless `include_value_nodata`,
3. increment a packed `uint32` pair key,
4. reject `math.MaxUint64` overflow,
5. track distinct fields, values, and pairs.

`Rows()` then sorts by `field_id` ascending and `crop_id` ascending before CSV emission.

Evidence:
- `internal/geospatial/pair_counts.go:341-400`
- `internal/geospatial/pair_counts.go:402-445`

### 5. CLI dispatch

`cmd/goet-geospatial/main.go` now special-cases `raster_pair_value_counts` before the older generic request validator path.

Why:
- this slice needs raw JSON parsing for stacked-mode fields not covered by the legacy request struct.

Evidence:
- `cmd/goet-geospatial/main.go:49-55`
- `cmd/goet-geospatial/main_gdal_test.go:229-278`

## Concrete Trace

Example: separate mode request with:

- `field_raster`: values `1 1 / 2 0`
- `value_raster`: values `5 0 / 5 9`
- both rasters use nodata `0`
- `chunk_rows=1`

Trace:

1. `run()` reads the request JSON and branches on `request.Operation == "raster_pair_value_counts"`.
2. `ExecuteRasterPairValueCounts` parses the raw JSON into `RasterPairValueCountsRequest`.
3. `executeRasterPairValueCountsGDAL` reads metadata for both rasters, confirms `UInt16`, resolves nodata, and confirms aligned grids.
4. Window 0 reads row 0 from both rasters:
   - `(1,5)` is counted,
   - `(1,0)` is skipped as value nodata.
5. Window 1 reads row 1 from both rasters:
   - `(2,5)` is counted,
   - `(0,9)` is skipped as field nodata.
6. `pairCountAccumulator.Rows()` returns sorted rows:
   - `1,5,1`
   - `2,5,1`
7. `ExecuteRasterPairValueCounts` writes:
   - `counts/field_crop.csv`
   - `counts/field_crop.metadata.json`
8. `run()` writes the compact result JSON.

Evidence:
- `cmd/goet-geospatial/main_gdal_test.go:229-278`

## Hidden Coupling and Risk

1. CLI dispatch now has a one-off raw JSON path.
   - Where: `cmd/goet-geospatial/main.go:49-55`
   - Risk: future geospatial operations with richer nested inputs may repeat this pattern until the shared contract type is widened.
   - Action: defer; this slice stays coherent, but the contract layer may need a later redesign.

2. Window reads use `gdal_translate -of XYZ` per chunk.
   - Where: `internal/geospatial/pair_counts_gdal.go:124-169`
   - Risk: this is memory-safe and simple, but subprocess and text-parsing overhead may be expensive on large rasters.
   - Action: defer; the slice required chunking and correctness, not a final performance path.

3. The packed `uint32` key depends on the `uint16` invariant.
   - Where: `internal/geospatial/pair_counts.go:357`
   - Risk: any future widening of field or crop dtype must revisit the key scheme.
   - Action: document by keeping the dtype gate strict in this slice.
