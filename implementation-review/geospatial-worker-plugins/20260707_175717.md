# Implementation Review: Geospatial Worker Plugins OS-008

## Intended Design

Strategic Concept: `docs/concepts/geospatial-worker-plugins/README.md`.
Operational Slice: `docs/concepts/geospatial-worker-plugins/008-polygonize-raster.md`.
Objective: add optional GDAL-backed `polygonize_raster` for converting contiguous categorical raster regions into vector polygons. This is explicitly not part of the CDL/Yan/Roy raster-pair count path.
HCI: EC-3 / operational slice / files(6)+test+doc.

Explicit constraints: fixture-sized tests, explicit 4/8 connectivity, nodata exclusion, value field, `max_features` guardrail, metadata with source evidence, artifact-path confinement, and default tests that do not require GDAL.

## Object / Transition Map

`PolygonizeRasterRequest` transitions from JSON request bytes to validated in-memory request when the envelope, raster input, declared artifact outputs, value field, connectivity, and max feature limit are valid, by `ParsePolygonizeRasterRequest`, while preserving artifact-relative output paths and bounded option values.

`Raster categorical cells` transition from raster band values to GPKG polygon features when GDAL utilities and a GDAL-enabled Go metadata build are available, by `ExecutePolygonizeRaster`, `preparePolygonizeRasterSource`, and `runGDALPolygonize`, while preserving declared nodata exclusion and the requested 4/8 connectivity policy.

`PolygonizeRasterMetadata` transitions from absent to written JSON metadata when polygonization succeeds and feature count is at or below `max_features`, by `writeJSONFile`, while preserving feature count, value field, connectivity, nodata policy, source raster evidence, vector evidence, GDAL version, and warnings.

Invalid transition: if output feature count exceeds `max_features`, the vector output is removed and metadata is not written.

## Execution Review

The CLI dispatcher in `cmd/goet-geospatial/main.go` routes `OperationPolygonizeRaster` to `ExecutePolygonizeRaster`.

`ParsePolygonizeRasterRequest` validates the stable request shape: `inputs.raster`, `outputs.vector`, `outputs.metadata_json`, `options.value_field`, `options.connectivity`, and `options.max_features`.

`ExecutePolygonizeRaster` resolves artifact-root-confined output paths using the existing `artifactPath` helper, collects raster metadata, prepares nodata-aware source input when nodata is declared, runs GDAL polygonize, counts output features with `ogrinfo`, rejects over-limit output, and writes metadata.

The persistent artifacts are the declared vector and metadata paths. A temporary raster source can be created for declared nodata handling and is removed by deferred cleanup.

## Concrete Trace

Input request: raster `classes.tif`, band `1`, nodata `0`, vector `polygonized_classes.gpkg`, metadata `polygonized_classes.metadata.json`, value field `value`, connectivity `4`, max features `10`.

`ParsePolygonizeRasterRequest` produces `PolygonizeRasterRequest`.
`ExecutePolygonizeRaster` collects `RasterMetadata`, prepares a single-band nodata-aware source with `gdal_translate`, runs `gdal_polygonize.py`, counts GPKG features via `ogrinfo`, sees 3 features, writes metadata, and returns an `OperationResult` with vector and metadata artifacts.

If the same raster is run with `max_features: 2`, the feature count check fails, the vector is removed, and the operation returns an error.

## Test Coverage

Covered:
- `TestParsePolygonizeRasterRequestAppliesDefaultsAndValidatesOptions`: valid parse, layer name, value field, connectivity, max feature values.
- `TestParsePolygonizeRasterRequestRejectsInvalidGuards`: missing raster, bad connectivity, zero max features, unsafe output path, invalid field name.
- `TestExecutePolygonizeRasterWritesVectorAndMetadata`: tiny raster polygonizes into values 1/2/3, nodata 0 excluded, metadata written.
- `TestExecutePolygonizeRasterConnectivityChangesDiagonalRegions`: 8-connectivity merges diagonal same-value cells.
- `TestExecutePolygonizeRasterEnforcesMaxFeatures`: post-run feature count guardrail rejects and removes vector output.
- `TestRunPolygonizeRasterWritesVectorAndMetadataArtifacts`: CLI dispatcher writes declared artifacts.

Not covered:
- A raster without any declared or band nodata. This does not block the slice because the acceptance criteria focus on nodata exclusion by default when nodata is declared.
- `gdal_polygonize` binary name fallback without `.py`. Low risk; production tries both names.

## Risks

`max_features` is enforced after GDAL polygonization, not before. This matches the OS allowance for a post-run count limit, but large rasters can still create large temporary output before rejection. The metadata warning records this.

The implementation reuses crop-vector `ogrinfo` helpers. This is a small same-package coupling; it avoids another vector-info file for this slice.

## Final EC Summary

What I should understand before moving on:
- `polygonize_raster` is optional/export/debug behavior, not a CDL/Yan/Roy count dependency.
- The value field stores source raster class values.
- Declared nodata is excluded through GDAL mask behavior.
- Connectivity is explicit and limited to 4 or 8.
- `max_features` is a post-run guardrail.

Questions I should answer:
- Is post-run `max_features` sufficient for expected fixture and debug use?
- Should a later slice add a true preflight bound for very large rasters?
- Should vector-info helpers become their own file when another vector operation needs them?
