## 1. Intended Design

The OS objective is to add an end-to-end fixture demonstration that proves CDL/Yan/Roy-style raster processing through geospatial plugin operations on tiny local inputs. Required flow: metadata check, optional alignment, pair counting, and a downstream summary with total and proportion fields, without polygonization.

## 2. Transition Model

`fixture field raster + cdl raster` transition to `pair-count + summary artifacts` when all three are small local `uint16` rasters with declared nodata and aligned/invariant-safe grids.

By operations:
- `raster_info` validates input metadata,
- `align_to_grid` produces an aligned value raster (even when already aligned),
- `raster_pair_value_counts` produces deterministic counts rows,
- summary script computes `total_count` and `proportion` and dominant flags.

Invariants:
- inputs remain worker-local paths in the test (`/work/...`),
- raster values remain `uint16` with explicit nodata policies,
- no polygonization request is executed in this path.

Failure transitions:
- any count mismatch aborts the test,
- missing expected operation fields aborts the test,
- any `"operation": "polygonize_raster"` in responses aborts the test.

## 4. Transition Order

1. Shell fixture setup creates 3x3 field/cdl grids and converts to GeoTIFF.
2. Raster metadata check (`raster_info`) confirms GDAL operation and raster artifacts.
3. Alignment request (`align_to_grid`) creates `aligned/cdl.tif`.
4. Pair count request (`raster_pair_value_counts`) writes deterministic counts and metadata.
5. A small Python step computes and writes `results/summary.csv`.
6. The script asserts exact expected counts/summary and checks no polygonize operation token.
