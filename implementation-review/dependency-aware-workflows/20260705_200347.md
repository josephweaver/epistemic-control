## Implementation Review: Dependency-Aware Workflows / 006 Record Terminal Work Item State

### Execution Order

1. In `cmd/controller/main.go` (`completePersistedWorkHandler` and `failPersistedWorkHandler`), the existing attempt-commit path now captures the completed or failed `WorkItemRecord` from persistence before responding and calls the dependency state updater.
2. In `cmd/controller/main.go` (`recordWorkItemDependencyTerminal`), the new helper resolves the work item from the run-scoped store and delegates to dependency state persistence.
3. In `cmd/controller/workflow_dependency_store.go`, the new `RecordWorkItemTerminalState` entry point validates inputs, no-ops when no dependency plan exists, and persists terminal-state transitions only when the membership actually changes.
4. In `cmd/controller/workflow_dependency_store.go`, helper functions (`updateDependencyPlanForWorkItemTerminal`, `findDependencyWorkItem`, `updateStepStateFromWorkItems`, `updateStageStateFromSteps`, `updateWorkflowStateFromStages`) now recalculate step and stage terminality from membership state.
5. In `cmd/controller/workflow_dependency_store_test.go`, added unit tests for completion-driven step/stage advancement, duplicate terminal-report idempotence, and failure propagation.
6. In `cmd/controller/main_test.go`, added integration tests for terminal report handlers that claim persisted work and assert dependency state transitions in submission context.
7. Updated `docs/concepts/dependency-aware-workflows/006-record-terminal-work-item-state.md` status to `Implemented`.
8. Updated `PROJECT_STATE.md` to record OS-006 completion.

### Public/Public-Facing Changes

- `cmd/controller/main.go` (`completePersistedWorkHandler`)
  - What it does: after persisting a completion, it sets membership state for the completed work item to `completed` or `skipped` and updates dependency model state.
  - Why it exists: to keep worker-facing attempt terminaling and dependency-state bookkeeping in sync.
  - Assumption: `CompleteAttempt` returns the authoritative work-item ID tied to the submission plan.
  - What could break: if attempt records are missing/mismatched, dependency updates are skipped or error out.

- `cmd/controller/main.go` (`failPersistedWorkHandler`)
  - What it does: after persisting a failure, it sets membership state to `failed` and recomputes owning step/stage state.
  - Why it exists: failure must propagate to dependency state for subsequent decision logic.
  - Assumption: `FailAttempt` returns a resolved `WorkItemID`.
  - What could break: repeated failures can still return success at persistence layer while membership updates must remain idempotent; non-idempotent behavior here would inflate completion state.

- `cmd/controller/main.go` (`recordWorkItemDependencyTerminal`)
  - What it does: centralizes dependency update by looking up the run ID from the work item record and delegating to the store-backed dependency updater.
  - Why it exists: keeps handlers small and avoids duplication.
  - Assumption: work items remain queryable after terminal attempt processing.
  - What could break: store coupling makes this path unavailable in deployments that bypass persisted work items.

- `cmd/controller/workflow_dependency_store.go` (`RecordWorkItemTerminalState`)
  - What it does: updates terminal state for a specific work-item membership, recalculates step/stage state from membership counts, and persists only when needed.
  - Why it exists: gives deterministic, idempotent dependency progression.
  - Assumption: each membership maps uniquely to one plan step.
  - What could break: if duplicate work-item IDs exist in the same plan (which `CreateWorkflowDependencyPlan` and membership checks should prevent), update behavior would be ambiguous.

- `cmd/controller/workflow_dependency_store.go` (`updateStageStateFromSteps`)
  - What it does: converts state per-step to aggregate stage state with failed precedence.
  - Why it exists: stage completion/failure is derived from all child steps.
  - Assumption: failed state should dominate completion.
  - What could break: if future terminal-state shape adds non-terminal intermediate values, this helper may need expansion.

### Concept Match

- Completing one item and incrementally advancing state: implemented.
- Completing all items in a step marks step complete: implemented.
- Completing only some fan-out items keeps step active/incomplete: implemented.
- Completing all steps in stage marks stage complete: implemented.
- Duplicate terminal reports do not double-transition: implemented.
- Failure marks membership + step + stage failure: implemented.
- Failure reflected through existing status model: implemented (terminal attempts contribute to submission status counts; no changes to status API in this slice).
- Downstream stage compilation: unchanged (none added), as required by this slice.

### Concrete Example Path

1. A test creates a submitted workflow run and builds dependency plan with one stage, one step, one membership.
2. Controller creates dependency plan and records membership for `workitem-id`.
3. `ClaimNextWork` returns the persisted work item and attempt.
4. `POST /work/complete` is handled by `completePersistedWorkHandler` and `CompleteAttempt` returns the same `WorkItemID`.
5. `recordWorkItemDependencyTerminal` fetches work item metadata and calls `RecordWorkItemTerminalState`.
6. `updateDependencyPlanForWorkItemTerminal` marks membership as `completed`, recalculates step and stage to `completed`, then persists the plan.
7. Tests read back step/stage state and run submission context to verify persisted terminal `workflow_state`.

### Edge Cases Covered by Tests

- Single-step completion path moves step and stage to completed.
- Multi-item step stays active when only partial completion occurs.
- Duplicate completion for same work item is idempotent.
- Completion path with handler-level assertion verifies persisted submission context state transitions.
- Failure path in handler verifies step and stage both become failed.

### Edge Cases Not Covered

- Duplicate failure reports after a non-idempotent terminal error from persistence layer.
- Mixed terminal states (e.g., one completed and one running, then later failure) for more complex fan-out with already-completed items are not directly asserted.
- Workflow-level `dependency workflow state failed` transition on first failure is not asserted; current helper derives running/completed from stage states only.
- No concurrent handler collision test for two terminal updates racing on same membership.

### Ambiguities / Risk

- `updateWorkflowStateFromStages` currently sets workflow state to running whenever any stage is not completed, even if a stage is failed. Existing status semantics still report failed via terminal attempt counts, but `DependencyState.State` does not currently become failed.
- `workflow_state` persistence currently has no direct linkage to terminal failure status and should be revisited in later slices if explicit dependency-state failure visibility is required.
- The path assumes `WorkItemID` uniqueness and no missing membership in a persisted run; both are currently enforced by existing plan/membership invariants but should be guarded by stronger assertion tests when failure is later terminal for scheduling decisions.

### What I should understand before moving on
- Terminal completion/failure handlers now only affect dependency-plan bookkeeping, not downstream stage compilation.
- Membership lookup is idempotent and no-ops on duplicate terminal transitions.
- Terminal-state status propagation currently happens in `workflow_dependency_store` and is only as rich as current stage/step states.

### Questions I should answer
- Should dependency `workflow_state` reflect failure explicitly in this slice or in the later failure-propagation slice?
- Should duplicate terminal reports be treated as success, conflict, or explicit no-op when attempt IDs mismatch?
- Do we want to persist additional diagnostics when a terminal update cannot find matching membership?
- Should idempotence also include repeated terminal reports with alternate attempt IDs after conflict?
- How should completion of mixed fan-out (`completed + skipped`) be surfaced in any future public status shape?
