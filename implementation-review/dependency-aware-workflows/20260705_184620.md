### Public Changes

- `internal/model/workflow_dependency.go` (new)
  - What it does: defines durable dependency state shapes and typed state enums used by controller storage and APIs.
  - Why it exists: requirement asks for central definition of valid state values and persisted dependency records.
  - Assumption: dependency orchestration is persisted independently from execution state and currently no external API depends on these concrete field names.
  - What could break if wrong: if a single typo in enum validation slips through, later slices that interpret state strings may accept invalid values and mis-schedule work.

- `internal/persistence/store.go` (`UpdateWorkflowRunSubmissionContext`)
  - What it does: updates only `submission_context_json` for an existing `workflow_instances` row.
  - Why it exists: creates a write path without changing schema.
  - Assumption: updating run context is safe for this product path and no other code expects submission context JSON bytes to be immutable.
  - What could break if wrong: this method can overwrite run metadata if call sites pass malformed JSON or stale payload; call sites currently validate JSON and plan integrity before writing.

- `workflowRunSubmissionContext` (`cmd/controller/main.go`)
  - What it does: now carries optional `DependencyState` data when marshaling workflow run records.
  - Why it exists: allows dependency plan state to be persisted in an existing JSON column without schema migration.
  - Assumption: legacy contexts without dependency state will continue to unmarshal successfully.
  - What could break if wrong: if future callers rely on strict `omitempty` assumptions, additional output size could alter JSON snapshots.

- `cmd/controller/workflow_dependency_store.go` (new)
  - What it does: exposes dependency state CRUD-like operations for controller-owned plan state, stores membership state in run context, and returns ordered stage/step projections.
  - Why it exists: implements OS-003 APIs while avoiding wiring into live submission and work assignment behavior.
  - Assumption: caller controls stage/step indices through normalized plans and passes a valid submission ID.
  - What could break if wrong: malformed stored context can cause plan reads to fail hard if validation is strict; this is intentional to prevent latent corrupted state.

- `cmd/controller/workflow_dependency_store_test.go` (new)
  - What it does: asserts behavior and invalid-state handling for deterministic order, duplicate prevention, and readback contracts.
  - Why it exists: requirement asks for direct tests of state-layer behavior without relying on HTTP endpoints.
  - Assumption: minimal behavior-focused coverage is sufficient for this slice.
  - What could break if wrong: hidden errors in integration with admission path may remain because endpoint behavior is intentionally unchanged.

### Concept Match

- `CreateWorkflowDependencyPlan` exists and persists plan state: yes.
- Duplicate stage/step records prevention: yes for duplicate input indices and repeated plan creation.
- `ListWorkflowStages` and `ListWorkflowSteps`: yes, deterministic ordering applied.
- `RecordCompiledWorkItemMembership` and duplicate work-item checks: yes by ID and index.
- `ReadStepState` / `ReadStageState`: yes.
- Invalid states rejected: yes via validation path and JSON decode checks in reads.
- No wiring into `/workflow` or `/work/next`: yes (unchanged code path and tests).

### Concrete Example Path

1. A controller test creates a persisted run (`insertTestPersistenceRunWithStage`).
2. It calls `CreateWorkflowDependencyPlan` with two normalized stages (`stageIndex 1` and `stageIndex 0`, intentionally out of order), each with step metadata.
3. `CreateWorkflowDependencyPlan` validates run existence and no existing plan, builds `WorkflowDependencyPlan`, and persists it using `UpdateWorkflowRunSubmissionContext`.
4. `ListWorkflowStages` returns two stages sorted as `0, 1` regardless of input order.
5. `ListWorkflowSteps` returns steps in stable stage-first then step-index order.
6. `RecordCompiledWorkItemMembership` appends one membership, then rejects duplicates by ID and index.
7. `ReadStepState` and `ReadStageState` return readback state with deterministic, cloned payload.

### Edge Cases Covered by Tests

- Empty/duplicate state strings in model enums.
- Missing run/workflow IDs in model plan validation.
- Duplicate stage creation attempt for same run.
- Deterministic stage and step ordering independent of input order.
- Membership dedupe on work-item ID and index.
- Missing plan returns explicit not-found behavior from list/read methods.
- Invalid stored dependency state is rejected by read/validation path.

### Edge Cases Not Covered

- Race-safe update semantics under concurrent membership recording (no lock around read-modify-write cycle in these APIs).
- Transition semantics for stage/step state changes after initial creation.
- Recovery behavior when stored dependency state bytes are partially corrupted versus fully invalid JSON.
- Behavior when a normalized plan references non-sequential step indexes with missing values.
- Long-running status transitions (active/completed/failed) are not covered because this slice intentionally excludes execution wiring.

### Ambiguities, Coupling, Risks

- `CreateWorkflowDependencyPlan` derives initial readiness from the minimum stage index, not insertion order. This is intentional but should be made explicit in later design docs if behavior is relied upon.
- State lives in `WorkflowRunRecord.SubmissionContextJSON` with a controller-specific `dependency_state` blob. This creates cross-slice coupling to controller admission/storage format.
- Duplicate prevention for stages/steps is currently enforced both by creation-time checks and by validation on reads; this avoids silent corruption but can make recovery fail hard if legacy contexts are damaged.
- Without DB schema migration, dependency state size contributes to single-row context growth; this may become a future limit concern.

### What I should understand before moving on

- Dependency plan state is now persisted as an additive section in `SubmissionContextJSON`; the state APIs are controller-local and do not represent runtime scheduling yet.
- The write path uses an explicit context-update call in persistence; it is not yet part of a transactional flow with work-item inserts.
- Stage index and step index are expected to be integer identifiers from normalized stage planning.
- Duplicate defense is by plan-level idempotence and membership-level dedupe.

### Questions I should answer

- Should readiness for initial stage be derived from smallest stage index or from explicit run metadata?
- Which transitions for stage/step/membership states are required in the next slice, and should they be validated as a DFA?
- Should membership records allow duplicate work-item IDs across retries if same logical item is retried intentionally?
- Do we need migration logic for old runs that contain `submission_context_json` without dependency state?
- What size/complexity limit should we set for dependency context before splitting into dedicated tables?
