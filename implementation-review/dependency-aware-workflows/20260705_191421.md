## Implementation Review: Dependency-aware Workflows / 004 Stamp Work Items With Step Instance Metadata

### Execution Order

1. `submitWorkflowRunToStore` in `cmd/controller/main.go` now normalizes workflow stages after compilation and before persistence work-item insertion.
2. The same function now creates the dependency plan (`CreateWorkflowDependencyPlan`) before inserting run work items when the compiled workflow has any items.
3. `splitCompiledWorkflowByStage` groups compiled items by normalized stage and preserves fan-out generation order while assigning per-step `WorkItemIndex`.
4. `persistenceRecordsFromCompiledStageResults` stamps deterministic runtime metadata into each `model.WorkItem` payload (workflow/step IDs, fingerprints, instance IDs, code version), creates `WorkItemRecord` and `QueuedWorkRecord` objects, and emits membership tuples with `(stageIndex, stepIndex, workItemID, workItemIndex)`.
5. `/workflow` submission loop persists work records, queues them, then stores memberships with `RecordCompiledWorkItemMembership`.
6. `workflow_stage_queue_test.go` adds unit coverage for stage grouping order and deterministic metadata behavior.
7. `docs/concepts/dependency-aware-workflows/004-stamp-work-items-with-step-instance-metadata.md` is marked implemented and `PROJECT_STATE.md` is updated.

### Public/Public-ish Changes

- `cmd/controller/main.go` (`submitWorkflowRunToStore`)
  - What it does: switches staging path from legacy `persistenceRecordsFromCompiledWorkflow` to stage-aware `splitCompiledWorkflowByStage` + `persistenceRecordsFromCompiledStageResults` and `RecordCompiledWorkItemMembership` when there are compiled work items.
  - Why it exists: matches OS-004 requirement to preserve submission-scoped item metadata and stage/step membership during admission.
  - Assumptions: `workflow.NormalizeStages` succeeds for any non-empty workflow being submitted and `CreateWorkflowDependencyPlan` is idempotent only at run granularity.
  - Failure risk: if normalization fails or stage plan diverges from compiled items, submission is rejected and no work is queued; acceptable for deterministic admission control.

- `cmd/controller/workflow_stage_queue.go` (`splitCompiledWorkflowByStage`)
  - What it does: transforms `workflow.CompileResult` into stage-scoped `CompileStageResult` values, including `WorkItemIndex` per step.
  - Why it exists: OS-004 allows only a narrow transformation layer; stage/step metadata should be derived mechanically from already-compiled work.
  - Assumptions: `workflow.WorkflowPlan` contains all compiled `StepID` values and authoritative `StageIndex`/`StepIndex`.
  - Failure risk: unknown/missing step IDs in compile output become hard admission errors; avoids writing corrupted membership and prevents partial queue state.

- `cmd/controller/workflow_stage_queue.go` (`persistenceRecordsFromCompiledStageResults`)
  - What it does: stamps deterministic identifiers and fingerprints in worker payloads and returns queued/item membership records.
  - Why it exists: makes `step_instance_id` deterministic within a submission and stable per logical item; satisfies `step_instance_id` acceptance criteria.
  - Assumptions: deterministic `stepInstanceID := runID + ":step:" + stepIndex` is unique enough for current execution contracts.
  - Failure risk: reusing a different `runID` scheme elsewhere in future code could break cross-system expectations for traceability.

- `cmd/controller/workflow_stage_queue_test.go`
  - What it does: verifies step grouping, step/work item indexing, deterministic ID behavior, and that memberships align 1:1 with persisted items.
  - Why it exists: protects the key OS acceptance criteria that were previously under-specified by existing tests.
  - Assumptions: these invariants remain expected across completion wiring (currently not changed in this slice).
  - Failure risk: future tests need to cover cross-slice behavior (runtime endpoints and dependency-state reads) where these IDs are consumed.

- `cmd/controller/main.go` (`persistenceRecordsFromCompiledWorkflow`)
  - What it does: legacy helper still exists but is no longer used by the `/workflow` admission path.
  - Why it exists: preserving previous behavior surface avoided scope creep and reduced risk in this implementation-only slice.
  - Assumptions: leaving legacy helper dormant is acceptable until next slice or cleanup pass.
  - Failure risk: dead code can diverge from new behavior if future callers accidentally reuse it.

### Concept Match Against OS-004

- Controller helper transforms staged compiled output into queue- and membership-ready data: yes.
- Membership record includes `submission_id`, `stage_index`, `step_index`, `work_item_id`, `work_item_index`: yes via `RecordCompiledWorkItemMembership` writes and stage-results mapping.
- Stable workflow/step identity fields in worker payload: yes (`WorkflowDefinitionID`, `WorkflowInstanceID`, `StepDefinitionID`, `StepFingerprint`, `StepInstanceID`, fingerprints).
- Deterministic `step_instance_id` and submission distinction: yes (`runID:step:<stepIndex>`).
- Work-item order preserved: yes; tests enforce per-step ordering for fan-out results.
- No changes to completion/failure, live status, worker behavior: yes.
- Tests remain narrow and mechanical: yes.

### Concrete Example

Input workflow has plan stages `[extract@index0, transform@index1]` and compile output emits:

1. `extract-a`, `extract-b`, then `transform-a`.

`splitCompiledWorkflowByStage` builds stage buckets from normalized plan; `extract-*` remain in stage 0 and `transform-a` in stage 1.
`persistenceRecordsFromCompiledStageResults` stamps:

- `WorkItemID` values: `run-<id>:extract-a`, `run-<id>:extract-b`, `run-<id>:transform-a`
- `StepInstanceID` for both extract items: `run-<id>:step:0`
- `WorkItemIndex` values for extract: `0`, `1`

Submission then stores all queue records and membership tuples in the same order as stamps, then starts workers using existing scaling flow.

### Edge Cases Covered by Tests

- Step index mapping and stage mapping for compile output with multiple stage boundaries.
- Preservation of fan-out generation order across multiple items for the same step.
- Deterministic step-instance IDs across same step/submission and change across submissions.
- Membership cardinality parity with queued items.
- Correct run-scoped `work_item_id` format in records.

### Edge Cases Not Covered

- Integration-level `/workflow` test asserting `workflow_dependency` membership rows exist after admission.
- Recovery behavior if stage plan and compile result diverge due to concurrent workflow mutation (guarded by single-request compile semantics, but not explicitly tested here).
- Duplicate step IDs in a normalized plan and fallback behavior for corrupted/legacy stored dependency state.
- Concurrent membership writes in a multithreaded admission context (current path is single-threaded at admission).

### Ambiguities / Risks

- `step_instance_id` is deterministic and not randomized; if another system later expects opaque IDs with stronger entropy, migration may be required.
- Legacy helper `persistenceRecordsFromCompiledWorkflow` remains present and could be accidentally reused, reintroducing random `StepInstanceID` behavior.
- `CreateWorkflowDependencyPlan` is now called inside submission path; any future call-order changes to stage persistence or work-queue insertion could require transactional boundaries.

### What I should understand before moving on

- OS-004 changed admission-time behavior only; there is still no completion/failure contract change in this slice.
- Membership is stored in controller dependency state, not exposed via worker payload beyond identifiers already present.
- A deterministic `StepInstanceID = runID:step:<stepIndex>` is now the active trace identity contract.
