### Public Changes

- `CompileStageWorkItem` (`internal/workflow/compile_stage.go`)
  - What it does: carries per-work-item membership metadata from a compiled stage: workflow ID, stage index, step index, step ID, work-item index, and the compiled `model.WorkItem`.
  - Why it exists: stage compilation in future slices needs deterministic membership for queueing and persistence.
  - Assumption: all required metadata derives from the same compile pass and step index ordering.
  - What breaks if wrong: downstream stage activation could map outputs to wrong step/work-item positions.

- `CompileStageResult` (`internal/workflow/compile_stage.go`)
  - What it does: returns compiled stage-wide results, including metadata-rich work items and compiled stage step descriptors.
  - Why it exists: replaces full-workflow compile output for just-in-time stage compilation.
  - Assumption: normalized plan stages are the accepted compile unit before queue insertion.
  - What breaks if wrong: controller code may queue extra stages early or lose ordering guarantees.

- `CompileWorkflowStage` (`internal/workflow/compile_stage.go`)
  - What it does: validates stage index, enforces workflow ID coherence, compiles only the selected normalized stage, and returns deterministic in-stage ordering with duplicate-item detection.
  - Why it exists: direct implementation of OS-002 requirement.
  - Assumption: the input `WorkflowPlan` matches the same workflow definition passed to compilation.
  - What breaks if wrong: duplicate IDs can escape to runtime, or wrong-stage items can be queued.

- New tests (`internal/workflow/compile_stage_test.go`)
  - What they do: validate sequential compilation, parallel-step fan-out order, metadata emission, stage index validation, duplicate ID detection, and compile-context error propagation.
  - Why they exist: keep stage-slice behavior explicit and low-touch.
  - Assumption: test helper fixtures (`testWorkflowResolver`, fan-out expression semantics, and stage normalization) stay stable.
  - What breaks if wrong: regressions can slip into controller stage execution without slice-level tests.

### Concept Match

- Stage-scoped compiler exists: implemented.
- Stage 0/1 sequential behavior: implemented.
- Parallel stage compiles all steps in stage: implemented.
- Ordered output by step then fan-out: implemented.
- Metadata includes step index, step ID, and work-item index: implemented.
- Invalid stage index errors: implemented.
- Intra-stage duplicate ID detection: implemented.
- Public behavior outside slice preserved: implemented because `CompileWorkflowResult` is unchanged.

### Concrete Example

Input:
- Workflow `cdl` with steps `extract` and `transform`.
- Both steps in parallel `batch`.
- `years = [2024,2025,2026]`.
- Normalized plan stage 0 contains both steps.

Flow:
1. `NormalizeStages` returns stage 0 with `extract` then `transform`.
2. `CompileWorkflowStage(resolver, workflow, plan, 0)` compiles `extract` first, then `transform`.
3. Output order becomes `extract-2024`, `extract-2025`, `extract-2026`, `transform-2024`, `transform-2025`, `transform-2026`.
4. Each item carries `WorkItemIndex` and `StepIndex` matching step order and fan-out order.

### Edge Cases Covered by Tests

- Stage 0 and stage 1 for sequential workflow steps are separated.
- Parallel same-stage step ordering across fan-out is deterministic.
- Duplicate IDs within one stage are rejected.
- Negative and out-of-range stage index errors are rejected.
- Step compilation errors include stage + step context in wrapped message.

### Edge Cases Not Covered

- Cross-stage duplicate work-item ID handling remains a future-slice concern (intentional to OS-002).
- Plan/workflow mismatch where stage data references a different workflow object shape is only partially validated via ID check.
- Behavior when a normalized plan contains impossible zero-step stage entries is not separately tested (slice assumes valid `NormalizeStages` output).

### Ambiguities, Coupling, and Risks

- `workflow.ID` mismatch check may reject plans assembled externally with blank or stale IDs.
- `CompileWorkflowStage` relies on `WorkflowPlan` to drive allowed stage boundaries and step ordering.
- This intentionally keeps compatibility by not changing `CompileWorkflowResult`.

### What I should understand before moving on

- Stage compilation is now bounded to normalized plan boundaries and is safe to call repeatedly for JIT workflows.
- Ordering guarantees are explicit: stage order from `NormalizeStages` and fan-out order from existing compiler behavior.
- Duplicate IDs are detected only inside one compiled stage, matching this slice scope.

### Questions I should answer

- Should `CompileWorkflowStage` enforce stronger plan/workflow schema checks beyond IDs?
- Is it acceptable to keep workflow ID validation here, or should controller own that check?
- Should empty `WorkflowPlan` with non-empty `workflow` be treated as a mismatch error vs index error?
- Should duplicate work-item IDs across stages be preemptively blocked in this slice or in later state slices?
- Is step-context error wrapping sufficient for operators, or should it include field-level resolver details?
