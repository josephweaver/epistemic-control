### Public Changes

- `internal/workflow/Step`
  - Added `ParallelWith string `json:"parallel_with,omitempty"``
  - Purpose: allow workflow step authors to request a group label for sibling-parallel execution in later slices.
  - Why: matches OS requirement for `parallel_with` in workflow documents.
  - Assumption: a label value is local to the workflow and optional.
  - Risk: if input is malformed, unknown labels are ignored by compile paths and only normalized in future slice logic.

- `internal/workflow/WorkflowPlan`, `WorkflowStage`, `WorkflowStageStep`
  - Added in `internal/workflow/stage.go`.
  - Purpose: provide a deterministic, dependency-ready stage plan representation.
  - Why: required to replace flat compile behavior and eventually compile one stage at a time.
  - Assumption: zero-based indexing is expected in downstream usage.
  - Risk: if later code assumes only flat `CompileWorkflowResult` behavior, migration bugs can occur until later slices wire both paths.

- `internal/workflow/NormalizeStages(workflow Workflow) (WorkflowPlan, error)`
  - Added in `internal/workflow/stage.go`.
  - Purpose: parse workflow steps into ordered stages with contiguous parallel-grouping.
  - Why: prevents accidental downstream dependency bypass by exposing dependency shape before queueing.
  - Assumptions:
    - Step IDs remain unique.
    - Empty/whitespace `parallel_with` means untagged.
    - Workflow with zero steps is invalid for this slice.
  - Risk:
    - If any future serializer expects blank `parallel_with` not preserved as empty string semantics, it can diverge.
    - Label reuse after group closure is rejected, so workflows relying on reusing labels intentionally will now fail.

### Concept Match

- `workflow.Step` has optional `parallel_with` field: implemented.
- Untagged steps in separate stages: implemented.
- Adjacent same non-empty labels merge: implemented.
- Step after parallel group moves to later stage: implemented.
- Second parallel group with different label creates later stage: implemented.
- Reusing closed label returns validation error: implemented.
- Duplicate step IDs error: implemented (reuses same message pattern as compile-time check).
- Zero-based indexes preserved in stage and step fields: implemented.
- Normalization does not compile work items: implemented by isolation in `NormalizeStages`.
- Existing compile tests remain intact, no cross-surface redesign: implemented.

### Concrete Example Path

Input steps:
1. `{ID: "extract", ParallelWith: "batch"}`
2. `{ID: "transform", ParallelWith: "batch"}`
3. `{ID: "notify"}`
4. `{ID: "archive", ParallelWith: "report"}`

`NormalizeStages` output:
- Stage 0 (`ParallelWith: batch`) contains steps 0 and 1.
- Stage 1 (`ParallelWith: ""`) contains step 2.
- Stage 2 (`ParallelWith: report`) contains step 3.
- Closed-label tracking sets `batch` as closed when entering stage 1.

### Edge Cases Covered by Tests

- Empty workflow rejected.
- Duplicate step IDs rejected.
- Untagged default-stage behavior.
- Contiguous parallel runs merged.
- Non-contiguous label reuse rejected.
- Transition from parallel group to untagged step.
- Transition between different parallel labels.
- Whitespace-only/extra-whitespace `parallel_with` trimmed and normalized.

### Edge Cases Not Covered Yet

- Zero-length and whitespace-only step IDs (already not introduced in this slice; currently handled by compile layer).
- Error behavior when `workflow.ID == ""` in `NormalizeStages` (slice did not define this explicitly).
- Extremely large workflows where stage memory layout might matter.
- Explicit JSON round-trip for `parallel_with` through workflow submission payloads.

### Ambiguities / Maintenance Risks

- `Step` already had no struct tags previously, so adding a tag only for `parallel_with` may be inconsistent with future JSON naming policy unless a broader field-tag migration happens.
- The function currently always treats trim-fail to empty as untagged; if later requirements want a hard validation error, this is a behavior contract change for this slice.
- `NormalizeStages` currently enforces closed-label reuse rejection immediately and does not expose a toggle.
- `Workflow.ID` is carried in `WorkflowPlan` even though this slice has no explicit ID-required rule.

### What I should understand before moving on

- `NormalizeStages` is a pure transformation and validation step; it does not compile or queue work.
- Stage membership is contiguous-only and order-preserving.
- A parallel label cannot appear after an untagged break once closed.
- Duplicate step IDs are still invalid at normalization time.
- This slice only creates planning state that later slices are expected to consume.

### Questions I should answer

- Should `workflow.ID` be required by `NormalizeStages` or deferred to compile-time only?
- Do we want to reject empty/whitespace `parallel_with` instead of normalizing to untagged?
- Should `WorkflowStageStep` carry the full canonicalized `Step` or just step ID/reference?
- Should repeated label reuse after direct contiguous reopen (without intermediate stage) be allowed in any later extension?
- Which serialization contract should `Step` follow for every field (camelCase vs snake_case)?
