## Implementation Review: Dependency-Aware Workflows / 005 Submit Only Initial Ready Stage

### Execution Order

1. In `cmd/controller/main.go` (`submitWorkflowRunToStore`), switched admission to:
   1) resolve submission source,
   2) compile only stage `0` after `workflow.NormalizeStages`,
   3) run admission preparation on stage-0 compiled payloads,
   4) persist stage-0 work records/membership rows only.
2. In `cmd/controller/main.go`, kept run/stage plan persistence intact, but guarded dependency-plan creation with stage availability and moved queue record generation to use the single compiled stage result.
3. In `cmd/controller/main.go`, preserved public acknowledgment shape while keeping existing route behavior (`StatusAccepted` + JSON payload).
4. In `cmd/controller/workflow_stage_queue.go`, changed `persistenceRecordsFromCompiledStageResults` to assign `work_item_index` per stage so compile-time step-local indexes do not collide across parallel steps.
5. In `cmd/controller/main_test.go`, added targeted tests for:
   - sequential two-step submission queues only stage-0,
   - parallel stage-0 queues all initial-stage items,
   - invalid non-contiguous `parallel_with` is rejected before queue/run persistence side effects.
6. In `cmd/controller/workflow_stage_queue_test.go`, updated deterministic index expectations to match stage-local indexing behavior.
7. Marked `docs/concepts/dependency-aware-workflows/005-submit-only-initial-ready-stage.md` as implemented.
8. Updated `PROJECT_STATE.md` to include slice-005 completion.
9. Added usage record at `../epistemic-control/usage/dependency-aware-workflows/20260705194437.md`.

### Public/Public-Facing Changes

- `cmd/controller/main.go` (`submitWorkflowRunToStore`)
  - What it does: admission now stops at compiled stage 0 for live submission, instead of compiling everything.
  - Why it exists: matches OS-005 objective of staging only initial-ready work at submit-time.
  - Assumption: later-stage work must remain blocked/uncompiled and become compilable via future slices.
  - What could break: if any downstream code expects `WorkflowDependencyPlan` to exist only after full compile, it will now observe empty queued/future-work state immediately after submit.

- `cmd/controller/workflow_stage_queue.go` (`persistenceRecordsFromCompiledStageResults`)
  - What it does: indexes persisted membership records per stage-local queue order instead of per-step local index.
  - Why it exists: prevents duplicate `work_item_index` collisions for parallel step fan-outs in the same stage.
  - Assumption: deterministic stage-local order should define both queue metadata and membership index.
  - What could break: if future stages depend on old step-local indexes, those consumers would need migration.

### Concept Match

- Compile only stage 0 only: implemented.
- Queue only stage-0 work after submit: implemented and tested.
- No stage-1 compilation during submit: implemented.
- Invalid parallel plan fails before queue mutation: implemented and tested by checking both active runs and queued rows remain empty.
- Existing public `submission_id` shape unchanged: implemented by preserving existing handler response text + accepted behavior.
- Future-stage work not counted as assignable pending: implemented because only stage-0 rows are enqueued.

### Concrete Example Path

1. A sequential workflow with two fan-out steps is posted to `/workflow`.
2. `submitWorkflowRunToStore` normalizes stages and calls `CompileWorkflowStage` with stage index `0`.
3. Admission prep populates source-admitted work item payloads.
4. `CreateWorkflowRun`, stage plan, and dependency plan are persisted.
5. `persistenceRecordsFromCompiledStageResults` generates only stage-0 work records, queue entries, and memberships.
6. `InsertWorkItems` and `EnqueueWorkItems` persist only those records.
7. `persistedWorkDemand` and worker-scaling are computed from currently queued work only.

### Edge Cases Covered by Tests

- Two-step sequential submission queues only two stage-0 items, no stage-1 items.
- Parallel group in stage-0 queues all stage-0 fan-out outputs.
- Invalid non-contiguous `parallel_with` returns error and writes neither run rows nor queued rows.
- Stage/workflow queue records preserve stage-local deterministic `work_item_index` values across parallel members.

### Edge Cases Not Covered

- Assertion of exact HTTP status/message for different validation failure classes in `/workflow`.
- Recovery behavior when a later slice partially inserts future-stage dependency metadata before stage-0 queueing.
- Full persistence audit (journal/log) proving no side effects when admission prep fails after run creation but before queue insert.

### Ambiguities / Risk

- Stage-0 compile path returns `err` on zero-step workflows as currently handled; this slice preserves empty-queue behavior, but acceptance would benefit from an explicit test asserting that case.
- `CreateWorkflowDependencyPlan` is still created for normalized plans even when no work rows are inserted (zero-step/compile-empty flow), which is behavior-preserving for now but should be documented if slice semantics expand.
- `work_item_index` numbering now comes from stage-local sequence, not step-local sequence; later consumers that assume step-local indexing will require coordination.

What I should understand before moving on
- Submission side effects are intentionally limited to stage-0 only; this changes admission timing but not public API contract.
- `submission_id` flow and queue scaling remain in the same controller path; only the compile frontier is narrowed.
- Stage-local work-item index is now the queue metadata canonical for stage-local deterministic ordering.

Questions I should answer
- Which future slice writes stage transitions and unblocks stage-1 in this same model?
- Should `work_item_index` be explicitly documented as stage-local canonical metadata in shared interfaces?
- Are invalid parallel topologies now expected to be surfaced as `400` or remain `500` in controller response semantics?
- Where should explicit non-empty but compile-empty stage-0 behavior be tested (status counts vs queue rows)?
- What transaction guarantees do we need if stage creation and queue insertion are decoupled under future concurrency?

Recommended next test or tiny experiment
- Add an explicit assertion for a single-step workflow where fan-out yields zero items to confirm queue length, acknowledgement count, and status counts all remain coherent.
