## Implementation Summary

This slice did not add production observability behavior. It updated the Python WorkItem smoke path so a submitted Python workflow is verified through the existing controller-owned log path:

```text
POST /workflow
go run ./cmd/worker <smoke-config>
go run ./cmd/demo-client logs <submission_id> --json
```

The smoke script now extracts the `submission_id` from the workflow acknowledgement, runs current-source controller and worker processes, verifies the promoted Python output file, fetches submission logs through `goet logs`, and requires at least one stdout or stderr log entry.

## Execution Order

1. Validate sibling demo fixtures and compile `scripts/hello.py`.
2. Start the controller from current source with `go run ./cmd/controller --config ./cmd/controller/demo-config.json`.
3. Submit `submissions/python-hello-local.json` with `worker_config.worker_max_count=0`.
4. Parse `submission_id` from the JSON acknowledgement body, with a `Location` header fallback.
5. Start the worker from current source with an absolute generated config path.
6. The generated worker config points `python_executable` at a smoke-only wrapper that emits one stdout line and one stderr line, then delegates to the selected Python interpreter.
7. Wait for controller idle, verify `cmd/worker/.run/data/python-hello-hello.json`, then call `goet logs <submission_id> --json`.
8. Require non-empty log entries and at least one `stdout` or `stderr` stream entry.

## Changed Surfaces

1. `scripts/python-workitem-smoke.ps1`
   - Adds response-header capture and `Parse-SubmissionId`.
   - Normalizes HTTP method names to uppercase before invoking curl.
   - Writes curl request bodies and generated worker config as UTF-8 without BOM so Go JSON decoders accept them under Windows PowerShell.
   - Uses `http://localhost:8080`, matching `cmd/controller/demo-config.json`.
   - Runs controller and worker via `go run` instead of possibly stale `.run/*.exe` binaries.
   - Generates a temp-dir Python wrapper to produce deterministic stdout/stderr for log verification.
   - Replaces worker-local attempt-log assertions with controller-owned `goet logs` assertions.

2. `PROJECT_STATE.md`
   - Updates the smoke-path description to document controller-owned log verification through `goet logs <submission_id>`.
   - Records that the smoke config uses a wrapper to create deterministic stdout/stderr observations.

3. `docs/concepts/execution-observability/011-update-observability-docs-and-smoke.md`
   - Marks the Operational Slice as `Implemented`.

## Design Comparison

The slice is implemented as documentation and smoke-script wiring only. No production endpoint, CLI flag, worker behavior, controller behavior, or model shape was added.

The implementation matches the OS acceptance criteria that the smoke path verifies controller-owned logs for a submitted Python workflow and no longer relies on worker fallback logs for normal success.

The implementation is slightly different from a pure fixture-based smoke because the sibling `hello.py` fixture does not emit stdout or stderr. The smoke script now supplies a temporary Python wrapper so the worker's Python subprocess captures deterministic stdout/stderr lines without modifying the sibling demo project.

## Concrete Example

1. The script receives `submission_id = run-0012930d6a8474a9398b0d59077f72cd`.
2. The worker invokes the generated wrapper as `python_executable`.
3. The wrapper prints `goet smoke stdout` and `goet smoke stderr`, then invokes the real Python interpreter.
4. The worker captures those lines, emits `model.LogObservation` records to `POST /observations/logs`, and the controller stores them.
5. `go run ./cmd/demo-client logs run-0012930d6a8474a9398b0d59077f72cd --json` returns two entries.

## Edge Cases Covered

1. Stale prebuilt controller/worker binaries are bypassed by using `go run`.
2. Windows PowerShell UTF-8 BOM behavior is avoided for HTTP request bodies and generated worker config JSON.
3. HTTP helper sends `GET`/`POST` instead of `Get`/`Post`.
4. Submission ID extraction handles JSON body acknowledgement and `Location` fallback.
5. Smoke verification fails if no controller-owned stdout/stderr log entries are returned.

## Edge Cases Not Covered

1. The script does not test filtered `goet logs` queries such as `--stream stdout` or `--tail`.
2. The script does not verify fallback log absence explicitly; it verifies normal success through controller-owned logs.
3. The script does not clean up the temporary wrapper under the system temp directory.
4. The script assumes `go run` is available and acceptable for this smoke path.

## Risks And Coupling

1. The wrapper exists because the current sibling demo fixture does not emit stdout/stderr. If the fixture later emits its own lines, the wrapper may become unnecessary.
2. The script now depends on the CLI JSON shape returned by `goet logs`.
3. Running current source avoids stale binaries but makes the smoke path slower than using prebuilt executables.

## What I Should Understand Before Moving On

This slice proves the public observability path from a submission ID. It does not prove a new logging implementation; it proves that existing worker emission, controller ingestion/storage, log-read API, and CLI rendering compose in the Python smoke path.

## Questions I Should Answer

1. Should the sibling Python fixture eventually emit its own stdout/stderr so the wrapper can be removed?
2. Should the smoke script verify `goet logs --stream stdout` and `--stream stderr` separately?
3. Should fallback-log absence become an explicit assertion?
4. Should the smoke wrapper be deleted during cleanup?
5. Should the smoke path build binaries once instead of using repeated `go run`?
