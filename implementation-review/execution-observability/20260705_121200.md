# Implementation Review: 007 Python Subprocess Log Emission

Strategic Concept: execution-observability
Operational Slice: 007-python-subprocess-log-emission
Date: 2026-07-05
Review Time: 2026-07-05

## Execution Review

1. `Worker.Run` continues to dispatch `python_script` work items through `runPythonScript`.
2. `runPythonScript` now stages source and runs the subprocess with existing `stdout`/`stderr` files in the attempt log directory unchanged.
3. After process completion, `runPythonScript` calls `emitPythonSubprocessLogLines`, which reads `stdout.log` then `stderr.log`.
4. `emitPythonSubprocessLogLinesFromPath` maps each captured line to `model.LogObservation` and calls `LogClient.SendLogObservationWithFallback` using `attemptLogDir` as fallback target.
5. The main execution path ignores any log-emission error so subprocess success/failure still follows original behavior.
6. Python output/evidence parsing still performs the existing `GOET_OUTPUT_JSON` read, canonicalization, hashing, and output promotion flow.

## Public Surface Changes

No new exported public API was added.

- `runPythonScript` now has a non-fatal side effect of replay-emitting subprocess lines.
- New internal helpers:
  - `emitPythonSubprocessLogLines`
  - `emitPythonSubprocessLogLinesFromPath`
- New test helper:
  - `newPythonSourceAndLogServer`

## Concrete Example Trace

1. Worker downloads source bundle for item `python-emit-001`, attempt `attempt-emit-001`.
2. Python script prints to stdout and stderr and writes `GOET_OUTPUT_JSON`.
3. Runner writes attempt logs to `attempts/attempt-emit-001/logs/stdout.log` and `stderr.log`.
4. Runner closes both files, calls emitter, scanner maps each line to a `LogObservation` with `component=worker-python`.
5. Delivery call to `/observations/logs` succeeds or, if not, `fallback-observations.jsonl` is written in the same log directory.

## Edge Cases Covered by Tests

- Normal emission for both streams with metadata and stream-level level checks.
- Controller log endpoint returning non-2xx while Python completes.
- No-op of script with no log lines in fallback path still writes a fallback file when endpoint fails.
- Existing evidence/hash integrity checks remain verified by pre-existing Python tests.

## Edge Cases Not Covered by Tests

- Very long single log lines beyond bufio.Scanner token limits.
- Mixed ordering guarantees across stdout and stderr streams.
- Missing `item.Source` with empty submission/run IDs.
- Concurrent failures when both stdout and stderr emission fail and one observation succeeds.

## Assumptions and Risk

- Assuming `item.Source.RunID` is the best available stable identity for controller routing.
- Assuming replaying logs after process completion is acceptable; this can reorder relative to live output.
- Assuming dropping empty/whitespace-only lines is acceptable for subprocess observation fidelity.

What I should understand before moving on:
- Why replay-emission was chosen over teeing during execution and what tradeoff it imposes.
- Which identity field is canonical for submission routing in future worker assignments.
- Whether empty log lines should be emitted as informational observations.

Questions I should answer:
- Which metadata field is currently the strongest evidence for submission-level ownership?
- What fallback behavior should apply if `/observations/logs` intermittently stalls but does not fail?
- How should global line ordering across stdout and stderr be handled for future replay consumers?
- What stream level should stderr use if non-error warnings are used in scripts?
- Where should log metadata validate worker identity in this model?
