# Implementation Review: 009 Submission Log Read API

Strategic Concept: execution-observability
Operational Slice: 009-submission-log-read-api
Date: 2026-07-05
Review Time: 2026-07-05

## Execution Review

1. On controller startup, `buildControllerServer` now captures log-read policy values and stores them on `Controller` (`logRootPath`, `logReadDefaultTail`, `logReadMaxTail`).
2. Route registration changes to `registerControllerRoutes` now map `\submissions\` to `submissionHandler` instead of directly binding status.
3. `submissionHandler` dispatches to `submissionStatusHandler` when path ends with `/status`, `submissionLogsHandler` when path ends with `/logs`, otherwise returns 404.
4. `submissionLogsHandler` validates method, admission state, store availability, path shape, and known-submission existence through `workflowStore.GetWorkflowRun`.
5. Query parsing happens in `parseSubmissionLogTail`, `parseSubmissionLogLevel`, `parseSubmissionLogStream`; unsupported values return HTTP 400.
6. `readSubmissionLogEntries` reads `submission.jsonl` plus `submissions/{id}/attempts/*.jsonl`, filters by query, parses timestamps, stable-sorts by timestamp then read order, and returns bounded tail with `truncated` metadata.
7. `submissionLogResponse` returns `submission_id`, `entries`, `tail`, and `truncated` as JSON.
8. Tests in `cmd/controller/log_read_test.go` cover known/unknown submission checks, filter semantics, invalid query behavior, tail truncation, and unknown suffix dispatch.

## Public Surface Changes and Assessment

- `Controller` fields `logRootPath`, `logReadDefaultTail`, `logReadMaxTail` (private fields in `main.go`)
  - What: adds startup-configured bounds and paths needed by log read handler.
  - Why: allows request parsing without recomputing policy.
  - Assumption: startup policy is resolved before server registration.
  - Risk: unset values would permit unbounded/empty behavior; this is guarded by startup validation.
  - Slice status: implemented.

- `submissionHandler` (`main.go`)
  - What: dispatches `/submissions/{id}/status` and `/submissions/{id}/logs`.
  - Why: preserves previous status endpoint and adds log endpoint in same resource space.
  - Assumption: request path is strictly suffix-based.
  - Risk: any unrecognized suffix under `/submissions/` now 404s (same family of behavior expected by route).
  - Slice status: implemented.

- `submissionIDFromSubmissionPath` + `submissionIDFromLogsPath` (`main.go`)
  - What: shared path parsing utility now used by status and logs handlers.
  - Why: avoids duplicate parsing and prevents nested slash IDs.
  - Assumption: submission IDs never include `/`.
  - Risk: malformed path with extra segments returns 404.
  - Slice status: implemented.

- `submissionLogResponse`, `submissionLogsHandler`, `readSubmissionLogEntries`, parsers/helpers (`log_read.go`)
  - What: new handler path for bounded log retrieval.
  - Why: expose controller-owned read API for submission logs.
  - Assumption: filesystem log sink path format remains `logRootPath/submissions/{id}/submission.jsonl` plus `.jsonl` files under `attempts/`.
  - Risk: log timestamp parsing errors stop whole response; scanner behavior can truncate very long lines.
  - Slice status: implemented.

- Tests in `log_read_test.go`
  - What: new coverage for endpoint behavior and parsing.
  - Why: verify the acceptance criteria with functional cases.
  - Assumption: test store + file sink remain consistent.
  - Risk: missing tests for invalid timestamp lines and malformed JSON rows.
  - Slice status: implemented.

## Edge Cases Covered by Tests

- Known submission id with no logs present.
- Unknown submission id.
- Query with valid filters and `tail` override.
- Invalid `tail`, `level`, `stream` values.
- Tail truncation with bounded response and `truncated=true`.
- Unknown `/submissions/{id}/bad` under `/submissions/` route.

## Edge Cases Not Covered by Tests

- Corrupt JSON lines under log files returning HTTP 500.
- Invalid log timestamps causing parse failures.
- Missing `logRootPath`/policy initialization in partial controller construction.
- Very long log lines that exceed `bufio.Scanner` token limits.
- Stable ordering across multiple attempt files with exact timestamp ties beyond sequence order from read order.

## Ambiguity and Risk

- OS text says "reads controller-owned logs" and also references not exposing worker-local files. Current implementation reads attempt files in the controller log path, which is compatible with current sink layout, but acceptance of additional worker-local paths remains implicitly excluded by directory policy only.
- In-recovery mode behavior remains blocked for this endpoint because of existing `requireNormalAdmission` usage; the spec does not state explicitly whether this is intended.
- The previous one-handler behavior for `/submissions/` status was replaced with suffix dispatch; if other submission subresources are later added, this branch needs to remain explicit.

## What I should understand before moving on

- Route flow now: startup policy -> route registration -> submission suffix dispatch -> known-submission check -> bounded read pipeline.
- Tail behavior is always bounded against policy and filtered before sorting/return.
- Log response intentionally omits any filesystem metadata.
- Failure behavior: malformed query and parse errors are client or 500 depending on source.

## Questions I should answer

- Why is log-read intentionally blocked in recovery mode, and is that acceptable for clients polling during startup?
- Do we need stable total ordering across all entries when timestamps collide, or is the current read-order tie-break sufficient?
- Should bad JSON lines in a log file return partial results or fail fast with 500?
- Should this endpoint include attempt-level stream provenance when `attempt_id` filter is absent?
- Should `tail` clamp or reject when above max, and should policy-driven max ever be overridden by client for observability debugging?

## Recommended next test or tiny experiment

- Add a regression test where one `.jsonl` contains an invalid line and one valid line to verify endpoint behavior and choose whether to fail-fast or skip malformed lines intentionally.
