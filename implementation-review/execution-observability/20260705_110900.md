# Implementation Review: 007 Python Subprocess Log Emission

Strategic Concept: execution-observability
Operational Slice: 007-python-subprocess-log-emission
Date: 2026-07-05
Review Time: 2026-07-05

## Execution Review

1. `runPythonScript` now captures subprocess logs into attempt-local `stdout.log` and
   `stderr.log` as before.
2. After subprocess completion and log file close, `runPythonScript` invokes
   `emitPythonSubprocessLogLines` with the staging paths and fallback directory.
3. `emitPythonSubprocessLogLinesFromPath` scans each completed log file line-by-line,
   maps each non-empty line to a `model.LogObservation`, and calls
   `LogClient.SendLogObservationWithFallback` for each line.
4. In the same process, submission evidence behavior (`GOET_OUTPUT_JSON`, SHA
   hashing, evidence fields, output promotion) remains unchanged and continues
   after logging side effects.
5. New tests validate successful emission for stdout/stderr lines and fallback file
   writes when `/observations/logs` returns HTTP 503.

## API / Surface Changes

No new exported production API was added.

- `runPythonScript` behavior changed: it now has a non-fatal side-effect that emits subprocess log observations after process completion.
- New internal helpers:
  - `emitPythonSubprocessLogLines`
  - `emitPythonSubprocessLogLinesFromPath`
  - `newPythonSourceAndLogServer` (test only)
  - `containsAll` (test only)

### Why these exist / assumptions

1. `emitPythonSubprocessLogLinesFromPath`
   - Exists to keep stream-specific metadata and transport behavior localized.
   - Assumes controller-facing log IDs can be derived from `item.Source.RunID`.
   - If that assumption is wrong, `submission_id` is empty and the sink writes
     controller-level logs instead of submission-scoped logs.
2. `LogClient.SendLogObservationWithFallback`
   - Reused existing transport path; assumption is controller endpoint and fallback
     path are correct and writable in normal environments.

### What could break

- If `item.Source` is ever unset for Python items, submission-level enrichment
  (`submission_id`, `run_id`) drops to empty, changing routing path.
- Empty stdout/stderr lines are intentionally skipped, so blank-line boundaries are
  not emitted as observations.
- The two-step replay means stream ordering across stdout and stderr is not
  preserved globally by design.

## Trace of Concrete Example

1. Worker fetches a Python work item with `attempt_id=attempt-001` and
   `source.run_id=run-123`.
2. Python process runs and writes two lines to `attempts/attempt-001/logs/stdout.log`
   and two lines to `attempts/attempt-001/logs/stderr.log`.
3. After process exit and closing files, runner calls
   `emitPythonSubprocessLogLines`.
4. A line such as `stdout one` from the stdout file becomes an observation with:
   `stream=stdout`, `level=info`, `component=worker-python`, `work_item_id`,
   `attempt_id`, `submission_id=run-123`, and a fresh `timestamp`.
5. Each send uses `SendLogObservationWithFallback`, and on failure writes the same
   observation to `attempts/attempt-001/logs/fallback-observations.jsonl`.

## Test Coverage Review

Covered by new tests:
- `TestWorkerRunWorkItemEmitsPythonSubprocessLogs`
  - stdout/stderr stream separation
  - level mapping
  - metadata presence (`work_item_id`, `attempt_id`, `submission_id`, `run_id`,
    ids, sequence)
  - line payload preservation for expected messages
- `TestWorkerRunWorkItemFallsBackToFallbackLogOnLogDeliveryFailure`
  - non-fatal work completion with controller delivery errors
  - fallback file creation

Not explicitly covered:
- Empty line emission behavior (blank lines are currently skipped).
- Concurrent delivery ordering with large logs.
- Worker identity propagation (`worker_id` is absent by design due missing source fields).
- Explicit check that one emission helper call returning an error does not abort subsequent line replay when multiple errors occur.

## Risk and Coupling Notes

- Best-effort behavior is implemented with a best-effort emission path but still
  reads all log files; this is simple and safe for current run sizes, but
  ignores extremely large line behavior and does not protect against scanner token
  limits.
- `submission_id` derivation from `Source.RunID` is an assumption that relies on
  run-to-submission identity mapping being stable.

What I should understand before moving on:
- How `submission_id` should be sourced for worker assignments once a separate
  explicit submission identity is added.
- Whether blank-line log content should produce observations or be dropped.

Questions I should answer:
- Should `worker-python` continue to be the component label, or should it be
  a shared controller-defined component vocabulary?
- Should sequence be monotonic across both streams instead of per stream?
- Should we gate fallback writes by failure type, or keep the current unconditional
  best-effort fallback behavior?
