# Implementation Review: 010 CLI Logs Command

Strategic Concept: execution-observability
Operational Slice: 010-cli-logs-command
Date: 2026-07-05
Review Time: 2026-07-05 12:08:13

## Execution Review

1. `parseCommand` now recognizes `logs` and dispatches to `parseLogsCommand`.
2. `parseLogsCommand` resolves `--controller-url`, optional `--tail`, `--level`, `--stream`, `--attempt-id`, and `--json` from `goet logs`.
3. `logsCommand` builds a resolver via `statusResolver`, creates an `internal/client.ControllerClient`, and calls `SubmissionLogs`.
4. `SubmissionLogs` resolves the controller URL, validates the tail filter, then calls `submissionLogs`.
5. `submissionLogs` builds `GET /submissions/{submission_id}/logs` with optional query parameters, handles status errors, decodes `SubmissionLogsResponse`, and returns parsed observations.
6. `logsCommand` prints observations via `formatSubmissionLog` for text output or `jsonMarshalLogs` for `--json`.

## Public API and Surface Changes

New or changed public surface in `internal/client`:
- `SubmissionLogsFilters` struct: new filter payload for tail/level/stream/attempt query parameters.
  - Why: gives callers a typed argument for bounded/log-filtered retrieval.
  - Assumption: callers pass coherent filter strings that match controller log query validation.
  - Risk: invalid values are validated only by downstream HTTP behavior; no pre-validation means server errors can occur at call time.
- `SubmissionLogsResponse` struct: new typed response shape matching `/submissions/{submission_id}/logs`.
  - Why: command output and API decoding need strong typing and structured rendering.
  - Assumption: controller keeps shape stable with `submission_id`, `entries`, `tail`, and `truncated`.
  - Risk: schema drift in controller responses will fail JSON decode.
- `SubmissionLogs` and unexported `submissionLogs` methods on `ControllerClient`.
  - Why: adds logs read support to client boundary.
  - Assumption: controller URL resolves from resolver and endpoint exists.
  - Risk: if resolver is misconfigured, all client calls fail before request construction.

New or changed CLI surface:
- `goet logs <submission_id>` command in `cmd/demo-client/main.go`.
  - Why: exposes logs read slice functionality to users.
  - Assumption: operator already has valid `submission_id` from status/submission output.
  - Risk: no follow/watch semantics by design; users expecting tailing must compose with external tools.
- New flags `--controller-url`, `--tail`, `--level`, `--stream`, `--attempt-id`, `--json`.
  - Why: supports default URL, bounded reads, optional filtering, and machine format.
  - Risk: unsupported future additional filters require CLI parser extension.

## Concrete Example Trace

User runs:
`goet logs sub_1234 --controller-url http://localhost:8080 --tail 20 --json`.

1. `parseCommand` routes to `parseLogsCommand`.
2. `splitFlagArgs` preserves `--tail 20` and `--json` as flags and returns positional `sub_1234`.
3. `parseLogsCommand` sets `SubmissionID="sub_1234"`, `Tail=20`, `TailSet=true`, `JSON=true`.
4. `logsCommand` creates a resolver from `--controller-url`, then calls `ControllerClient.SubmissionLogs`.
5. `submissionLogs` sends `GET /submissions/sub_1234/logs?tail=20`.
6. API response decodes into `SubmissionLogsResponse`.
7. `logsCommand` outputs JSON payload on stdout.

## Edge Cases Covered by Tests

- Command parsing:
  - missing submission ID
  - extra positional arguments
  - invalid/missing tail values
  - disallowed `--watch` and `--follow`
  - optional filter flags and defaults
- Execution path:
  - text output formatting with and without stream/attempt ID
  - JSON output parseability
- Client path:
  - happy-path endpoint query assertion
  - invalid tail and empty submission checks before HTTP
  - 404 unknown submission translation

## Edge Cases Not Covered by Tests

- Empty `entries` response with non-empty metadata.
- Server returning large but valid successful bodies (bounded by existing behavior).
- `logs` command handling `--json` output with malformed response body.
- Path-escape behavior for submission IDs containing spaces or unusual UTF-8 runes (client path is escaped, but command UX is not explicitly tested).
- Combined optional filter interactions when zero-value vs omitted fields are intentionally not sent.

## Hidden Coupling and Risks

- `splitFlagArgs` is now aware of logs-only flags and is shared across all commands.
  This is currently intentional to keep CLI parsing behavior consistent, but adding command-specific unknown flags will continue to rely on this shared pre-pass.
- `goet logs` and `goet status` both route through `statusResolver`, which constrains log URL overrides to the same controller URL handling style as status.
- Text output is intentionally lossy for metadata that is present in structured JSON (e.g., submission/workflow IDs), which may reduce diagnosability unless `--json` is used.

What I should understand before moving on:
- CLI log retrieval is fully implemented and test-backed through `goet logs`.
- `internal/client` owns query construction and endpoint decoding.
- Text output is compact; JSON output is the only metadata-complete format.

Questions I should answer:
1. Why was `--json` implemented as full response echo instead of custom pretty output?
2. Which metadata fields from `LogObservation` do we intentionally lose in text output?
3. What caller should handle unknown `level/stream/attempt-id` values if future client-side validation is added?
4. Should `--tail` default stay client-driven or be fully server-driven forever?
5. Do we need a structured error type for log retrieval failures to support CLI exit behavior?

Recommended next test or tiny experiment:
