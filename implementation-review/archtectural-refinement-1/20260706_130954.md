## Changed Surfaces

`workflow_step_output_facts` table:
What it does: stores one logical output fact per `(run_id, step_index)` with optional bounded JSON, SHA-256, byte count, pruned flag, output kind, and timestamps.
Why it appears to exist: the OS says empty fan-out `[]` and pruned aggregate metadata cannot always be reconstructed from `completed_work`.
Assumption: `(run_id, step_index)` is the durable logical step key created by `workflow_dependency_steps`.
If wrong: output facts could collide or fail to represent repeated logical steps in a future workflow model.
Design match: fully implemented for aggregate and empty fan-out facts; `skipped` is schema-allowed but no new skipped/reuse path is implemented in this slice.

`WorkflowStepOutputFactRecord` and store methods:
What they do: expose upsert/get methods for the new table and validate JSON, hashes, byte counts, output kind, and timestamps.
Why they exist: controller code needs a narrow persistence boundary without direct SQL in controller files.
Assumption: upsert is acceptable because a running aggregate fact can later become a pruned terminal fact.
If wrong: preserving `created_at` while updating payload may be insufficient for audit needs.
Design match: fully implemented for current acceptance criteria.

Controller aggregate write path:
What it does: after `aggregateStepOutputJSON` completes a step, the controller writes a fact with `output_kind = "aggregate"`.
Why it exists: aggregate step output metadata is not just a completed work row, especially after membership and step output JSON are pruned.
Assumption: `RecordCompletedWorkItemOutput` is the controller point where logical aggregate output becomes known.
If wrong: another completion path could create aggregate output without writing this fact.
Design match: fully implemented for existing output-resolution behavior.

Controller empty fan-out write path:
What it does: `markCompiledStageEmptyStepsCompleted` writes `output_kind = "empty_fanout"` with `output_json = "[]"`.
Why it exists: zero-work fan-out has no work item, no attempt, and no `completed_work` row.
Assumption: absence from `stageResult.WorkItems` plus no existing memberships means the step is an empty fan-out outcome.
If wrong: a compile bug that omitted work items could be persisted as a successful empty fan-out.
Design match: fully implemented for the existing empty fan-out mechanism.

Terminal pruning fact refresh:
What it does: when terminal pruning runs after a later failure, already-written fact rows are updated with pruned metadata.
Why it exists: a step can aggregate while the workflow is still running and become pruned only after another step fails.
Assumption: an existing fact's `OutputKind` should be retained; missing facts default to `aggregate`.
If wrong: a missing empty-fanout fact pruned later could be mislabeled as aggregate.
Design match: implements the required pruned aggregate metadata behavior.

## Concrete Trace

A two-stage workflow completes step 0 while step 1 is still blocked. `RecordCompletedWorkItemOutput` canonicalizes `{"value":"a"}`, aggregates it into step 0, prunes membership output JSON, and writes `workflow_step_output_facts(run_id, step_index=0, output_json='{"value":"a"}', output_kind='aggregate', output_json_pruned=0)`.

If step 1 later fails, `updateDependencyPlanForWorkItemTerminal` marks the workflow failed and `pruneDependencyOutputsIfTerminal` clears step 0 output JSON while retaining hash and bytes. `upsertTerminalPrunedWorkflowStepOutputFacts` reloads the existing output kind and updates the fact to `output_json=NULL`, `output_json_pruned=1`, preserving hash and byte count.

## Risks And Ambiguities

The schema bump rejects existing version 3 stores because there is no migration path in the current SQLite initializer. That matches the repo's current schema-version pattern but matters for durable local databases.

Fact writes and submission-context updates are not transactional together. A failure between those two writes can leave one durable surface updated and the other not. That coupling already exists in this controller/store shape; OS 002d/002e should reduce reliance on submission-context state.

Empty fan-out terminal completion keeps existing in-memory behavior: the step output remains `[]` rather than being pruned immediately in `markCompiledStageEmptyStepsCompleted`. This preserves current behavior but means the new fact may be unpruned for an all-empty terminal workflow.

## What I Should Understand Before Moving On

`workflow_step_output_facts` is not a replacement for `completed_work`; it stores logical step-level facts that do not safely fall out of completed work rows.
