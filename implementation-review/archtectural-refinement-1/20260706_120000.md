Changes and Validation
- `internal/persistence/store_test.go`
  - Added `TestStoreClaimNextWorkCanClaimAcrossRunIDsWithSameWorker`.
  - What it does: creates two runs, one worker, and two queued work items from different runs; the same worker claims both successfully.
  - Why it exists: it directly tests the slice rule that a worker is reusable capacity, not run-owned.
  - Assumption encoded: `ClaimNextWork` uses only `queued_work`/`work_items` state, not `workers.run_id`, for claim eligibility.
  - Failure mode if wrong: test would fail on the second claim by returning same-run-only behavior or no second claim.

- `docs/data-model.md`
  - Changed `workers.run_id` notes from authoritative ownership wording to legacy launch-context wording.
  - Why it exists: aligns documentation with model intent and prevents future mistaken coupling to scheduling.
  - Assumption encoded: scheduling and ownership must derive from `work_items.run_id` through attempts/history joins.
  - Failure mode if wrong: future readers could reintroduce run-boundary coupling from docs.

- `docs/concepts/architectural-refinement-1/001-decouple-workers-from-run-id.md`
  - Status changed to `Implemented`.
  - Why it exists: marks slice completion and makes state visible for review continuity.
  - Assumption encoded: keeping `workers.run_id` is acceptable until a migration/specific cleanup slice justifies removal.
  - Failure mode if wrong: conceptual status can drift from actual behavior.

- `docs/concepts/architectural-refinement-1/README.md`

Edge Cases Covered by Tests
- Claim succeeds with worker identity present.
- Reuse of one worker identity across two queue items in different runs.
- Queue progress preserves existing order and claim state updates (`running_work` and `work_item_attempts`) for both claims.

Edge Cases Not Covered
- Legacy fixtures that directly write `workers.run_id` without any claims.
- Migration consequences of removing `workers.run_id` (not attempted).
- Recovery paths that might inspect `workers.run_id` outside this slice’s scope.

Potential Ambiguity / Future Risk
- The slice intentionally keeps `workers.run_id` despite it being misleading. A follow-up slice is still needed if we later want true schema cleanup.
- `execution_handle` is still stored alongside worker rows; recovery code that interprets it as run-bound may need explicit documentation.

What I should understand before moving on
- Work claim authority currently comes from queued work (`queued_work` + `work_items`) and `run_id` is only authoritative there through `work_items`.
- The OS objective is satisfied by behavior and explicit docs even though the schema column remains for compatibility.
- Any further cleanup slice should be treated separately and include migration or compatibility policy.
