# Sensitive Variable Propagation Review

## Read First

- Strategic Concept: [Sensitive Variable Propagation](../../../go-etl/docs/concepts/sensitive-variable-propagation/README.md)
- Operational Slice: [002 Protected Reference Model](../../../go-etl/docs/concepts/sensitive-variable-propagation/002-protected-reference-model.md)
- Objective: add a language-neutral protected-reference model for sensitive values that should not be resolved by the controller.
- Budget: `EC-3 / operational slice / files(7)+test+doc`

The slice contract explicitly requires:

- protected-reference declarations parse and validate;
- missing provider, missing key, and unsupported provider are rejected;
- protected references are implicitly sensitive;
- resolved values never store plaintext for protected references;
- safe rendering uses the redaction label;
- type information is preserved;
- `go test ./internal/variable` passes.

## Object / Transition Map

### 1. Variable declaration

`Variable` transitions from a typed literal declaration to a protected-reference declaration when `protected_ref` is present.

- Data object: `internal/variable.Variable`.
- Before: `name + type + expression + optional sensitive`.
- After: `name + type + optional expression` or `name + type + protected_ref`.
- Preconditions: valid name, valid type, provider present, key present, supported provider.
- Invariants:
  - protected references are sensitive even if `sensitive` is omitted;
  - `expression` is omitted for protected-reference declarations;
  - the redaction label defaults to `${provider.key}` when omitted.
- Failure states:
  - missing provider;
  - missing key;
  - unsupported provider;
  - ambiguous declaration with both `expression` and `protected_ref`.

### 2. Literal / resolved value creation

`Variable` transitions to `ResolvedValue` through `ParseLiteral` or `Resolver.Resolve`.

- Data object: `internal/variable.ResolvedValue`.
- Before: protected-reference declaration in a `Variable`.
- After: typed sensitive resolved value with `ProtectedRef` metadata and no plaintext `Value`.
- Preconditions: validated protected-reference declaration in scope.
- Invariants:
  - `Sensitive` is true;
  - `ProtectedRef` is preserved;
  - `Value` is not used to store secret plaintext;
  - `String()` returns the redaction label;
  - JSON rendering includes `protected_ref` rather than a plaintext value.
- Failure states:
  - the declaration is invalid;
  - the protected reference is used where plaintext string interpolation is required.

## Transition Order

### 1. Decode and validate the protected reference

`ProtectedRef.UnmarshalJSON` performs a strict JSON decode, rejects unknown fields, validates provider/key, and normalizes the redaction label.

### 2. Decode and normalize the variable declaration

`Variable.UnmarshalJSON` accepts a declaration with `protected_ref` and no `expression`, marks it sensitive, and stores the protected reference on the variable.

### 3. Resolve without plaintext

`ParseLiteral` and `Resolver.resolveVariable` detect `ProtectedRef` and call `protectedReferenceValue`, which returns a typed sensitive `ResolvedValue` with no plaintext materialization.

### 4. Render safely

`ResolvedValue.String`, `MarshalJSON`, and the aggregate sensitivity helpers treat protected references as sensitive and render the redaction label instead of a placeholder or plaintext.

## Concrete Example

Input:

```json
{
  "name": {"namespace": "workflow", "key": "gdrive_token"},
  "type": "string",
  "protected_ref": {
    "provider": "worker_env",
    "key": "GOET_GDRIVE_TOKEN"
  }
}
```

Trace:

1. `Variable.UnmarshalJSON` parses the declaration.
2. `ProtectedRef.UnmarshalJSON` validates provider and key, then assigns the default label `${worker_env.GOET_GDRIVE_TOKEN}`.
3. `Variable.Validate` accepts the declaration as protected and sensitive.
4. `Resolver.Resolve` returns a `ResolvedValue` with `Type=string`, `Sensitive=true`, `ProtectedRef` set, and `Value=nil`.
5. `ResolvedValue.String()` returns `${worker_env.GOET_GDRIVE_TOKEN}`.
6. `safeJSONText` includes `protected_ref` and omits a plaintext value.

If the provider were `client_env`, the declaration would fail validation before resolution.

## Test Coverage Review

Covered:

- `TestVariableJSONRoundTripIncludesProtectedReference`
  - Exercises protected-reference declaration parsing and JSON emission.
  - Proves implicit sensitivity and redaction label normalization.
- `TestProtectedReferenceJSONRejectsInvalidProviderAndKey`
  - Exercises validation failure paths for missing provider, missing key, and unsupported provider.
- `TestProtectedReferenceAllowsTestProvider`
  - Proves the deferred `test` provider is accepted.
- `TestParseLiteralPreservesProtectedReferenceWithoutPlaintext`
  - Exercises protected-reference literal resolution.
  - Proves `Value` stays empty and safe rendering uses the label.
- `TestResolverPreservesProtectedReferenceWithoutPlaintext`
  - Exercises controller-side resolution through `Resolver.Resolve`.
  - Proves the resolved value keeps the reference and does not materialize plaintext.

Not covered:

- Protected references nested inside larger string interpolations.
- Protected references nested inside structured object and list values.
- End-to-end controller assignment payload propagation.
- Worker-side secret lookup or redaction behavior.

Those gaps are acceptable for this slice because the OS explicitly excludes worker resolution, persistence, and controller envelope changes.

## Hidden Coupling and Maintenance Risk

- `Resolver.String` still expects a plaintext string value, so calling it on a protected reference will fail. That is consistent with this slice, but downstream code must not assume every `string`-typed resolved value is plaintext.
- `interpolate` still expects a resolved scalar value. If a protected reference is embedded inside a larger string expression, the current code path is not a safe transport for it. This is a later-slice concern.
- Provider support is hard-coded to `worker_env` and `test`. Any future provider requires an explicit slice and review.
- Protected-reference metadata now exists in both declaration and resolved-value shapes. That is intentional, but it means later controller and worker slices must preserve it consistently rather than recomputing it.

## Budget and Slice Fit

- The implementation stayed within the slice intent.
- It did not expand into controller persistence, worker secret lookup, or external secret stores.
- It changed only the variable package production files needed to model, validate, resolve, and safely render protected references.
- The new test file is justified because the slice introduces a distinct declaration and resolution contract.

## What I should understand before moving on

- A protected reference is not plaintext and must survive controller-side planning as metadata only.
- The controller-side variable package now preserves the reference and type without reading any secret source.
- Safe rendering is the contract for public outputs, not plaintext materialization.

## Questions I should answer
