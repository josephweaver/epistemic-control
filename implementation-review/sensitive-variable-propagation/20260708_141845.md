# Implementation Review: Sensitive Variable Propagation OS-006

Date: 2026-07-08

## Intended Design

Strategic Concept: Sensitive Variable Propagation.

Operational Slice: `docs/concepts/sensitive-variable-propagation/006-python-subprocess-secret-materialization.md`.

Objective: extend Python work-item execution so protected refs can be materialized for user-authored Python subprocesses through explicit minimal runtime surfaces without placing secrets on the command line.

HCI: EC-3 / operational slice / files(6)+test+doc.

Explicit constraints:

- Do not pass sensitive plaintext on subprocess command-line arguments.
- Materialize only explicitly requested protected refs.
- Support `env` mode and, because this implementation includes it, `file` mode.
- Scrub captured stdout/stderr for exact materialized secret values.
- Reject `GOET_OUTPUT_JSON` that contains an exact materialized secret before persistence.
- Remove temp secret files after success and failure.
- Do not implement cloud stores, real Google Drive, artifact scanning, or controller persistence changes.

## Object / Transition Map

`model.ExecutionEnvelopeProtectedReference` transitions from non-secret metadata to a Python subprocess runtime surface when `Materialize` is present, by `Worker.materializePythonProtectedRefs`, while preserving the invariant that plaintext is not written to argv, `GOET_INPUT_JSON`, model JSON, or persisted output.

Objects:

- Persistent data: none added. `WorkItem` and its execution envelope remain persisted metadata.
- In-memory data: `SensitiveValue`, `Redactor`, `exec.Cmd.Env`.
- Config data: worker environment variables addressed by `worker_env` refs.
- External artifacts: attempt-local stdout/stderr logs; attempt-local temp secret files under `work/secret-materializations`.

Failure states:

- Missing `worker_env` key fails before subprocess start with a redaction-label error.
- Unsupported materialization mode fails validation or worker setup.
- Output JSON containing a materialized secret fails before `pythonLogicalOutput` and before data-root persistence.
- Non-zero Python exit redacts logs and removes file-mode materializations before returning failure.

## Transition Order Review

1. The Python work item enters `Worker.runPythonScript`.

   State before: `model.WorkItem` has normal Python parameters and may have `ExecutionEnvelope.Variables.ProtectedRefs`.

   Operation: existing source staging, entrypoint resolution, `GOET_INPUT_JSON` writing, and stdout/stderr file setup still occur first.

   Invariant: the input document contains the work item metadata, not resolved plaintext secrets.

2. Protected refs are materialized.

   Code: `Worker.materializePythonProtectedRefs` in `cmd/worker/work_python.go`.

   Preconditions: a protected ref has a non-nil `Materialize` field; provider is supported by `WorkerEnvProtectedValueResolver`; the worker environment contains the named key.

   State after:

   - `env` mode appends exactly `Target=plaintext` to `exec.Cmd.Env`.
   - `file` mode creates `work/secret-materializations/secret-N`, writes plaintext with `0600`, and appends `Target=file_path` to `exec.Cmd.Env`.
   - Every materialized `SensitiveValue` is registered with an attempt-local `Redactor`.

   Invariant: plaintext is not appended to `exec.Cmd.Args`.

3. Python subprocess runs.

   Code: existing `exec.Command(w.pythonExecutable(), append([]string{entrypointPath}, args...)...)`.

   State after: Python may have written stdout, stderr, and `GOET_OUTPUT_JSON`.

   Failure behavior: on non-zero exit, stdout/stderr files are closed, scrubbed by `scrubPythonSubprocessLogs`, file materializations are removed by deferred cleanup, and the error returned does not include the secret.

4. Captured logs are scrubbed.

   Code: `scrubPythonSubprocessLog` reads each log file, applies `Redactor.RedactBytes`, and rewrites changed logs through `atomicWriteFile`.

   Invariant: emitted log observations and persisted attempt log files see redacted exact secret strings.

5. Structured output is checked.

   Code: `runPythonScript` checks whether redacting `outputJSON` changes the bytes. If yes, it returns `GOET_OUTPUT_JSON contains a materialized sensitive value`.

   State after rejection: `pythonLogicalOutput` is not called, artifact promotion is not attempted, and `DataDir/item.OutputFilename` is not written.

6. Successful output proceeds through existing logic.

   Code: `pythonLogicalOutput`, `atomicWriteFile`, evidence hash calculation, and evidence wrapper generation are unchanged except that logs were scrubbed before their SHA-256 values are computed.

## Changed Surfaces

`ParameterMaterialization.Validate` in `internal/model/work_item.go`:

- New: accepts `file` in addition to `env`.
- Modified: rejects targets with leading/trailing whitespace or `=`.
- Role: establishes that materialization targets are environment variable names, because this slice passes both env secrets and file paths through subprocess environment variables.

`Worker.materializePythonProtectedRefs`:

- New helper in `cmd/worker/work_python.go`.
- Role: resolves `worker_env` refs only at the Python execution boundary and only when `Materialize` is present.
- Assumption: `file` mode path delivery is through the target env var, which the OS permits as one of the file-mode path-passing options.

`scrubPythonSubprocessLogs` and `scrubPythonSubprocessLog`:

- New helpers in `cmd/worker/work_python.go`.
- Role: rewrite captured stdout/stderr before log observation emission and hash evidence.
- Limitation: exact string redaction only; transformed, split, encoded, or artifact leaks remain outside the protection claim.

## Concrete Trace

Input:

- Work item `python-secret-env-001`.
- Protected ref `gdrive_token`: provider `worker_env`, key `GOET_TEST_PYTHON_ENV_SECRET`, materialize `env` target `GDRIVE_TOKEN`.
- Worker environment contains `GOET_TEST_PYTHON_ENV_SECRET=goet-secret-python-env-006`.

Execution:

1. `runPythonScript` stages `scripts/run.py`.
2. `materializePythonProtectedRefs` builds an envelope if needed.
3. Resolver reads the worker env value and returns `SensitiveValue`.
4. Redactor registers `goet-secret-python-env-006` with label `${worker_env.GOET_TEST_PYTHON_ENV_SECRET}`.
5. `command.Env` receives `GDRIVE_TOKEN=goet-secret-python-env-006`.
6. Python reads `os.environ["GDRIVE_TOKEN"]`, prints it, and writes output without the secret.
7. Worker rewrites stdout/stderr so the secret is replaced with the label.
8. Worker checks `GOET_OUTPUT_JSON`; it does not contain the secret.
9. Worker persists canonical output and evidence.

If the env key is missing, step 3 fails and Python does not start.

## Risks And Ambiguities

- File-mode target semantics: the implementation uses the materialization `target` as an environment variable that receives the temp file path. This matches the OS allowance to pass file paths through env or `GOET_INPUT_JSON`, but the model field name does not distinguish env-var targets from future input-json targets.
- Output rejection uses exact string matching through the existing redactor. This satisfies the OS but will not detect encoded or transformed secrets.
- Log files are rewritten after subprocess exit. A crash between Python writing logs and worker redaction could leave raw attempt logs on local disk until cleanup or investigation.

## Final EC Summary

What I should understand before moving on:

- A protected ref is still metadata until `runPythonScript` calls the worker resolver.
- `env` mode passes plaintext only through subprocess environment, not argv.
- `file` mode writes plaintext under the attempt work directory and passes only the file path through env.
- The redactor is registered before Python starts and is applied to stdout/stderr before evidence/log emission.
- Exact secrets in `GOET_OUTPUT_JSON` cause failure instead of becoming dependency output.

Questions I should answer:

- Should future file-mode declarations explicitly say whether the path is exposed through env or input JSON?
- Should log files be streamed through a redacting writer instead of rewritten after process exit?
- Should target env names be restricted further than "no whitespace and no equals"?

Recommended next test or tiny experiment:

- Add a controlled sink test in OS-007 that drives a printed Python secret through log observation fallback and verifies the fallback JSONL does not contain plaintext.
