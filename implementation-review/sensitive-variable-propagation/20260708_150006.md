# Implementation Review: Sensitive Variable Propagation OS 008

Date: 2026-07-08

## Intended Design

Strategic Concept: `docs/concepts/sensitive-variable-propagation/README.md`.

Operational Slice: `docs/concepts/sensitive-variable-propagation/008-credentialed-worker-fixture-smoke.md`.

Objective: prove the sensitive-variable boundary with fake worker-local credentials, no real providers, and no large data.

Active HCI: `EC-3 / operational slice / files(6)+test+doc`.

Explicit acceptance criteria from OS 008:

- a small runbook explains how to run the fixture smoke;
- worker-local env secret is available to the worker but not the controller;
- controller can submit, compile, and assign work containing only the protected ref;
- worker resolves the protected ref and executes the fixture;
- captured output does not contain the raw fixture secret;
- controller status does not contain the raw fixture secret;
- persisted completion/failure evidence does not contain the raw fixture secret;
- the runbook states this proves the sensitive boundary, not a real credentialed provider.

## Object / Transition Map

System sentence:

`worker_env:GOET_FIXTURE_TOKEN` transitions from non-secret controller metadata to worker-local plaintext materialization when the assigned Python work item declares `protected_ref` plus `materialize.env`, by `scripts/sensitive-variable-fixture-smoke.ps1` submitting a temporary workflow and starting a worker with `GOET_FIXTURE_TOKEN`, while preserving the invariant that the raw sentinel `goet-fixture-secret-008-do-not-persist` is absent from controller status, submission status, worker output, captured logs, controller logs, and SQLite persistence.

Persistent data objects:

- controller admitted source cache for the temporary demo workflow and Python script;
- controller SQLite persistence at `.run/controller/workflow-execution.sqlite`;
- worker output file `cmd/worker/.run/data/sensitive-variable-fixture-check.json`;
- worker captured stdout/stderr logs under `cmd/worker/.run/tmp/attempts/<attempt>/logs/`.

In-memory/config objects:

- temporary workflow JSON generated by `Write-FixtureFiles`;
- `model.WorkItem` containing `fixture_token` with `protected_ref`;
- worker process environment containing `GOET_FIXTURE_TOKEN`;
- Python subprocess environment containing materialized `FIXTURE_TOKEN`.

Preconditions:

- sibling `../go-etl-demo-project` exists;
- `go`, `curl.exe`, and Python are available;
- controller can bind `localhost:8080`;
- existing sensitive-variable implementation from OS 001-007 is present.

Failure states:

- missing Python/go/demo project stops the smoke before submission;
- fixture reports failure if token is missing, hash mismatch occurs, or the secret reaches argv/input JSON;
- script fails if raw secret appears in any checked controlled sink;
- script fails if redaction label is absent from captured logs, controller logs, or SQLite persistence.

## Transition Order Review

1. Temporary fixture source is created.

`Write-FixtureFiles` writes a Python script, workflow JSON, and submission JSON into `../go-etl-demo-project`. These are external fixture artifacts, not committed source files. The workflow declares `fixture_token` as a string protected reference with provider `worker_env`, key `GOET_FIXTURE_TOKEN`, and env materialization target `FIXTURE_TOKEN`.

Invariant: the generated submission payload is checked so it does not contain the raw fixture secret.

2. Controller starts without the secret.

The smoke clears `$env:GOET_FIXTURE_TOKEN` before `Start-Process go run ./cmd/controller`. It checks `/status` before and after submission and rejects any occurrence of the raw sentinel.

Invariant: the controller process does not inherit `GOET_FIXTURE_TOKEN` from the smoke shell.

3. Workflow is submitted and assigned.

The script posts the source-reference submission to `/workflow` and starts the worker separately after overriding `worker_config.worker_max_count` to `0`. This forces the assignment to happen through the explicit worker process started by the smoke.

Evidence: the passing smoke run reported a generated submission ID and one completed work item through `goet status --json`.

4. Worker resolves and materializes the secret.

The smoke sets `$env:GOET_FIXTURE_TOKEN` only immediately before `go run ./cmd/worker <temp-config>`. The generated Python script reads materialized `FIXTURE_TOKEN`, compares its SHA-256 to a public expected hash, and emits non-secret booleans.

Invariant: the script checks that the secret did not reach Python argv or `GOET_INPUT_JSON`.

5. Captured output and logs are checked.

The generated Python script deliberately prints the raw secret to stdout and stderr. The smoke verifies captured stdout/stderr do not contain the raw sentinel and do contain `${worker_env.GOET_FIXTURE_TOKEN}`. It also verifies the controller log endpoint contains the label and not the raw sentinel.

6. Persistence is checked.

The smoke scans `.run/controller/workflow-execution.sqlite` bytes using read/write file sharing. It fails if the raw sentinel appears and requires the redaction label to appear.

Invariant: persisted controller evidence can contain protected-ref labels, not plaintext.

## Concrete Trace

Input state:

- controller has no `GOET_FIXTURE_TOKEN`;
- worker process is later started with `GOET_FIXTURE_TOKEN=goet-fixture-secret-008-do-not-persist`;
- workflow contains only `protected_ref: {provider: worker_env, key: GOET_FIXTURE_TOKEN}`.

Operations:

1. `Write-FixtureFiles` creates `workflows/sensitive-variable-fixture.json`.
2. Controller starts with `cmd/controller/demo-config.json`.
3. Smoke posts the source-reference submission to `/workflow`.
4. Controller admits source, compiles one `python_script` work item, and persists/assigns protected-ref metadata.
5. Worker claims work from `/work/next`.
6. Existing worker Python materialization resolves `worker_env:GOET_FIXTURE_TOKEN` and passes plaintext as `FIXTURE_TOKEN`.
7. Python fixture compares SHA-256 and prints the secret to stdout/stderr.
8. Existing worker redactor rewrites captured stdout/stderr and reports completion.
9. Smoke checks output file, logs, statuses, and SQLite persistence.

Final state:

- submission status is `completed`;
- worker output has non-secret proof fields;
- captured and controller logs contain `${worker_env.GOET_FIXTURE_TOKEN}`;
- raw sentinel is absent from checked controlled sinks.

If the key precondition fails:

- without worker `GOET_FIXTURE_TOKEN`, existing worker materialization fails before Python starts, and the smoke does not mark the slice complete.

Not covered:

- POSIX shell version of the smoke. The OS allowed PowerShell and shell scripts but did not require both.
- Real HPCC/GDAL/Google Drive/rclone. OS 008 explicitly excludes these.
- Transformed, encoded, split, or artifact-file leaks. The strategic concept and runbook state these are outside phase-1 protection.

## Risks and Ambiguities

PowerShell dependency:

- The implemented smoke is PowerShell-only. This is acceptable for the current Windows repo context and existing smoke pattern, but a future Linux CI path would need `scripts/sensitive-variable-fixture-smoke.sh`.

Live SQLite byte scan:

- The script scans the live SQLite file with read/write sharing. This proves absence of the exact sentinel in persisted bytes at that moment, but it is still an exact-value byte scan, not semantic row inspection.

Temporary sibling files:

- The smoke writes into `../go-etl-demo-project` and deletes files afterward. If interrupted hard, those files may remain and should be deleted manually. They are intentionally not committed in go-etl.

Controller port:

- The smoke assumes localhost port `8080`, matching existing smoke scripts and demo config. A competing local process blocks the run.

## What I Should Understand Before Moving On

- `protected_ref` is durable controller metadata, not the secret.
- `GOET_FIXTURE_TOKEN` exists only in the worker process in this smoke.
- `FIXTURE_TOKEN` is the deliberate Python subprocess materialization.
- Raw exact-value leaks are checked across controlled sinks; transformed leaks and artifacts remain out of scope.
- OS 008 proves the boundary with a fake credential, not a real provider.
