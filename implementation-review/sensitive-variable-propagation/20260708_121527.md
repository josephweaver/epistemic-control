## Intended Design

Strategic Concept: `docs/concepts/sensitive-variable-propagation/README.md`

Operational Slice: `docs/concepts/sensitive-variable-propagation/001-sensitive-metadata-and-safe-rendering.md`

Objective from the OS:

- add sensitivity metadata to typed variables and resolved values;
- propagate sensitivity through ordinary resolution;
- make safe rendering the default for sensitive resolved values;
- stay inside `internal/variable`;
- avoid controller, worker, persistence, and subprocess changes.

Explicit non-goals from the OS include secret lookup, worker materialization, persistence changes, and end-to-end smoke coverage.

## EC Object / Transition Map

1. `Variable` transitions from declared typed expression to resolved typed value when `Resolver.Resolve` or `ParseLiteral` evaluates it, by merging declaration sensitivity with resolved expression sensitivity, while preserving the invariant that a sensitive dependency cannot be declassified by a non-sensitive destination.

Evidence:

- OS propagation rules in `001-sensitive-metadata-and-safe-rendering.md`
- `internal/variable/resolver.go`
- `internal/variable/literal.go`
- `internal/variable/sensitivity_test.go`

2. `ResolvedValue` transitions from ordinary typed value to safe-rendered diagnostic value when formatted or JSON-serialized, by redacting sensitive scalar plaintext and retaining only non-secret labels and provenance, while preserving the invariant that default rendering does not emit exact sensitive plaintext.

Evidence:

- OS acceptance criteria for safe rendering and safe JSON
- `internal/variable/sensitivity.go`
- `internal/variable/sensitivity_test.go`

3. Aggregate `ResolvedValue` transitions from non-sensitive to sensitive aggregate when any child is sensitive, by computing aggregate metadata in object/list constructors, while preserving the invariant that child sensitivity remains visible through field and index accessors.

Evidence:

- OS propagation rules for objects, lists, and accessors
- `internal/variable/variable.go`
- `internal/variable/accessor.go`
- `internal/variable/sensitivity_test.go`

## Transition-Ordered Review

1. Variable declaration parsing

Before:
- `Variable` held `Name` and `TypedExpression` only.

After:
- `Variable` also holds `Sensitive bool`.
- JSON round-trip now accepts and emits `sensitive`.

Code:
- `internal/variable/variable.go`

Invariant:
- Existing non-sensitive declarations still validate and decode the same way.

2. Literal resolution

Before:
- `ParseLiteral` returned only type/value/object/list information.

After:
- `ParseLiteral` calls `mergeSensitivity` after `parseLiteralExpression`.
- A declared sensitive literal now resolves as sensitive with deterministic provenance.

Code:
- `internal/variable/literal.go`
- `internal/variable/sensitivity.go`

Failure behavior:
- Literal parsing failures still return the original typed-expression parse errors.

3. Resolver propagation

Before:
- `resolveVariable` returned only the expression result.
- Interpolation returned only the final string.

After:
- `resolveVariable` merges declaration sensitivity after expression resolution.
- `interpolate` also tracks whether any referenced component was sensitive and returns the first available redaction/provenance metadata.
- Whole-value references already carried the child `ResolvedValue`; now they also carry its sensitivity metadata unchanged.

Code:
- `internal/variable/resolver.go`

Invariant:
- `Sensitive:false` on the destination does not remove sensitivity inherited from a reference, accessor, list item, object field, or interpolation component.

4. Aggregate sensitivity and safe rendering

Before:
- `ResolvedObject` and `ResolvedList` only wrapped children.
- `ResolvedValue` had no rendering policy.

After:
- `ResolvedValue` now carries `Sensitive`, `RedactionLabel`, and `Provenance`.
- `ResolvedObject` and `ResolvedList` mark the aggregate sensitive if any child is sensitive.
- `ResolvedValue.String`, `ResolvedValue.GoString`, and `ResolvedValue.MarshalJSON` now default to safe rendering.

Code:
- `internal/variable/variable.go`
- `internal/variable/sensitivity.go`

Invariant:
- Sensitive scalar plaintext stays available in memory through `Value`, but default rendering surfaces do not expose it.

## Concrete Example Trace

Example from `TestResolverPropagatesSensitivityThroughObjectListAccessorAndInterpolation`:

1. Input declaration:
   - `workflow.token` is a sensitive string literal with value `goet-secret-sentinel-001`.
2. Resolver reads `workflow.record`, whose `token` field is `${token}`.
3. `resolveReference` resolves `workflow.token` to a scalar `ResolvedValue` with:
   - `Type=string`
   - `Value=goet-secret-sentinel-001`
   - `Sensitive=true`
   - `Provenance=workflow.token`
4. `resolveExpression` builds `record` as an object and `ResolvedObject` marks the aggregate sensitive because one child is sensitive.
5. `workflow.token_copy` resolves `${record.token}`; `ApplyFieldAccessor` returns the child `ResolvedValue`, preserving sensitivity.
6. `workflow.label` interpolates `"token=${token}"`; `interpolate` uses the plaintext in memory to produce `Value="token=goet-secret-sentinel-001"` but also returns the sensitivity metadata.
7. `label.String()` returns the redaction label, not the plaintext, so diagnostics stay safe.

If the precondition fails:
- Missing `token` still fails through the existing resolver error path.
- Type mismatch still fails through the existing `reference has type ... want ...` path.

## Hidden Coupling And Risk

1. Aggregate provenance selection is first-sensitive-child wins.
   - Location: `aggregateSensitivity` in `internal/variable/sensitivity.go`
   - Risk: multi-secret aggregates do not preserve all contributing labels.
   - Action: acceptable for this slice because the OS only requires a non-secret deterministic label, not full provenance sets.

2. Safe JSON shape is new and local to `ResolvedValue.MarshalJSON`.
   - Location: `internal/variable/sensitivity.go`
   - Risk: if another package later depends on a different serialized shape for `ResolvedValue`, that package will need an explicit serializer instead of generic `json.Marshal`.
   - Action: acceptable now because this slice is specifically about safe diagnostic serialization.

3. Plaintext still exists in-memory in `ResolvedValue.Value`.
   - Location: the model itself
   - Risk: callers can still leak by printing `Value` directly.
   - Action: consistent with the OS and concept README, which only require safe default rendering, not elimination of in-memory plaintext.

## Final EC Summary

### What I should understand before moving on

1. `Variable.Sensitive` is declaration metadata, not a new type.
2. `ResolvedValue` now carries sensitivity metadata and safe-render behavior.
3. Sensitivity propagation happens during literal parsing and resolver evaluation.
4. A reference or interpolation that depends on a sensitive source stays sensitive.
5. Default string and JSON rendering redact sensitive plaintext, but plaintext still exists in-memory for explicit use.

### Questions I should answer

1. Is first-sensitive-child provenance good enough for later slices, or will the concept need multi-source provenance?
2. Should later slices expose an explicit method for plaintext access instead of using `ResolvedValue.Value` directly?
3. Does any downstream package currently rely on `json.Marshal(ResolvedValue)` using the raw struct shape?
4. Should safe JSON always include a scalar `value` field even for zero values such as `false` or `0`?
5. When protected references arrive in slice `002`, should their redaction labels override declaration-derived labels?
