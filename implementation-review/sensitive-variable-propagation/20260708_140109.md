## 1. Intended Design

Strategic Concept: `docs/concepts/sensitive-variable-propagation/README.md`.

Operational Slice: `docs/concepts/sensitive-variable-propagation/005-trusted-go-workitem-sensitive-context.md`.

Objective: update trusted in-process Go work-item handlers so they receive public values and typed sensitive values through a worker operation context rather than through an unstructured parameter map.

Explicit non-goals from the OS:
- No Python subprocess materialization.
- No external secret stores.
- No controller persistence changes.
- No data asset credential use.
- No smoke tests.
- No arbitrary plugin sandboxing.

Acceptance criteria reviewed:
- Worker dispatch builds separated public and sensitive context values.
- Protected refs are resolved only for the assigned operation's declared needs.
- A trusted Go handler consumes a sensitive value in memory.
- Sensitive values are not converted to command-line arguments.
- Safe handler logging redacts registered sensitive plaintext.
- Formatting `OperationContext` does not print plaintext.
- Handlers without sensitive requirements do not receive secrets.
- Existing worker operations still function.
- Worker tests pass.

## 2. EC Object / Transition Map

Sentence 1:

`model.WorkItem` transitions from assigned worker payload state to `OperationContext` state when the work item type is a known trusted Go operation, by `Worker.operationContext`, while preserving the invariant that public values and sensitive values are stored in separate maps.

Objects:
- Persistent data object: none created by this slice.
- In-memory data objects: `model.WorkItem`, `model.ExecutionEnvelope`, `OperationContext`, `variable.ResolvedValue`, `SensitiveValue`, `Redactor`.
- Config objects: worker process environment for declared `worker_env` references.
- External artifacts: worker log file when handler logging occurs.
- Before: dispatch has a validated work item with parameters and optional execution envelope.
- After: trusted Go handler receives an `OperationContext`.
- Preconditions: work item type is `write_demo_output`, `summarize_input_file`, `cache_data`, or `commit_data`.
- Invariants: Python work items bypass this context path; unsupported work item types do not build a context.
- Evidence: `cmd/worker/worker.go`, `cmd/worker/work_context.go`, `TestWorkerRun`, `TestWorkerRunSummarizeInputFile`.

Sentence 2:

`ExecutionEnvelopeProtectedReference` transitions from unresolved reference to in-memory `SensitiveValue` when its variable name is listed in `trustedGoSensitiveNeeds(item.Type)`, by `WorkerEnvProtectedValueResolver.ResolveProtectedValue`, while preserving the invariant that undeclared protected refs are not resolved.

Objects:
- Persistent data object: none.
- In-memory data objects: `SensitiveValue`, `OperationContext.Sensitive`, `Redactor`.
- Config object: worker environment key named by the protected ref.
- Before: envelope contains protected refs.
- After: only declared names are in `OperationContext.Sensitive` and registered with `OperationContext.Redactor`.
- Failure states: missing declared worker env key returns an error before handler execution.
- Evidence: `work_context.go`; tests `TestOperationContextSeparatesPublicAndSensitiveValues`, `TestOperationContextDoesNotResolveSensitiveValuesForHandlerWithoutNeeds`, and `TestOperationContextRejectsMissingDeclaredWorkerEnv`.

Sentence 3:

`SensitiveValue` transitions from in-memory secret value to redacted worker-controlled log message when the trusted handler calls `secret.Plaintext()` and logs through `OperationContext.Logger`, by `workerSafeLogger.Log`, while preserving the invariant that registered plaintext does not leave through the worker log.

Objects:
- In-memory data objects: `SensitiveValue`, `Redactor`, `workerSafeLogger`.
- External artifact: `worker.log`.
- Before: `writeDemoOutput` has a `demo_secret` sensitive value.
- After: worker log contains the redaction label, not the plaintext.
- Evidence: `cmd/worker/work_demo.go`; test `TestTrustedGoHandlerConsumesSensitiveValueWithSafeLogging`.

## 4. Transition-Order Review

### Trusted Go dispatch

Input object: validated `model.WorkItem`.

Precondition: `runWorkItem` sees a known trusted Go work item type.

Operation: `runWorkItem` skips Python, rejects unsupported types, builds `OperationContext`, and passes it to `writeDemoOutput`, `summarizeInputFile`, `cacheData`, or `commitData`.

After state: handler entrypoints receive `OperationContext`; their existing internal behavior mostly continues through `ctx.WorkItem`.

Failure behavior: if context construction fails, dispatch returns `build operation context: ...`.

Evidence: `cmd/worker/worker.go`.

### Context construction

Input object: work item plus declared sensitive names from `trustedGoSensitiveNeeds`.

Precondition: dispatch asks for a context for a trusted Go operation.

Operation: `operationContext` uses the work item's existing execution envelope or builds one from parameters, converts public values into `variable.ResolvedValue`, resolves only declared protected refs, registers resolved plaintext with a redactor, and attaches `workerSafeLogger`.

After state: context has `Public`, `Sensitive`, `Redactor`, and `Logger`.

Failure behavior: missing declared `worker_env` key returns a sanitized resolver error.

Evidence: `cmd/worker/work_context.go`.

### Trusted handler consumption and logging

Input object: `OperationContext` passed to `writeDemoOutput`.

Precondition: `ctx.Sensitive["demo_secret"]` exists.

Operation: `writeDemoOutput` calls `secret.Plaintext()` and sends a message containing the plaintext to `ctx.Logger.Log`.

After state: `workerSafeLogger` redacts the registered plaintext before calling `Worker.log`.

Invariant: the direct plaintext use is visible at the call site through `Plaintext()`, and the log file does not contain the sentinel secret.

Evidence: `cmd/worker/work_demo.go`; `TestTrustedGoHandlerConsumesSensitiveValueWithSafeLogging`.

## 6. Concrete Trace

Initial input:

```go
model.WorkItem{
    ID: "demo-sensitive-001",
    Type: model.WorkItemTypeWriteDemoOutput,
    OutputFilename: "result.txt",
    Parameters: model.Parameters{
        "demo_secret": {
            Type: "string",
            ProtectedRef: &variable.ProtectedRef{
                Provider: "worker_env",
                Key: "GOET_DEMO_SECRET",
            },
        },
    },
}
```

Environment:

```text
GOET_DEMO_SECRET=plain-secret
```

Call sequence:
1. `Worker.Run(item)` calls `runWorkItem`.
2. `runWorkItem` identifies `write_demo_output` as trusted Go.
3. `trustedGoSensitiveNeeds` returns `[]string{"demo_secret"}`.
4. `operationContext` builds an envelope from parameters.
5. `WorkerEnvProtectedValueResolver` resolves `GOET_DEMO_SECRET`.
6. `OperationContext.Sensitive["demo_secret"]` receives `SensitiveValue`.
7. `OperationContext.Redactor` registers `plain-secret`.
8. `writeDemoOutput` calls `secret.Plaintext()`.
9. `writeDemoOutput` sends a log message containing the plaintext to `ctx.Logger.Log`.
10. `workerSafeLogger` redacts `plain-secret` to `${worker_env.GOET_DEMO_SECRET}` before writing `worker.log`.

Failed precondition:
- If `GOET_DEMO_SECRET` is absent, context construction fails and the handler does not run.

## 8. Risks and Ambiguities

Risk: `write_demo_output` now declares optional `demo_secret`.
- Location: `trustedGoSensitiveNeeds`.
- Impact: This is a minimal trusted-handler proof path, not a product credential feature.
- Action: acceptable for OS-005 because the allowed tests mention one small trusted Go handler receiving a fake sensitive value.

Risk: context public values preserve domain-specific types as `variable.Type{Kind: raw}`.
- Location: `operationValueType`.
- Impact: `cache_data`, `artifact_manifest`, and other domain types do not become invalid merely because they are not primitive variable types.
- Action: defer a richer public value abstraction until a later slice requires handlers to consume `ctx.Public` directly.

Risk: only handler entrypoints changed; many helper functions still accept `model.WorkItem`.
- Location: cache/commit/demo/summary internals.
- Impact: The top-level handler boundary is sensitive-aware, but internal refactoring is intentionally incomplete.
- Action: acceptable under the OS and HCI scope; future slices can migrate internal public parameter access if needed.

## 10. Final EC Summary

### What I should understand before moving on

- Trusted Go dispatch now builds `OperationContext`; Python dispatch does not.
- Sensitive refs are resolved by declared variable name, not by enumerating all protected refs.
- `write_demo_output` is the small trusted-handler proof path for in-memory sensitive consumption.
- The safe logger redacts values registered in the context redactor before writing the worker log.
- Context formatting reports metadata and names, not plaintext values.

### Questions I should answer

- Should `demo_secret` remain the long-term test proof path, or should a dedicated trusted test operation replace it later?
- Should public operation values get their own worker-local public value type instead of reusing `variable.ResolvedValue` for domain payload types?
- Which real trusted Go operation should first declare a production sensitive need?
