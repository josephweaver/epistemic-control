# Implementation Review: Sensitive Variable Propagation OS 007

Date: 2026-07-08

## Intended Design

Strategic Concept: `docs/concepts/sensitive-variable-propagation/README.md`

Operational Slice: `docs/concepts/sensitive-variable-propagation/007-controlled-sink-redaction-tests.md`

Objective: add focused sentinel tests proving that exact materialized sensitive values do not appear in GOET-controlled persistence, status, logs, events, worker errors, captured subprocess output, or structured output records.

HCI: `EC-3 / operational slice / files(6)+test+doc`

Constraints and non-goals:

- Prefer no production changes.
- Do not add new secret semantics unless tests expose a concrete gap.
- Do not scan artifacts by default; document the limitation.
- Do not test transformed or encoded leaks in this phase.

## Object / Transition Map

`ResolvedValue` transitions from sensitive plaintext in memory to redacted string or JSON rendering when `String`, `GoString`, or `MarshalJSON` is called, by using its safe rendering behavior, while preserving the invariant that the exact sentinel value is absent from rendered text.

`Python work item` transitions from materialized worker-env secret available to subprocess to redacted worker-owned failure/log records when the subprocess writes the secret to stdout/stderr and exits non-zero, by registering the materialized value with the attempt-local redactor and scrubbing captured logs before returning the failure, while preserving the invariant that the failure error and captured stdout/stderr do not contain the exact sentinel.

`Python work item` transitions from materialized worker-env secret available to subprocess to rejected structured output when `GOET_OUTPUT_JSON` contains the exact secret, by comparing output JSON against the redactor before promotion, while preserving the invariant that the logical output file is not persisted.

`Python artifact` transitions from staged artifact file to promoted artifact file when `GOET_OUTPUT_JSON` declares the artifact without containing the secret, by normal artifact promotion, while preserving the documented phase-1 limitation that artifact file contents are not scanned.

`Persistence rows` transition from queued/running work records to terminal completed/failed records when `CompleteAttempt` and `FailAttempt` are called with redacted text, by storing serialized row fields, while preserving the invariant that inspected row text does not contain the exact sentinel.

`Controller status JSON` transitions from persisted queued work to HTTP status response when `statusHandler` runs, by calculating counts from the workflow store, while preserving the invariant that the response JSON does not expose protected plaintext.

## Transition Review

Safe rendering:

- Test: `TestResolvedValueSafeRenderingRedactsControlledSinkSentinel`.
- Input object: `variable.ResolvedValue` with `Value` set to `goet-secret-sentinel-007-do-not-persist`, `Sensitive: true`, and a redaction label.
- Operation: `String`, `GoString`, and `json.Marshal`.
- Evidence: the test checks each rendered string with `strings.Contains(text, controlledSinkSentinel007)`.
- Design match: satisfies the acceptance criterion for safe variable rendering.

Worker failure and captured output:

- Test: `TestWorkerRunWorkItemControlledSinkSentinelRedactsFailureStdoutAndStderr`.
- Input object: Python `model.WorkItem` with a `worker_env` protected ref materialized into `GDRIVE_TOKEN`.
- Operation: Python script reads the env var, prints it to stdout and stderr, then exits `7`.
- Evidence: the test verifies the Go error does not contain the sentinel and both `stdout.log` and `stderr.log` contain the redaction label instead of the sentinel.
- Design match: satisfies worker failure path and captured stdout/stderr acceptance criteria.

Structured output rejection:

- Test: `TestWorkerRunWorkItemRejectsPythonOutputContainingMaterializedSecret`.
- Input object: Python `model.WorkItem` with materialized `GDRIVE_TOKEN`.
- Operation: Python writes `{"leak": secret}` to `GOET_OUTPUT_JSON`.
- Evidence: the test expects `GOET_OUTPUT_JSON contains a materialized sensitive value`, checks the error lacks the sentinel, and checks the output file was not written.
- Design match: satisfies structured output JSON rejection or non-persistence.

Artifact limitation:

- Test: `TestWorkerRunWorkItemDoesNotScanArtifactContentsForControlledSinkSentinel`.
- Input object: Python `model.WorkItem` with materialized secret.
- Operation: Python writes the secret into an artifact file while `GOET_OUTPUT_JSON` declares only the artifact metadata.
- Evidence: the test expects success, confirms `WorkEvidence.OutputJSON` lacks the sentinel, and confirms the promoted artifact file contains the sentinel.
- Design match: documents and tests the phase-1 artifact non-scanning limitation.

Controller status:

- Test: `TestStatusHandlerControlledSinkSentinelOmitsProtectedPlaintext`.
- Input object: persisted work item with a protected reference payload and no plaintext sentinel.
- Operation: `statusHandler` returns controller status JSON from the persistence-backed controller.
- Evidence: the raw HTTP body is inspected for absence of the sentinel and decoded to verify expected counts.
- Design match: satisfies the controller status acceptance criterion. It proves the status response does not add or expose protected plaintext, but it does not inject plaintext into controller storage because earlier slices intentionally reject that state.

Persistence:

- Test: `TestStoreControlledSinkSentinelPersistsOnlyRedactedSerializedRecords`.
- Persistent objects: `workflow_instances`, `work_items`, `work_item_attempts`, `completed_work`, and `failed_work`.
- Operation: insert protected-reference work-item payloads, claim work, complete one attempt with redacted output JSON, fail one attempt with redacted error text, then query raw row text.
- Evidence: concatenated row text is checked for absence of the sentinel and presence of the redaction label.
- Design match: satisfies the persistence-oriented acceptance criterion where practical.

## Concrete Trace

Example: `TestWorkerRunWorkItemControlledSinkSentinelRedactsFailureStdoutAndStderr`.

Initial input:

- Env var `GOET_TEST_CONTROLLED_SINK_FAILURE_SECRET=goet-secret-sentinel-007-do-not-persist`.
- Work item parameter `gdrive_token` is a `worker_env` protected reference materialized as env var `GDRIVE_TOKEN`.

Execution:

1. `Worker.Run` dispatches the Python work item.
2. `runPythonScript` builds an execution envelope if needed.
3. `materializePythonProtectedRefs` resolves the worker env secret, appends `GDRIVE_TOKEN=<secret>` to subprocess env, and registers the plaintext in the attempt redactor.
4. Python prints the secret to stdout and stderr, then exits with code `7`.
5. `runPythonScript` closes log files, calls `scrubPythonSubprocessLogs`, and returns a failure error.
6. The test reads the captured log files and checks that exact sentinel text is absent and the worker-env redaction label is present.

Failure precondition:

- If the redactor is not registered, stdout/stderr logs would still contain the sentinel and the test would fail.

## Test Coverage

Covered edge cases:

- Sensitive scalar safe rendering through `String`, `GoString`, and JSON.
- Python subprocess failure after the secret has been materialized.
- Captured stdout and stderr containing the exact secret.
- Structured output JSON containing the exact secret.
- Non-persistence of rejected structured output.
- Raw SQLite row text for workflow-run context, work-item payload, attempts, completed output JSON, and failed error.
- Artifact content deliberately not scanned.
- Controller status JSON from persistence-backed status handler.

Not covered:

- Transformed, encoded, hashed, or partial secret leaks. This is explicitly out of scope for OS 007.
- External service logs and network exfiltration. This is outside the phase-1 guarantee in the Strategic Concept.
- In-process Go handler log events with the exact OS 007 sentinel. Existing slice 005 tests cover safe logger behavior, but this OS implementation did not add a new sentinel test there.
- Controller dependency status failure reasons with a materialized secret. Controller-side plaintext materialized secrets should not exist under the current contract, so manufacturing that state would be misleading.

## Risks And Ambiguities

- The controller status sentinel test proves the status JSON omits a sentinel that is not stored in controller persistence. This matches the phase-1 protected-reference model, but it is weaker than a hypothetical test where controller storage contained plaintext and status redacted it. That stronger state should remain invalid, not normalized.
- The persistence test calls the persistence layer with redacted values. The persistence layer itself is not a secret scanner. The worker/controller boundary must reject or redact before persistence.
- Artifact non-scanning is intentionally proven by persisting a sentinel in a temp artifact. This is acceptable as a phase-1 limitation test, but reviewers should understand it is documenting a known weaker boundary.

## What I Should Understand Before Moving On

- Controller-side persistence stores protected references and redacted labels, not materialized worker secrets.
- Worker-side subprocess execution may materialize a secret, but GOET-owned failure strings and captured logs must not retain the exact plaintext.
- `GOET_OUTPUT_JSON` is a controlled dependency-output surface and is rejected when it contains an exact materialized secret.
- Artifact file contents are intentionally not scanned in phase 1.

## Questions I Should Answer

- Should dependency status failure reason rendering get a specific redaction regression test, even though controller plaintext is invalid?
- Should artifact non-scanning be called out in user-facing docs beyond this test and concept README?
