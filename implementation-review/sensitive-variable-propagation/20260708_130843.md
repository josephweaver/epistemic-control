## Intended Design

Objective: preserve public resolved values and sensitive protected references through controller compilation, assignment payloads, workflow-run snapshots, status output, and fingerprints without persisting plaintext secrets.

Constraints and non-goals:

- Do not add worker secret lookup or subprocess materialization.
- Do not persist plaintext sensitive values in work-item rows, attempt rows, status JSON, or workflow-run snapshots.
- Reject or explicitly defer unsupported client-env/client plaintext propagation.
- Existing non-sensitive workflows must still compile and run.

## Object / Transition Map

`model.WorkItem` transitions from flat parameter payload to safe assignment payload when the controller calls `WithExecutionEnvelope`, by creating `goet/execution-envelope/v1`, while preserving the invariant that public values are in `variables.public` and protected references are in `variables.protected_refs`.

`model.Parameter` transitions from resolved parameter to persistence-eligible parameter when `Parameter.Validate` runs, by accepting public `value` or protected `protected_ref` but rejecting sensitive plaintext values, while preserving the invariant that controller persistence cannot store a sensitive value without a protected reference.

`workflow` fan-out parameter binding transitions from `variable.ResolvedValue` to `model.Parameter` when `bindParameterAccessors` copies value metadata, by preserving `Sensitive`, `RedactionLabel`, and `ProtectedRef`, while preserving the invariant that protected references are not collapsed into `Value`.

`workflow_instances.submission_context_json` transitions from proposed run snapshot to persisted run snapshot when `workflowRunRecordFromAdmittedManifest` calls `safeWorkflowRunSubmissionVariables`, by rejecting sensitive plaintext submission variables, while preserving the invariant that client-submitted plaintext secrets do not become durable run state.

## Execution Order Review

1. Fan-out compilation resolves parameter accessors through `internal/workflow/fanout.go`.
   - Public values keep `Value`.
   - Protected-reference values keep `ProtectedRef`, `Sensitive`, and `RedactionLabel`.
   - Sensitive plaintext values remain marked sensitive and are rejected later at controller persistence.

2. Controller work-item persistence builds the execution envelope.
   - `cmd/controller/workflow_stage_queue.go` calls `WithExecutionEnvelope` before hashing and JSON persistence.
   - `cmd/controller/main.go` does the same for raw work submission and legacy metadata helper output.
   - Work-item and input fingerprints use `ExecutionEnvelope.Variables`, so protected-reference identity can affect hashes only through provider/key/redaction/materialization metadata.

3. Controller assignment returns an envelope to workers.
   - `nextPersistedWorkHandler` rebuilds the envelope after cache/commit-data hydration and before encoding `/work/next`.
   - This keeps assignment JSON aligned with any public parameters added during hydration.

4. Workflow-run snapshot creation rejects submitted sensitive plaintext variables.
   - `safeWorkflowRunSubmissionVariables` rejects `Sensitive && ProtectedRef == nil`.
   - Protected-reference variables remain valid snapshot inputs because they contain non-secret reference metadata.

## Acceptance Criteria Check

- Controller compilation preserves protected references without resolving them: implemented through fan-out metadata preservation and envelope construction.
- Assignment payloads sent to workers contain protected references, not plaintext: implemented in `/work/next`; tested by `TestNextWorkHandlerReturnsExecutionEnvelopeProtectedRefsWithoutPlaintext`.
- Controller status output shows redacted labels, not plaintext: status does not include parameter payloads; assignment and model JSON expose redaction labels for protected references.
- Workflow-run snapshots and work-item records do not contain plaintext sentinel secrets: tested by persistence and submission-context rejection tests.
- Fingerprints exclude plaintext sensitive values: sensitive plaintext parameters are rejected before hashing; protected-reference fingerprints use non-secret envelope variables.
- Protected-reference identity in fingerprints is documented: recorded in the OS implemented state section.
- Unsupported client-env-to-remote-worker secret propagation is rejected or deferred: existing variable protected-ref validation rejects `client_env`; submitted sensitive plaintext variables are now rejected.
- Existing non-sensitive workflows still compile and run: `go test ./...` passed.
- Relevant controller and persistence tests pass: controller/model/workflow tests and full repo tests passed.

## Risks And Deferred Edges

- Source-controlled workflow files may still contain sensitive plaintext if a user commits it. This slice prevents controller-generated work-item and submitted-variable persistence from converting sensitive values into ordinary controller state, but it does not police customer source repositories.
- Workers still consume the legacy `parameters` map. The envelope is present now, but worker-side secret lookup and materialization remain deferred to later slices.
- `materialize` validation currently accepts `env` only. That matches the OS example and avoids implementing later file/stdin materialization behavior.
