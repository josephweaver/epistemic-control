## Execution Order

1. `runPythonScript` stages the source bundle under `TmpDir/attempts/<attempt_id>`.
2. Before building `input.json` or starting `exec.Command`, `runPythonScript` calls `Worker.materializeDataAssets(item, staging.WorkDir)`.
3. `materializeDataAssets` reads concrete assets from `parameters.data_assets`, validates each as `model.BoundDataAsset`, and materializes each asset.
4. Reference assets resolve `<configured-root>/<safe-relative-path>`, hash the file, verify optional expected size/SHA-256, and return a manifest entry.
5. Worker-cache assets derive or validate a cache key, use `<asset-cache-root>/<cache-key>/source`, and either verify a cache hit or stream/copy a cache miss into a temporary file before rename.
6. Cache misses write `manifest.json` with pinned source size and SHA-256. Cache hits rehash `source` and compare it to `manifest.json`.
7. Successful materialization writes `data-assets.json` in the attempt work directory using schema `goet/materialized-data-assets/v1`.
8. Python receives `GOET_DATA_ASSETS_JSON=<attempt-work-dir>/data-assets.json` only when assets exist.

## Changed Config Fields

`Config.AssetCacheDir` in `cmd/worker/config.go`

- What it does: optionally overrides the worker asset cache root.
- Why it appears to exist: OS 006 allows an asset-cache-root setting and specifies a cache shape under `DataDir/cache/assets`.
- Assumption: if omitted, `DataDir/cache/assets` is acceptable for local worker cache.
- Risk if wrong: deployments that require cache on separate storage must configure `asset_cache_dir`; default cache writes may land on the wrong filesystem.
- OS comparison: fully implemented for worker-side cache root selection.

`Config.MaxAssetBytes`

- What it does: sets the maximum bytes copied, downloaded, or hashed for local/test materialization.
- Why it appears to exist: OS 006 requires a small configurable maximum download/copy size for tests and local runs.
- Assumption: zero means use a conservative default of 5 MiB.
- Risk if wrong: real data runs need an explicit larger limit, or materialization will fail before plugin execution.
- OS comparison: fully implemented for local-file copy, HTTP streaming, and reference hashing.

`Config.DataLocationRoots`

- What it does: maps data-location names to worker-visible filesystem roots.
- Why it appears to exist: OS 006 requires `local_file` and `registered_location` to resolve configured named roots plus safe relative paths.
- Assumption: root names in bound assets exactly match config map keys.
- Risk if wrong: compiler or deployment config naming drift causes materialization failure before Python starts.
- OS comparison: fully implemented for named-root resolution.

## New Worker Functions and Types

`Worker.materializeDataAssets`

- What it does: converts `parameters.data_assets` into a materialized manifest file under the attempt work directory.
- Why it appears to exist: OS 006 requires the worker to write a `goet/materialized-data-assets/v1` manifest and pass it to Python.
- Assumption: the current transport for concrete bound assets is `parameters.data_assets`; `model.WorkItem` has no top-level bound-asset field.
- Risk if wrong: future compiler work may need to move this data to a first-class field and update this reader.
- OS comparison: implemented differently at the carrier boundary, but the worker materialization behavior is implemented.

`assetMaterializer.materializeReference`

- What it does: resolves `local_file` or `registered_location` assets under a configured root without copying bytes, then records size and SHA-256 evidence.
- Why it appears to exist: OS 006 requires reference mode for read-only shared data.
- Assumption: referenced assets in this slice are regular files, not directories.
- Risk if wrong: directory assets or extracted selections require later slice support.
- OS comparison: fully implemented for tiny file fixtures.

`assetMaterializer.materializeWorkerCache`

- What it does: fills or reuses `<cache-root>/<cache-key>/source` and writes/reads `<cache-root>/<cache-key>/manifest.json`.
- Why it appears to exist: OS 006 requires worker-cache materialization with immutable reuse behavior.
- Assumption: local process atomic rename is sufficient for the fixture/local worker cache.
- Risk if wrong: cross-worker concurrent cache fills on shared filesystems may need locks in a later slice.
- OS comparison: fully implemented for single-worker fixture behavior; cross-worker locking remains out of scope.

`copyHashWithLimit`

- What it does: streams from an `io.Reader` through a fixed buffer, writes to a destination, hashes chunks, counts bytes, and rejects reads past the configured limit.
- Why it appears to exist: OS 006 requires streaming reads and maximum-size enforcement.
- Assumption: a 32 KiB buffer is adequate for fixture and local worker behavior.
- Risk if wrong: no correctness risk; only throughput tuning for real data.
- OS comparison: fully implemented for this slice.

`cleanDataRelativePath`

- What it does: rejects empty, absolute, drive-qualified, backslash, `.`, `..`, empty-segment, and unclean slash-relative paths.
- Why it appears to exist: OS 006 requires unsafe registered-location and local-file relative paths to fail before filesystem reads.
- Assumption: worker-side rules should match the defensive posture of the model/artifact validators.
- Risk if wrong: overly strict paths with trailing slash or repeated slash fail; that is acceptable for file assets in this slice.
- OS comparison: fully implemented for local-file/cache-key/registered-location paths used here.

## Concrete Trace

Example: local-file worker-cache asset with cache key `local/cache-key`.

1. Test creates `DataLocationRoots["fixture"] = <temp>/fixture` and writes `<temp>/fixture/input.txt`.
2. Work item carries `parameters.data_assets` with `provider=local_file`, `location_name=fixture`, `path=input.txt`, `strategy=worker_cache`, and `cache_key=local/cache-key`.
3. `boundDataAssetsFromWorkItem` JSON-decodes the parameter into `[]model.BoundDataAsset` and calls `BoundDataAsset.Validate`.
4. `materializeWorkerCache` maps the cache key to `<DataDir>/cache/assets/local/cache-key/source`.
5. On miss, `resolveNamedLocationPath` verifies `input.txt` is safe and inside `<temp>/fixture`.
6. `copyFileWithLimit` streams the fixture into a temp file, hashes it, then renames it to `source`.
7. `writeCacheManifest` records schema `goet/worker-asset-cache/v1`, byte count, SHA-256, provider, cache key, and immutability.
8. `materializeDataAssets` writes `data-assets.json` with schema `goet/materialized-data-assets/v1`.
9. `runPythonScript` adds `GOET_DATA_ASSETS_JSON` to the subprocess environment.

## Edge Cases Covered

- Missing configured root fails.
- Bad bound asset validation fails before filesystem access.
- Hash mismatch fails before Python starts.
- Size mismatch fails before Python starts.
- HTTP response over the max byte limit fails and removes the incomplete file.
- Cache source tampering fails on the next cache hit.
- Existing cache with a different expected hash fails instead of replacing immutable bytes.

## Edge Cases Not Covered

- HTTP non-2xx response is implemented but not tested.
- Missing cache manifest with existing source is implemented as failure but not directly tested.
- Explicit `immutable: false` behavior is not separately tested.
- Concurrent cache writes are not tested; OS 006 declared cross-worker locking out of scope.
- Directory references are rejected by implementation and not tested because this slice uses tiny file fixtures.
- HTTP client timeout/retry policy is not implemented; real data transfer policy is deferred.

## Ambiguity and Maintenance Risks

- Carrier ambiguity: `parameters.data_assets` is a worker-local bridge because `model.WorkItem` has no top-level `DataAssets` field. This should be resolved when controller/compiler slices start emitting concrete bound assets.
- Cache mutability ambiguity: the implementation always verifies and reuses an existing cache hit. It does not implement mutable refresh behavior for `immutable: false`; OS 006 acceptance criteria focus on immutable behavior.
- Reference hashing: reference mode hashes files and applies `max_asset_bytes`. That satisfies tiny-fixture evidence, but large shared read-only datasets will need explicit configuration or later evidence policy.
- HTTP uses `http.Get` with the default client. That is enough for local `httptest`; production downloads will need timeout/retry controls in a later data-provider slice.

## What I should understand before moving on

- The worker now owns data-byte acquisition for `local_file`, `http`, and `registered_location`; the controller still does not download or open those assets.
- Cache identity is either an explicit safe `cache_key` or a derived SHA-256 identity from the bound asset declaration.
- A cache manifest pins bytes even when no expected hash is supplied.
- `GOET_DATA_ASSETS_JSON` is only set after successful worker-side verification.
- Archive extraction, rclone, data-argument interpolation, and publication are still not implemented.

## Questions I should answer

- Should bound assets remain in `parameters.data_assets`, or should `model.WorkItem` gain a first-class `DataAssets []BoundDataAsset` field before compiler integration?
- Should reference-mode evidence hash full large files by default, or should large shared references support metadata-only evidence with explicit policy?
- What should `immutable: false` mean in worker-cache mode?
- Should HTTP provider behavior get a configured timeout before real public data smoke runs?
- Should cache manifests be considered public worker evidence or private worker cache metadata?
