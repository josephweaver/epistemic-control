## Changed Config Field

`Config.SevenZipExecutable` in `cmd/worker/config.go`

- What it does: loads optional JSON field `seven_zip_executable`.
- Why it appears to exist: OS 007 requires `archive.type: "seven_zip"` to fail clearly without a configured extractor and to use a configured external executable when present.
- Assumption encoded: bare executable names such as `7z` should stay PATH-resolved, while relative path-like values such as `tools/7z` should resolve relative to the config file.
- What could break: a user who expected `"7z.exe"` to resolve relative to the config directory will instead get PATH lookup behavior.
- OS comparison: fully implements the missing-extractor reservation and structured-executable configuration path; it does not prove real 7z extraction with a real archive, which the OS explicitly excludes from default tests.

## Changed Materialization Path

`assetMaterializer.materialize` and `assetMaterializer.extractArchive` in `cmd/worker/data_asset_materializer.go`

- What it does: materializes the source asset as before, then if `asset.Archive` is present, extracts selected archive members and replaces `MaterializedDataAsset.LocalPath` with the selected file or directory.
- Why it appears to exist: OS 007 defines archive extraction as a worker materialization transform after source acquisition.
- Assumption encoded: cache-backed archives extract under `<asset-cache>/<cache-key>/extracted`; reference-mode archives extract under `<workDir>/data-assets/<binding>/extracted`.
- What could break: repeated reference-mode materialization in the same work directory clears that binding extraction root before re-extracting.
- OS comparison: fully implements ZIP archive materialization for `local_file` and `http`; registered-location archives should follow the same reference path but are not separately tested.

## New Extractor Functions

`extractArchiveSelection` in `cmd/worker/archive_extractor.go`

- What it does: validates the archive declaration, cleans selectors, prepares the extraction root, and dispatches to ZIP or seven_zip extraction.
- Why it appears to exist: centralizes the archive transform so the materializer remains provider-oriented.
- Assumption encoded: every extraction starts from a cleared extraction root.
- What could break: concurrent materialization of the same cache key in the same worker cache could race because this slice did not add cache/extraction locking.
- OS comparison: implemented as specified for local tests; cross-worker locking was not requested.

`extractZIPSelection`

- What it does: opens the ZIP central directory, validates every ZIP entry name, and streams only selected file entries to selected output paths.
- Why it appears to exist: OS 007 requires safe ZIP member extraction and rejection of zip-slip/path traversal attempts.
- Assumption encoded: unsafe unselected ZIP entries should reject the archive rather than be ignored.
- What could break: an otherwise usable third-party archive containing an unrelated unsafe entry will fail even if the selected member is safe.
- OS comparison: fully implements the safety criteria and “extract only selected member contents”; implemented more strictly than a selected-only entry validation.

`extractSevenZipSelection`

- What it does: fails when `seven_zip_executable` is missing; when configured, invokes `exec.Command(executable, "x", "-y", "-o<root>", source, members...)`, moves extracted member paths to selected `as` paths when needed, then verifies expected selected outputs.
- Why it appears to exist: OS 007 reserves Yan/Roy-style 7z extraction through an external executable without requiring 7z in default tests.
- Assumption encoded: 7z extracts the requested member path under the provided output root before the worker renames that path to the selector's `as` path.
- What could break: real 7z archives with unusual path normalization, duplicate names, or non-file selected members may need additional handling.
- OS comparison: implemented for the reserved external-executable boundary with fake-executable argv coverage; real 7z archive behavior remains out of default-test scope.

`directoryManifestEvidence`

- What it does: walks selected extraction directory files, records ordered `{path,size_bytes,sha256}` entries, and hashes canonical JSON of that ordered list.
- Why it appears to exist: OS 007 requires selected-directory evidence to use the same ordered file-manifest hashing convention as artifact directories.
- Assumption encoded: selected directory `selected_size_bytes` is the sum of selected extracted file sizes and `selected_sha256` is the directory manifest hash.
- What could break: empty selected directories are hashable; this should only occur when all selected members are optional and missing.
- OS comparison: implements the documented convention from the concept README.

## Concrete Trace

For local ZIP `archive.zip` with selector `{member:"2023_30m_cdls.tif", as:"cdl.tif", expose:"selected_path"}`:

1. `Worker.materializeDataAssets` decodes `parameters.data_assets` into `model.BoundDataAsset`.
2. `assetMaterializer.materialize` chooses `reference` and `materializeReference` verifies the source ZIP file under the configured `fixture` root.
3. `extractArchive` chooses work-dir extraction root `<work>/data-assets/input_data/extracted`.
4. `extractArchiveSelection` validates selectors and clears/creates the extraction root.
5. `extractZIPSelection` validates ZIP entry names, opens only `2023_30m_cdls.tif`, and writes it atomically to `cdl.tif`.
6. `archiveSelectionResult` sets `local_path` to the extracted `cdl.tif` and records selected/member size and SHA-256.
7. `Worker.materializeDataAssets` validates and writes `data-assets.json`.

## Ambiguities and Risks

- Real 7z archive layout behavior remains unproven; this is consistent with the OS default-test boundary but should be tested before Yan/Roy real-data use.
- Extraction roots are cleared before extraction. This keeps manifests deterministic, but there is no lock for concurrent extraction of the same cache key.
- ZIP validation rejects any unsafe ZIP entry, even unselected ones. This is conservative and may reject archives that selected-only extraction could otherwise use.
- The selected-directory manifest entries are not persisted separately, only their canonical hash. That matches current manifest shape but may limit debugging if a directory hash differs later.

## What I Should Understand Before Moving On

- Archive extraction is now part of worker data asset materialization, not plugin code and not controller code.
- The manifest `local_path` now points to extracted content when an archive declaration is present.
- Directory selected evidence is a hash of an ordered file manifest, not a raw directory hash.
- 7z is only reserved unless the deployment config supplies an executable and real behavior is later tested.

## Questions I Should Answer

- Should unselected unsafe ZIP entries reject the whole archive, or should the worker validate only selected entries?
- Should cache-backed extraction be locked before real concurrent workers use shared cache keys?
- Should selected-directory manifests include the ordered file list for debugging, or is the hash enough?
- Should a fake 7z structured-argument test be added before OS 008 or the Yan/Roy fixture slice?
- Should `seven_zip_executable: "7z.exe"` be PATH-resolved or config-relative on Windows?
