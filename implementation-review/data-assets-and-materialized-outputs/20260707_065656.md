### Change review

#### `validateArtifactManifestOutputJSON`

What it does:
Validates artifact manifests only when `output_json` is schema-tagged as `goet/artifact-manifest/v1`. Non-manifest logical outputs continue through the existing path unchanged.

Why it appears to exist:
The slice requires invalid artifact manifest shapes to fail only when the controller can safely distinguish manifest output from ordinary logical output. The top-level schema is that discriminator.

Assumptions it encodes:
- Artifact manifests are opt-in by top-level schema.
- Structural validation is sufficient; controller-side artifact byte reads remain forbidden.

What could break if the assumption is wrong:
- A producer that omits the schema will be treated as ordinary output, so it will not receive manifest validation or status summarization.
- A future manifest version will need an explicit recognizer update or it will also be treated as ordinary output.

Comparison to slice:
Fully implemented.

Concrete example:
`{"schema":"goet/artifact-manifest/v1","storage_scope":"worker_data","artifacts":[{"name":"summary","kind":"file","path":"reports/summary.json"}]}` is accepted and later persisted in canonical compact form.

Edge cases covered by tests:
- recognized valid manifest
- recognized invalid manifest with `..` path
- non-manifest JSON that still has a different schema

Edge cases not covered by tests:
- recognized manifest with invalid numeric fields such as negative `stage_index`
- malformed manifest where `schema` is present but non-string

Risks:
- Manifest recognition logic now exists in controller output code only. A later public API that also needs manifest summaries should reuse it rather than reimplementing schema detection.

#### `submissionArtifactOutputs` and `submissionStatusResponse`

What they do:
Build compact per-work-item artifact summaries from persisted terminal attempt output JSON and append them to submission status responses as `artifact_outputs`.

Why they appear to exist:
The slice requires status JSON to expose compact artifact evidence without dumping full manifests by default.

Assumptions they encode:
- Per-work-item summaries are sufficient for “find produced artifacts” status use.
- Artifact names in declared order are a stable enough summary.
- Status should summarize only completed non-skipped outputs.

What could break if the assumption is wrong:
- If operators need stage/step coordinates or artifact paths in status, the current summary is too compact.
- If a skipped completion should still surface prior artifact evidence, that evidence is currently omitted.

Comparison to slice:
Fully implemented for the stated compact-summary requirement.

Concrete example:
After one work item completes with a two-artifact manifest, `GET /submissions/<id>/status` returns the normal submission fields plus:

```json
"artifact_outputs":[
  {
    "work_item_id":"test-001",
    "artifact_count":2,
    "artifact_names":["summary","tile-output"],
    "storage_scope":"worker_data"
  }
]
```

Edge cases covered by tests:
- one completed manifest-backed work item
- omission of the raw `"artifacts"` array from default status output

Edge cases not covered by tests:
- multiple manifest-backed work items and ordering
- mixed manifest and non-manifest completions in the same submission
- persisted corrupt manifest JSON encountered during status read

Risks:
- `submissionStatus()` still returns `model.SubmissionStatus` without artifact summaries. The HTTP surface has the new data, but any internal caller re-encoding the typed struct will not automatically include `artifact_outputs`.

#### `completedWorkMatchesRequest`

What it does:
Treats repeated completion reports with identical terminal evidence as the same completed attempt even if their fallback `completed_at` differs.

Why it appears to exist:
The added manifest validation work made an existing timing-sensitive idempotency assumption visible during `go test ./cmd/controller`.

Assumptions it encodes:
- Duplicate completion identity is defined by attempt ID plus terminal evidence, not by an implicit controller-generated completion timestamp.

What could break if the assumption is wrong:
- A caller cannot use a changed `completed_at` alone to distinguish two otherwise identical completion reports.

Comparison to slice:
Implemented differently than specified only in the sense that this is a regression-prevention adjustment, not a user-visible slice feature. It is still directly justified by the slice acceptance criterion that existing non-artifact completion behavior must keep passing.

Concrete example:
Two `POST /work/complete` calls for the same `attempt_id` and same `output_json` now both return success even if the second request is built in a later second.

Edge cases covered by tests:
- duplicate dependency completion remains idempotent through the existing controller test

Edge cases not covered by tests:
- explicit duplicate requests that differ only by supplied `completed_at`

Risks:
- If later audit requirements treat `completed_at` as part of the immutable terminal identity, this relaxed match will need reevaluation.

### What I should understand before moving on

- Artifact manifest handling is schema-gated. No schema means ordinary logical output.
- The controller still does not read artifact bytes or artifact directories; it only validates the declared manifest shape and safe relative paths.
- Status summarization is currently an HTTP-response extension, not a new persisted table or a new typed submission-status model.

### Questions I should answer

- Should `artifact_outputs` become part of the shared typed submission-status model so the CLI can preserve it automatically?
- Do future operators need artifact paths or run/stage/step coordinates in the summary, or is count/name/scope enough?
- Should future manifest versions be recognized by a version family rule or remain exact-schema opt-in?
