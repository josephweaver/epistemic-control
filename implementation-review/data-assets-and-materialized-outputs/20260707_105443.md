## Implementation In Execution Order

1. The smoke scripts create tiny fixture input data under `.run/fake-hpcc-data-assets`.
2. The scripts create temporary source-reference workflow files under `../go-etl-demo-project/.goetl-smoke/fake-hpcc-data-assets`.
3. The scripts create a controller config beside a copied `defaults.json`.
4. The controller starts with an execution environment using local transport, Bash dialect, Slurm scheduler, and `WorkerRuntime`.
5. `WorkerRuntime.Prepare` creates runtime directories, writes `runtime/config/worker.json`, and includes `asset_cache_dir` plus `data_location_roots`.
6. The controller compiles the source-reference workflow and starts worker capacity through `startConfiguredWorkers`.
7. `SlurmScheduler` submits the generated worker script to fake `sbatch`.
8. Fake `sbatch` runs the generated Slurm script in foreground for deterministic smoke behavior.
9. The worker materializes one plain data asset, copies and extracts one zip archive member, resolves Python `${data.<alias>.local_path}` args, runs the Python script, promotes one CSV artifact, and publishes it to `published_data`.
10. The scripts poll submission status and verify artifact-manifest evidence, promoted bytes, published bytes, and log paths.

## Changed APIs And Config Fields

`ExecutionComponentSettings` in `cmd/controller/execution_environment.go`

What it does: replaces `map[string]string` settings with `map[string]any` plus typed accessors `String` and `StringMap`.
Why it exists: worker runtime settings need nested `data_location_roots`.
Assumption: current settings are either strings or objects with string values.
Break risk: a future component needing numeric, bool, list, or nested object settings will need new accessors instead of ad hoc type assertions.
Design match: fully implements the slice need for non-private named roots in runtime settings.

`WorkerRuntime.DataDir`, `WorkerRuntime.AssetCacheDir`, `WorkerRuntime.DataLocationRoots`

What it does: lets controller runtime preparation write worker config values for data root, asset cache root, and named data location roots.
Why it exists: the worker already supports these fields in `cmd/worker/config.go`; the controller could not previously pass them through execution-environment config.
Assumption: named roots are deployment/runtime configuration, not workflow source constants.
Break risk: if a read-only root should never be created, `Prepare` currently calls directory creation for all named roots.
Design match: fully implements the OS requirement that the worker receives named roots inside the fake execution environment.

`WorkerConfig.AssetCacheDir` and `WorkerConfig.DataLocationRoots`

What it does: writes `asset_cache_dir` and `data_location_roots` into `runtime/config/worker.json`.
Why it exists: worker data materialization and publication use those config fields.
Assumption: JSON field names match `cmd/worker/config.go`.
Break risk: drift between controller-side and worker-side config structs would break smoke setup.
Design match: fully implemented and covered by `TestWorkerRuntimePrepareWritesWorkerConfig`.

`WorkerRuntime.createRuntimeDirs`

What it does: uses `os.MkdirAll` when the transport is `LocalTransport`; otherwise keeps the existing `mkdir -p` remote command.
Why it exists: the smoke starts a local controller process, and direct `mkdir -p` is not portable from Windows without a shell.
Assumption: local execution-environment paths are valid local filesystem paths after `filepath.FromSlash`.
Break risk: a caller using `LocalTransport` with remote-style absolute paths could create unexpected local directories.
Design match: implemented differently than a real HPCC path, but necessary for the local fake Slurm runbook.

`FakeHPCCWorkerScriptConfig` and `WriteFakeHPCCWorkerScriptConfig`

What it does: adds configurable script generation while keeping `WriteFakeHPCCWorkerScript`.
Why it exists: the old helper hardcoded `go run ./cmd/worker ./cmd/worker/demo-config.json`; the smoke needs generated worker config paths.
Assumption: fake-HPCC scripts should use the same `SlurmWorkerScriptConfig` contract as real Slurm scripts.
Break risk: the default helper now points at `/data/goetl/...`, so old tests or scripts expecting `go run ./cmd/worker` must use the configurable helper.
Design match: fully implements the need for a smoke-specific worker script without broad scheduler redesign.

`scripts/fake-hpcc-data-assets-smoke.ps1` and `.sh`

What they do: create fixtures, generate temporary workflow/config/submission files, start the controller, submit source-reference workflow, wait for completion, and verify evidence/files.
Why they exist: OS 011 asks for one repeatable fake-HPCC smoke path and a script when practical.
Assumption: local development can use fake `sbatch` as the minimum fake-HPCC scheduler boundary.
Break risk: PowerShell full run was not executed in this environment because `pwsh` is unavailable; Windows PowerShell parsed the help path only.
Design match: partially implements the OS wording around Singularity. It proves configured execution environment plus fake Slurm, but not SingularityCE image execution.

`docs/concepts/data-assets-and-materialized-outputs/fake-hpcc-data-assets-smoke.md`

What it does: records exact prerequisites, run commands, generated paths, proved behavior, and unsupported cases.
Why it exists: the OS acceptance criteria require a runbook and unsupported-case statement.
Assumption: real SSH, Dockerized Slurm, and Singularity are separate follow-up boundaries.
Break risk: if the team interprets OS 011 as requiring real Singularity now, this implementation is incomplete by that stricter reading.
Design match: documents the implemented local fake Slurm boundary honestly.

## Concrete Trace

Run:

```bash
bash scripts/fake-hpcc-data-assets-smoke.sh
```

The script writes `.run/fake-hpcc-data-assets/controller.json` and copies `cmd/controller/defaults.json` beside it. The controller loads that config, prepares `WorkerRuntime` rooted at `.run/fake-hpcc-data-assets/runtime`, and writes `runtime/config/worker.json` with `fixture_data` and `published_data`. The workflow binds `input_data` to `fixture-data/input.txt` and `archived_note` to `fixture-data/archive.zip`. The worker writes `reports/summary.csv` under `GOET_ARTIFACT_DIR`, reports it in `GOET_OUTPUT_JSON`, promotes it under `worker-data/artifacts/raw/fake-hpcc-data-assets-smoke/reports/summary.csv`, publishes it to `published-data/reports/summary.csv`, and writes the artifact manifest to `worker-data/fake-hpcc-data-assets-smoke.json`.

## Tests And Validation

Passing:

Edge cases covered:

- controller rejects non-string settings where strings are expected;
- runtime writes nested data roots to worker config;
- local runtime directory preparation does not depend on shell `mkdir -p`;
- default and configured fake-HPCC worker script generation;
- full smoke verifies archive extraction indirectly through Python input and artifact publication.

Edge cases not covered:

- full PowerShell smoke under PowerShell 7;
- real Dockerized Slurm;
- real SSH login-node transport;
- SingularityCE image execution;
- data root names with invalid semantic meaning but valid strings;
- protection against accidentally creating named roots that should be read-only.

## Ambiguity And Maintenance Risks

- The OS target state names Singularity worker runtime, but the implemented smoke uses fake Slurm plus a local Go worker process through the configured execution environment. The runbook records this as unsupported rather than claiming it is proven.
- `ExecutionComponentSettings` is intentionally minimal. It will need extension if later execution components require typed numeric or list settings.
- `WorkerRuntime.Prepare` creates all configured `data_location_roots`; this is convenient for fixture roots but may be too eager for production read-only roots.
- The live controller still hardcodes `local:demo` to the sibling demo project. The smoke depends on that development bridge.

## What I Should Understand Before Moving On

The new smoke proves that source-reference workflow admission, configured worker launch, worker data-root config, data materialization, archive extraction, artifact promotion, and publication work together through fake `sbatch`. It does not prove real HPCC, SSH, Dockerized Slurm, or Singularity.

## Questions I Should Answer

- Should named read-only data roots be created by `WorkerRuntime.Prepare`, or should missing read-only roots be treated as a preflight error?
- Should `ExecutionComponentSettings` grow typed helpers for int, bool, list, and nested component objects now, or only when a slice needs them?
- Should fake-HPCC smoke eventually run against Dockerized Slurm by default instead of local fake `sbatch`?
- Should `local:demo` repository mapping become controller config before the next smoke grows more fixtures?
- Should the PowerShell script continue to support Windows PowerShell, or should it require PowerShell 7 only?
