## Changed Public API and Behavior

### `internal/model.PublishedDataAsset`

- What it does: represents compact publication evidence with `name`, `from_artifact`, `storage_scope`, `location_name`, `path`, `size_bytes`, `sha256`, and `overwrite_policy`.
- Why it appears to exist: the slice requires publication evidence to be returned in the logical output rather than buried in ad hoc JSON.
- Assumptions encoded: publication is only for a registered named location, and the final target path is already a safe slash-relative path.
- What could break if the assumption is wrong: if later slices need other storage scopes, this evidence model will need either new scope values or a broader location object.
- OS comparison: fully implements the evidence shape the slice asked for, with the small difference that the evidence is typed and nested in the artifact manifest.

### `internal/model.ArtifactManifest.PublishedAssets`

- What it does: carries published-asset evidence next to promoted artifact evidence in the same logical output envelope.
- Why it appears to exist: the worker already returns artifact-manifest JSON after promotion, so publication evidence needed a typed home in that same document.
- Assumptions encoded: downstream code will read the artifact manifest for both artifact and publication evidence.
- What could break if the assumption is wrong: if a consumer only inspects `script_output`, it will miss publication evidence.
- OS comparison: implemented, though the slice did not require this exact placement; it only required compact logical output evidence.

### `internal/model.BoundPublishTarget.Target`

- What it does: accepts the slice’s `target` JSON spelling as an alias for the existing `target_name` field.
- Why it appears to exist: the slice document’s example uses `target`, while the worker model already used `target_name`.
- Assumptions encoded: `target` and `target_name` are synonymous when both are present, and conflicting values are rejected.
- What could break if the assumption is wrong: if later compiler code treats the two fields as distinct, the worker will reject that payload instead of guessing.
- OS comparison: implemented as a compatibility alias, not a redesign.

### `Worker.publishPromotedArtifacts`

- What it does: resolves publish bindings from `parameters.publish` or `parameters.publish_targets`, maps them to promoted artifacts by `from_artifact`, copies file or directory artifacts to a temp destination under the configured named location root, hashes the copied target, renames it into place, and rolls back completed targets if a later publish step fails.
- Why it appears to exist: OS 010 adds the worker-side `put_data` path without creating a registry or controller-side byte reader.
- Assumptions encoded: publish targets only reference artifacts that were already promoted, and the worker config already names the publish root.
- What could break if the assumption is wrong: concurrent writers can still race between the existence check and rename, and that race is not solved by this slice.
- OS comparison: fully implements the requested worker-side publication flow for tiny local fixtures.

### `Worker.runPythonScript`

- What it does: continues to stage source, run Python, promote artifacts, and now also publish selected promoted artifacts before marshaling the final logical output.
- Why it appears to exist: publication is a post-promotion worker concern, not a controller concern.
- Assumptions encoded: publication failure should fail the work item rather than return a success manifest.
- What could break if the assumption is wrong: a successful script with a bad publish target will now fail the whole work item, which is the intended behavior for this slice.
- OS comparison: matches the slice ordering requirement.

## Concrete Example Trace

For the file-artifact test case:

1. Python writes `GOET_ARTIFACT_DIR/reports/example.csv`.
2. `pythonLogicalOutput` promotes that artifact into `DataDir/artifacts/raw/<work-item>/reports/example.csv`.
3. `publishPromotedArtifacts` reads the `publish` parameter, resolves `published_data` to a configured root, and resolves the target path `reports/example.csv`.
4. The worker copies the promoted file into a temp path under the named location root.
5. The worker hashes the temp copy, renames it into place, and records `size_bytes` plus `sha256`.
6. The final `goet/artifact-manifest/v1` output includes one `published_assets` entry.

## Edge Cases Covered By Tests

- Unknown artifact name.
- Unknown named location root.
- Unsafe publish path.
- `fail_if_exists` when the target file already exists.
- File publication.
- Directory publication.
- Directory manifest hashing after copy.
- Manifest round-trip and validation for the new evidence shape.

## Edge Cases Not Covered By Tests

- Concurrent publication races between the existence check and rename.
- Partial rollback failure if a previously published target cannot be removed during cleanup.
- Cross-volume rename behavior on unusual filesystem setups.
- Mixed file and directory publication in one work item.
- Publish targets that use the compatibility `target` alias without `target_name` in the same test case.

## Ambiguities, Coupling, and Maintenance Risk

- The slice example uses `target` while the existing model used `target_name`; this implementation accepts both, but future compiler code should keep that contract explicit.
- Publication is currently modeled as a worker-local post-promotion phase, not as a first-class controller work item.
- The evidence lives inside `ArtifactManifest` because that was the smallest place to put it without introducing another logical-output envelope.
- The rollback behavior is best-effort and assumes a single worker owns the publish target path.

## Status Against OS

Fully implemented:

- Publish file artifacts to named locations.
- Publish directory artifacts to named locations.
- Validate target paths before filesystem writes.
- Reject unknown artifacts.
- Reject unknown named locations.
- Enforce `fail_if_exists`.
- Return size and SHA-256 evidence after copy.
- Preserve promoted artifact behavior when no publish target is present.

Implemented differently than the text example:

- The publication evidence is nested in `goet/artifact-manifest/v1` as `published_assets` rather than being emitted as a separate manifest type.

## What I Should Understand Before Moving On

- The worker now owns publication after promotion, just as it already owned artifact promotion after Python execution.
- Publication is path-safe and evidence-producing, but still scoped to a single worker process and a predeclared named location root.
- The new evidence type is intentionally compact; it records the final path and hash, not the full copy history.

## Questions I Should Answer

- Should future slices keep using `parameters.publish`, or should they move to a dedicated top-level work-item field?
- Do we want the `target` alias to remain permanent, or should the controller always emit `target_name`?
- Should multi-target publication become all-or-nothing across all rename steps, or is best-effort rollback sufficient?
- Should the manifest carry a separate publication kind field if later scopes include non-registered locations?
- Should publication evidence later be split out into its own manifest schema?
