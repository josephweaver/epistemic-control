# Implementation Review: OS-017 Commit Data Published Output Work Items

## Intended Design

Strategic Concept: Data Assets and Materialized Outputs.

Operational Slice: `docs/concepts/data-assets-and-materialized-outputs/017-commit-data-published-output-work-items.md`.

Objective: compile publish bindings into explicit `commit_data` work items that publish selected promoted artifacts to declared durable store locations.

HCI budget: `EC-3 / operational slice / files(10+)+test+doc`.

Explicit non-goals: catalog registration, arbitrary absolute publish paths, credentials, real Google Drive upload, destructive overwrite by default, cache_data implementation, and global retention/cleanup.

## Object / Transition Map

`BoundPublishTarget` transitions from a compute parameter to a downstream `commit_data` work item when workflow stage planning sees `publish` or `publish_targets`, by `workflow.PlanCommitDataWorkItems`, while preserving the invariant that the producing compute item remains the local artifact producer and the publish operation has its own dependency and resource constraints.

`commit_data` transitions from queued dependent work to executable worker payload when its producer has completed, by `Controller.enqueueReadyCacheDataDependents` and `Controller.hydrateCommitDataWorkItem`, while preserving the invariant that the worker consumes a completed `goet/artifact-manifest/v1` output rather than an uncompleted producer state.

`ArtifactDescriptor` transitions from worker-data artifact evidence to `PublishedDataAsset` evidence when `Worker.commitData` copies the artifact to a registered location, by `publishPromotedArtifactsForTargets`, while preserving safe relative paths, `fail_if_exists`, and target byte/hash verification.

## Transition Review

Planning:

- Object entering: compiled compute `model.WorkItem` with `publish` or `publish_targets`.
- Operation: `PlanCommitDataWorkItems`.
- Result: compute item no longer carries publish bindings; a generated `commit_data` item carries `commit_data` payload, `depends_on` producer ID, output filename, and a target-location resource constraint.
- Failure states: invalid publish parameter shape, invalid target, duplicate generated ID, or invalid resource constraint.

Controller runtime:

- Object entering: persisted producer completion and dependent `commit_data` worker payload.
- Operation: normal completion now calls the ready-dependent enqueue helper after dependency terminal recording; claimed `commit_data` items are hydrated from the completed producer output.
- Result: `artifact_manifest` is injected into the claimed worker payload.
- Failure states: source not listed in `depends_on`, completed source not found, or completed source output is not an artifact manifest.

Worker publish:

- Object entering: `commit_data` payload plus `artifact_manifest`.
- Operation: `Worker.commitData`.
- Result: selected artifact is copied to a configured registered location through a temp path, revealed as final target, verified, and emitted as `goet/published-data-assets/v1`.
- Failure states: missing artifact name, unsafe source/target path, unknown publish location, existing final target under `fail_if_exists`, size mismatch, or SHA mismatch.

## Concrete Trace

Input: compute item `compute-a` has a publish target `publish_summary` from artifact `summary`.

1. `CompileWorkflowStage` compiles the compute item.
2. `PlanCacheDataWorkItems` handles data inputs first.
3. `PlanCommitDataWorkItems` removes `publish`, creates `commit-data-<sha>`, sets `depends_on: ["compute-a"]`, and attaches `target:target-local/published-data-write:published_data <= 1`.
4. The controller initially queues only dependency-free work.
5. When `compute-a` completes with `goet/artifact-manifest/v1`, completion recording stores the output and enqueues the ready `commit_data` dependent.
6. `/work/next` claims `commit_data`, decodes its payload, finds the completed producer output, and adds `artifact_manifest`.
7. `Worker.commitData` validates payload and manifest, finds artifact `summary`, copies it to the named publish root under `reports/summary.csv`, verifies evidence, and reports compact published evidence.

If the artifact name is missing, `publishPromotedArtifactsForTargets` fails before writing the target.

## Test Coverage

Covered:

- `TestPlanCommitDataWorkItemsSplitsPublishBindingFromCompute`: planning creates explicit commit work, dependency, and resource constraint.
- `TestCompileWorkflowStagePlansCommitDataAfterCacheData`: stage compilation includes cache and commit planning together.
- `TestHydrateCommitDataWorkItemUsesCompletedProducerArtifactManifest`: controller hydrates commit payload from completed producer output.
- `TestCommitDataPublishesFileArtifactAndEmitsEvidence`: worker publishes a file and emits compact target evidence.
- `TestCommitDataRejectsMissingArtifactUnsafePathAndExistingTarget`: missing source artifact, unsafe path, and `fail_if_exists` failures.
- `TestStoreClaimNextWorkCommitDataPublishedLocationMutexAdmitsOnlyOne`: resource admission blocks concurrent commits to the same publish location.
- Model tests validate `CommitDataWorkItemPayload` and `PublishedDataAssetManifest`.

Not covered:

- Directory artifact `commit_data` through the new operator boundary. Directory copy helpers are covered by existing publication and artifact-promotion tests, but not through `Worker.commitData`.
- End-to-end source-reference smoke for explicit `cache_data -> compute -> commit_data`; this is deferred to OS-018.

## Risks And Ambiguities

- The OS allowed-file list did not name `cmd/controller`, but runtime consumption of completed artifact manifests required a controller bridge. The change is narrow but should be reviewed explicitly.
- The enqueue helper is still named `enqueueReadyCacheDataDependents` even though it now queues ready dependents after normal compute completion too. The behavior is correct for this slice, but the name is now misleading.
- `commit_data` hydration reads terminal attempts at claim time. This avoids speculative compile-time output injection, but a later retry/reuse design may want a more explicit dependency-output lookup API.
- Directory publish verification is implemented through existing directory manifest evidence but lacks a dedicated `commit_data` test in this slice.

## Budget And Slice Fit

The implementation fits the requested `files(10+)+test+doc` budget. It used more than the OS allowed-file examples because controller hydration was necessary to satisfy the acceptance criterion that `commit_data` consumes completed artifact manifests. New worker artifact-promotion files were included as build closure for already-tracked references.

## Final EC Summary

What to understand before moving on:

- `commit_data` is a separate work item, not compute cleanup.
- The producer dependency is explicit in `depends_on`.
- The worker receives the artifact manifest only after the producer completes.
- Publication is constrained by a target-location write mutex.
- `fail_if_exists` remains the only supported final-target overwrite behavior.

Questions to answer:

- Should the ready-dependent enqueue helper be renamed in a cleanup slice?
- Should directory `commit_data` get a dedicated test before OS-018?
- Should completed-output hydration move out of `cmd/controller/main.go` into a named dependency-output helper?

Recommended next test:

Run OS-018 as an integration smoke that proves `cache_data -> compute -> commit_data` from source-reference workflow submission through worker completion.

5. What happens when the final publish target already exists under `fail_if_exists`?
