### Change review

#### `ValidateArtifactRelativePath`

What it does:
Rejects empty, whitespace-padded, absolute, UNC-like, backslash, drive-qualified, dot-segment, dot-dot-segment, double-slash, and trailing-slash paths. Returns a clean slash-separated relative path.

Why it appears to exist:
The slice requires a structural validator in `internal/model` so controller and worker code can share one artifact-relative path contract.

Assumptions it encodes:
- Artifact paths are always slash-separated, even on Windows.
- Cleaned-path rewrites such as `tiles//out` or `tiles/out/` should fail instead of silently normalizing.

What could break if the assumption is wrong:
If a later slice intentionally wants to accept normalized-but-equivalent directory references, current validation will reject them and force upstream formatting changes.

Comparison to slice:
Fully implemented.

Concrete example:
`tiles/tile-001/report.parquet` passes because it is non-empty, relative, backslash-free, segment-clean, and unchanged by `path.Clean`.

Edge cases covered by tests:
- Windows-style `C:/...`
- Backslashes
- `.` and `..`
- double slashes
- trailing slash

Edge cases not covered by tests:
- lowercase drive prefixes such as `c:/...`
- leading/trailing tabs rather than spaces

Risks:
- `hasArtifactDrivePrefix` is duplicated logic rather than shared with `internal/reposource/path.go`, so future path-policy changes could drift.

#### `ArtifactManifest`, `ArtifactDescriptor`, `EffectiveSchema`, `Validate`

What they do:
Provide the transport shape and structural validation for artifact manifests, including default schema resolution and descriptor validation.

Why they appear to exist:
The slice needs a compact shared manifest model for files and directories that workers can report and controllers can persist later.

Assumptions they encode:
- Only `file` and `directory` are valid artifact kinds in this slice.
- `storage_scope` is required now even though persistence and worker promotion are out of scope.
- Empty `schema` should mean the default v1 schema rather than an error.

What could break if the assumption is wrong:
- A later producer that emits another kind such as `symlink` or `archive_member` will fail validation until the model is extended intentionally.
- If downstream code expects `Validate` to mutate `Schema`, it will not; callers must use `EffectiveSchema` explicitly.

Comparison to slice:
Fully implemented for the specified acceptance criteria.

Concrete example:
A worker can build:

```go
ArtifactManifest{
    StorageScope: "attempt",
    Artifacts: []ArtifactDescriptor{{
        Name: "summary",
        Kind: ArtifactKindFile,
        Path: "reports/summary.json",
        SHA256: "...",
    }},
}
```

`EffectiveSchema()` resolves to `goet/artifact-manifest/v1`, `Validate()` accepts the manifest, and the descriptor path is validated through `ValidateArtifactRelativePath`.

Edge cases covered by tests:
- valid file artifact with `sha256` and `size_bytes`
- valid directory artifact with `manifest_sha256`
- no-artifact manifest
- empty `name`
- empty `kind`
- empty `path`
- unsafe path inside a descriptor
- unsupported kind

Edge cases not covered by tests:
- negative `stage_index` and `step_index`
- negative `size_bytes` and `record_count`
- explicit unsupported `schema` value

Risks:
- The model distinguishes file and directory evidence by field names and accepted kinds, but it does not yet enforce mutually exclusive hash fields by kind. That matches the slice acceptance criteria, but later slices may want tighter invariants.

### What I should understand before moving on

- This slice only defines a shared transport contract and structural path safety helper in `internal/model`.
- It does not yet change worker execution, controller persistence, promotion, or published-asset behavior.
- `EffectiveSchema()` supplies the default schema value; `Validate()` checks compatibility but does not rewrite the stored field.

### Questions I should answer

- Should future slices enforce kind-specific evidence more strictly, such as rejecting `manifest_sha256` on `file` artifacts?
- Should path validation logic eventually be shared between `internal/model` and `internal/reposource` to avoid policy drift?
- Should a future manifest constructor set `Schema` eagerly so serialized output always includes the default explicitly?
