Objective: attach resource constraints and per-transfer limits to `cache_data` work items so inbound data movement is scheduled and bounded.

Explicit constraints:

- use existing integer resource constraints for admission;
- do not implement true request-per-minute rate limiting;
- use local HTTP fixtures and fake rclone tests;
- do not use shell strings for rclone;
- keep real Google Drive, real CDL downloads, credentials, token buckets, and distributed bandwidth accounting out of scope.

## Object / Transition Map

`BoundDataAsset` transitions from "resolved source asset without transfer policy" to "resolved source asset with optional transfer policy" when a provider template binds or a test/build path constructs the asset, by storing `DataAssetTransferPolicy`, while preserving existing provider, cache, archive, and integrity validation.

Evidence: `internal/model/data_asset.go`, `internal/model/data_provider.go`, `internal/model/data_asset_test.go`.

`cache_data` compiled work item transitions from "deduplicated materialization work with no source admission fact" to "deduplicated materialization work with persisted source admission facts" when `PlanCacheDataWorkItems` creates a cache item, by deriving `WorkItemResourceConstraint` from provider/source identity and `transfer_policy.max_concurrent_source_transfers`, while preserving the existing resource-admission persistence path.

Evidence: `internal/workflow/cache_data_plan.go`, `internal/workflow/cache_data_plan_test.go`, `cmd/controller/workflow_stage_queue.go` existing stamping/persistence behavior.

`cache_data` payload transitions from "worker materialization facts only" to "worker materialization facts plus diagnostic/enforcement transfer policy" when planned, by carrying `resource_constraints`, `transfer_policy`, and `transfer_limits`, while preserving legacy zero-value payload compatibility.

Evidence: `internal/model/work_item.go`, `cmd/worker/work_cache_data.go`.

`provider acquisition` transitions from "unbounded per-transfer stream/copy" to "optionally bounded single-transfer stream/copy" when `max_bytes_per_second` is present, by sleeping in the copy loop through `dataAssetThrottleSleep`, while preserving max-size and hash-while-streaming behavior.

Evidence: `cmd/worker/data_asset_materializer.go`, `cmd/worker/data_asset_materializer_test.go`.

`gdrive_rclone acquisition` transitions from "structured rclone copyto args" to "structured rclone copyto args with an optional `--bwlimit`" when transfer policy requests it, by appending only the allowed bandwidth argument, while preserving no-shell invocation and redaction behavior.

Evidence: `cmd/worker/gdrive_rclone_provider.go`, `cmd/worker/gdrive_rclone_provider_test.go`.

## Execution Order Review

1. Model validation accepts optional transfer policy.

`DataAssetTransferPolicy.Validate` rejects negative concurrency, negative bandwidth, negative max bytes per second, empty provider arg names, unsupported provider arg names, and blank/whitespace-padded provider arg values. The only accepted provider arg is `rclone_bwlimit`.

Failure state: invalid policies fail before planner or worker execution.

2. Planner derives `cache_data` source constraints.

`CacheDataResourceConstraints` validates the bound asset, derives one provider/source resource key, sets `requested_units=1`, `operator<=`, and sets `target_units` from `max_concurrent_source_transfers` or default `1`.

Invariants:

- resource keys are deterministic for the same provider/source;
- different source keys do not share capacity;
- the current model remains integer admission only, not request-per-minute.

3. Planner stores constraints in both compiled item and payload.

The compiled item `ResourceConstraints` is what persistence already stores and claim-time admission already enforces. The payload copy gives worker/provider context and preserves the OS payload shape.

4. Persistence admission enforces the source capacity.

No persistence production code changed. Existing `ClaimNextWork` evaluates running units for matching resource keys. New tests prove the source-mutex and source-capacity behavior using provider-style resource keys.

5. Worker enforces per-transfer limits.

HTTP and local copy acquisition pass `asset.TransferPolicy.MaxBytesPerSecond` into the stream-copy helper. The copy loop hashes while streaming, writes to a temp path, and calls an injectable sleeper after chunks when a byte/sec cap is configured.

6. Rclone receives bandwidth policy.

`gdriveRcloneProvider.copyToArgs` appends `--bwlimit <value>` only from `provider_args.rclone_bwlimit`, `requested_bandwidth_mib_per_second`, or whole-MiB conversion from `max_bytes_per_second`. It still calls `exec.CommandContext` with an argument slice.

## Concrete Trace

Input: one HTTP bound asset from `https://example.invalid/2023_30m_cdls.zip` with `transfer_policy.max_concurrent_source_transfers=2`.

Planner path:

- `PlanCacheDataWorkItems` reads `parameters.data_assets`.
- `CacheDataAssetKey` computes the physical asset key.
- `CacheDataPayload` calls `CacheDataResourceConstraints`.
- `cacheDataProviderResourceKey` produces `provider:http:example.invalid/download`.
- The compiled cache item receives one constraint: requested `1 <= 2`.
- The compute item receives a dependency on the cache item.

Persistence path:

- Controller staging code stamps `WorkItemID` and `CreatedAt`.
- The existing store persists `work_item_resource_constraints`.
- `ClaimNextWork` admits up to two running work items for that source key and blocks the third until one completes.

Worker path:

- `cacheDataPayloadFromWorkItem` decodes payload and bound asset.
- `assetMaterializer.materialize` uses worker-cache acquisition.
- `downloadHTTP` streams to a temporary file, hashes bytes, applies max-size checks, optionally sleeps for byte/sec throttling, then renames into the cache path after a successful copy.

If the resource key is already at capacity, the candidate stays queued and an unconstrained compute item later in the queue can still be claimed.

## Risks And Notes

- Resource key sanitization lowercases and replaces unsupported characters with `-`. This is deterministic and avoids private/raw path segments, but deployments that need stronger aliases should still provide safe remote/root names.
- Only provider-source constraints are generated by default. Target cache-write/archive-extract resource keys from the OS naming section are not generated yet because default target-wide mutexes would make independent sources block each other. This matches the OS tests but leaves target/cache/archive limits for explicit future policy if needed.
- HTTP throttling uses sleep-after-chunk rather than a precise clocked token bucket. This satisfies the slice's per-transfer cap requirement for fixture/test behavior but is not a distributed or exact bandwidth controller.
- `max_bytes_per_second` to rclone bwlimit conversion uses whole MiB only when no explicit `rclone_bwlimit` or requested MiB/sec is provided. The explicit provider arg is the precise path.
- The concept README had pre-existing uncommitted restructuring in the worktree. The go-etl commit intentionally did not include those unrelated README changes.

## Final EC Summary

What to understand before moving on:

- Admission constraints decide how many `cache_data` items for one provider/source may run.
- Transfer limits decide how aggressive one admitted transfer may be.
- The controller enforces the former through existing resource-admission persistence.
- The worker/provider enforces the latter through stream throttling or rclone `--bwlimit`.
- This slice does not implement true request-per-minute limits or distributed bandwidth accounting.

Questions to answer:

- Should target-wide cache-write or archive-extract constraints be added as explicit configurable policy in a later slice?
- Should source aliases be separate declarative fields for private provider identities?
- Should HTTP throttling eventually use a more precise clocked rate limiter?

Recommended next test or tiny experiment:

- Add a fixture workflow with two HTTP sources and one shared source to observe queue behavior through the controller API, not only the persistence store.

Five short quiz questions:

1. Which field controls how many `cache_data` items for one source can be claimed?
2. Why does `max_bytes_per_second` not replace resource admission?
3. Where is the source resource constraint persisted?
4. Why do independent source keys avoid blocking each other?
5. How does the rclone provider receive bandwidth policy without a shell string?
