## Execution Order Summary

1. A worker receives a `model.WorkItem` whose `parameters.data_assets` contains a concrete `model.BoundDataAsset` with `provider: "gdrive_rclone"`.
2. `boundDataAssetsFromWorkItem` decodes and validates the asset. This uses existing model validation for remote names and slash-relative drive paths.
3. `materializationStrategy` now defaults `gdrive_rclone` to `worker_cache`.
4. `materializeWorkerCache` computes or uses the cache key and checks the existing cache source plus manifest first.
5. On cache miss, `acquireSource` dispatches to `acquireGDriveRclone`.
6. `acquireGDriveRclone` rejects disabled provider use, missing `rclone_executable`, and reserved `file_id` access before invocation.
7. `gdriveRcloneProvider.copyTo` builds `rclone copyto` arguments as a Go slice, optionally with `--config <path>`, and invokes `exec.CommandContext`.
8. The fake or real rclone writes to a temporary cache path. The worker hashes and size-counts that temporary file, enforces `max_asset_bytes`, then renames it to `source`.
9. Existing integrity checks verify expected SHA-256 and expected size. Existing cache-manifest logic records immutable evidence.
10. If the bound asset has `archive`, the acquired cached file enters the existing `extractArchive` path.
11. The worker writes `GOET_DATA_ASSETS_JSON` with materialized evidence before Python execution.

## Changed API, Struct, Function, Config, and Tests

### `Config` fields in `cmd/worker/config.go`

What it does: adds `RcloneExecutable`, `RcloneConfigPath`, and `EnableGDriveRcloneProvider` JSON-backed worker config fields.

Why it appears to exist: OS 008 says rclone executable, config path, and enablement are worker configuration concerns, not workflow JSON.

Assumption: a deployment can preconfigure rclone credentials and remotes outside GOET workflow declarations.

Could break if wrong: if deployments need per-workflow rclone flags or credentials, this deliberately refuses that model and would need a later sensitive-variable/configuration slice.

Specification match: fully implemented for the OS-required config boundary.

### `materializationStrategy` and `acquireSource` in `cmd/worker/data_asset_materializer.go`

What it does: defaults `gdrive_rclone` to `worker_cache`, rejects reference mode, and dispatches cache misses to `acquireGDriveRclone`.

Why it appears to exist: rclone downloads produce a worker-local source file that should reuse the same immutable cache/integrity/archive behavior as HTTP and local-file cache mode.

Assumption: `gdrive_rclone` assets should not be exposed as direct references.

Could break if wrong: a future mounted Google Drive backend would need a different provider or explicit reference semantics.

Specification match: fully implemented for path-based rclone acquisition into cache.

### `gdriveRcloneProvider` in `cmd/worker/gdrive_rclone_provider.go`

What it does: adapts a configured executable to `rclone copyto remote:path tempPath`, hashes the temp file, enforces max size after copy, promotes it to the cache source path, and redacts config paths and common token forms in failure output.

Why it appears to exist: OS 008 requires a narrow adapter, structured command arguments, no arbitrary workflow flags, clear subprocess failures, and redaction.

Assumption: path-based `remote:path` access is enough for this slice; `file_id` is reserved.

Could break if wrong: real Shared Drive runs that require file-ID addressing will fail clearly and need a later adapter extension.

Specification match: fully implemented except optional stat metadata is not implemented. The OS allowed optional stat evidence; max-size enforcement is applied after download when stat evidence is unavailable.

### Fake-rclone tests in `cmd/worker/gdrive_rclone_provider_test.go`

What it does: re-enters the Go test binary as a fake rclone executable, records argv JSONL, copies a tiny fixture file, simulates failure, and emits fake secret-bearing output for redaction checks.

Why it appears to exist: OS 008 requires default tests without real Google Drive, OAuth, internet, or rclone credentials.

Assumption: argv recording is sufficient evidence that the worker passed structured arguments rather than a shell-composed string.

Could break if wrong: a future adapter change could still use `exec.Command` but alter argument order; tests would need intentional updates.

Specification match: fully implemented for default fake-rclone coverage.

### `TestMain` update in `cmd/worker/data_asset_materializer_test.go`

What it does: dispatches `GOET_FAKE_RCLONE=1` child processes to `runFakeRclone`.

Why it appears to exist: the existing fake-7z test approach already uses test-binary re-entry, and fake-rclone follows that local pattern.

Assumption: worker package tests keep a single `TestMain`.

Could break if wrong: adding another `TestMain` in the package would fail to compile.

Specification match: fully implemented.

## Concrete Trace

Example asset:

```json
{
  "binding_name": "drive_data",
  "provider_name": "drive_provider",
  "kind": "fixture",
  "provider": "gdrive_rclone",
  "location": {
    "type": "gdrive_rclone",
    "remote": "landcore",
    "drive_path": "Risk Model 2021/Data/ReleaseData.7z"
  },
  "cache": {
    "strategy": "worker_cache",
    "cache_key": "gdrive/landcore/source.7z"
  }
}
```

Trace:

- `boundDataAssetsFromWorkItem` validates the remote and drive path.
- `materializationStrategy` selects `worker_cache`.
- `materializeWorkerCache` checks `<asset_cache>/gdrive/landcore/source.7z/source` and `manifest.json`.
- On miss, `acquireGDriveRclone` confirms enablement and executable config.
- `copyToArgs` builds `["--config", configPath, "copyto", "landcore:Risk Model 2021/Data/ReleaseData.7z", tempPath]` when a config path exists.
- Fake rclone copies the fixture to `tempPath`.
- `hashFileWithLimit` records source evidence.
- `materializeWorkerCache` verifies expected evidence, writes `manifest.json`, and returns a materialized asset with provider type `gdrive_rclone`.
- If `archive` is present, `extractArchive` receives the cached source file and exposes the selected file or directory.

## Edge Cases Covered By Tests

- Provider disabled.
- Provider enabled but `rclone_executable` missing.
- Drive path validation failure before fake-rclone invocation.
- Structured argv with a path containing spaces.
- Config path argument loading from worker config.
- Tiny fixture copy into worker asset cache.
- Observed SHA-256 evidence.
- Expected SHA-256 mismatch cleanup.
- Expected size mismatch cleanup.
- Immutable cache reuse without a second fake-rclone invocation.
- Rclone subprocess failure.
- Redaction of config path, `access_token=...`, and bearer token output.
- Failure path does not leave a valid cache entry.
- Acquired rclone ZIP file flows into archive extraction.
- Existing `go test ./cmd/worker` non-rclone tests still pass.

## Edge Cases Not Covered By Tests

- Real rclone installation behavior.
- Real Google Drive or Shared Drive remotes.
- Rclone `lsjson` or stat metadata.
- Very large files beyond the max-size failure path.
- Concurrent workers racing on the same cache key.
- Windows-specific rclone behavior outside normal process spawning.
- File-ID based Google Drive access.

## Ambiguity and Maintenance Risk

- The OS describes an optional `Stat` boundary and max-size enforcement after stat when size is available. This implementation does not call rclone stat/lsjson, so max-size is enforced only after copy. That is consistent with "optional size/modtime evidence" but should be revisited before full-size data runs.
- Redaction is pattern-based. It covers config paths and common token strings but cannot prove all possible rclone credential output is redacted.
- `context.Background()` gives the rclone process no explicit timeout. Long-running downloads depend on worker/process management rather than adapter-level cancellation.
- Cache writes still use the existing simple local cache behavior. There is no cross-worker locking for identical cache keys.

## What I Should Understand Before Moving On

The new provider does not make Google Drive a GOET-native storage backend. It lets the worker call a preconfigured external `rclone` executable to copy one path-based remote file into the same worker cache path used by other providers, then relies on existing cache, integrity, manifest, and archive code.

## Questions I Should Answer

- Should a later slice add `rclone lsjson` stat evidence before download for large-file max-size rejection?
- Should worker config grow a timeout for external materializer commands?
- Should the cache layer get a per-cache-key lock before real multi-worker rclone use?
- What redaction policy is required for real rclone debug output if operators enable verbose rclone logging?
- Is file-ID access actually needed for the first LandCore Shared Drive run?
