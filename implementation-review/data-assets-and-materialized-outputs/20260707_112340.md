# Implementation Review: DAAMO-012 CDL Yan/Roy Fixture Pipeline

Date: 2026-07-07
Strategic Concept: Data Assets and Materialized Outputs
Operational Slice: `go-etl/docs/concepts/data-assets-and-materialized-outputs/012-cdl-yanroy-fixture-pipeline.md`

## Implementation In Execution Order

1. The sibling demo project declares fixture data locations, provider templates, and publish targets in `../go-etl-demo-project/project.json`.
2. Tiny fixture inputs were added under `../go-etl-demo-project/data/fixtures/cdl-yanroy`: one field-ID CSV grid, one CDL CSV grid inside a ZIP archive, one crop-code lookup CSV, and one dominant-share policy JSON file.
3. `../go-etl-demo-project/workflows/cdl-yanroy-fixture.json` defines one source-reference Python workflow step that binds four concrete data aliases: `cdl_grid`, `field_grid`, `crop_lookup`, and `assignment_policy`.
4. `../go-etl-demo-project/scripts/cdl_yanroy_fixture.py` reads those local paths from CLI arguments, computes field/CDL composition and dominant-crop rows, writes two CSV artifacts under `GOET_ARTIFACT_DIR`, and declares both artifacts through `GOET_OUTPUT_JSON`.
5. `go-etl/scripts/cdl-yanroy-fixture-smoke.ps1` and `.sh` create temporary controller/worker runtime roots, submit the committed workflow, and verify promoted/published output. PowerShell local and fake-HPCC modes were run successfully.
6. `PROJECT_STATE.md` and the OS document were updated to record implemented status and validation evidence.

## Changed Surfaces

### Demo `project.json`

What it does: Adds declarative fixture data locations, provider templates, and published data asset targets.

Why it appears to exist: The OS acceptance criteria require provider definitions for CDL grid, Yan/Roy field grid, crop lookup, assignment policy, and named published outputs.

Assumptions: Current GOET admission tolerates extra project metadata fields and stores the canonical project source without interpreting these provider templates.

What could break: If future project parsing rejects unknown fields or changes provider-template names, the fixture project may need a schema migration.

Spec comparison: Fully implemented as fixture declarations. The runnable workflow still carries bound worker-facing data assets because current GOET core does not compile project-level provider templates into work-item data assets.

### Fixture Workflow

What it does: Compiles one `python_script` work item for `fixture_tile_001` and passes resolved `${data.<alias>.local_path}` arguments plus `${artifact_dir}` output paths.

Why it appears to exist: The OS requires a workflow step that binds data aliases by `year` and `tile`, runs Python with resolved local paths, and publishes selected artifacts.

Assumptions: The worker materializer receives concrete bound assets directly in `parameters.data_assets`.

What could break: If later workflow compilation moves provider binding out of work-item parameters, this fixture should be updated to use the new compiler surface.

Spec comparison: Fully implemented for one deterministic year/tile fixture.

### Python Fixture Script

What it does: Reads rectangular integer CSV grids, reads lookup and policy files, ignores background field IDs, counts crop pixels per field, computes crop fractions, chooses dominant crop by count with lowest-code tie policy, writes composition and dominant-crop CSVs, and emits artifact declarations.

Why it appears to exist: This is the fixture-sized CDL/Yan/Roy product proof without raster/geospatial dependencies.

Assumptions: The fixture grids have matching dimensions; all non-background crop codes exist in the lookup; `dominant_share_v1` uses `dominant_share` and `lowest_crop_code`.

What could break: Unknown crop codes, non-rectangular grids, or future policy methods fail fast rather than silently producing rows.

Spec comparison: Fully implemented for the declared fixture policy. It intentionally does not implement real raster IO, RCI, or alternative crop science.

### Smoke Scripts

What they do: Run local and fake-HPCC fixture smokes and compare the published CSV files to expected deterministic text.

Why they appear to exist: The OS requires smoke validation through source admission, data binding, materialization, archive extraction, Python execution, artifact promotion, publication, and controller completion.

Assumptions: `localhost:8080` is available; Go is on `PATH`; Python is available; the sibling demo project is present. PowerShell fake-HPCC uses a local fake `sbatch` shim and launches the generated worker command directly to avoid WSL/Windows path translation.

What could break: A running controller on port 8080, missing Python, or changed generated Slurm command shape could break the smoke.

Spec comparison: Local smoke is fully implemented. Fake-HPCC smoke is implemented for the repository's local fake Slurm boundary; it does not claim real Singularity, real SSH, or real HPCC coverage.

## Concrete Trace

For `year=2023` and `tile=fixture_tile_001`, the worker materializes `cdl_grid` by copying `archives/cdl_2023_fixture_tile_001.zip` into the worker cache and extracting `cdl_2023_fixture_tile_001.csv` as `cdl.csv`. It references `yanroy/fixture_tile_001.csv`, `lookups/crop_codes.csv`, and `policies/dominant_share_v1.json` from the configured `fixture_data` root. The worker renders those local paths into Python args. The script counts five valid pixels for field `1`, with four crop-code `5` pixels and one crop-code `1` pixel, writes `field_cdl_composition.csv`, marks crop-code `5` as dominant, writes `field_dominant_crop.csv`, and declares both artifacts. The worker promotes both artifacts and publishes them to `published_data/field_cdl_composition/year=2023/tile=fixture_tile_001/...` and `published_data/field_dominant_crop/year=2023/tile=fixture_tile_001/...`.

## Edge Cases Covered By Tests

- CDL input is ZIP-backed and selected by archive member name.
- Four data aliases resolve through `${data.<alias>.local_path}`.
- Background field ID `0` is ignored.
- Composition rows include full field/crop distribution, not only dominant crop.
- Published composition and dominant CSVs match deterministic expected content.
- Local and fake-HPCC smoke paths reach controller completion.
- Worker package regression tests still pass.

## Edge Cases Not Covered By Tests

- Non-rectangular grid rejection.
- Mismatched CDL and field grid dimensions.
- Unknown crop code rejection.
- Dominant-share ties.
- Dominant fraction below the policy threshold.
- Multiple years or multiple tiles.
- Real 7z Yan/Roy archives, real CDL rasters, real Google Drive, real credentials, and real HPCC/Singularity.

## Risks And Ambiguities

- Provider templates in `project.json` are declarative fixture metadata; current execution still uses concrete bound work-item assets. This matches the existing worker surface but is not a full provider-template compiler.
- The PowerShell fake-HPCC shim avoids WSL path translation by launching the generated worker command directly after recording the generated Slurm script. This is a smoke boundary for local fake Slurm behavior, not proof of a Linux HPCC shell.
- The OS allowed production file list named `cdl-yanroy-fixture-pipeline.md` without the `012-` prefix, while the actual requested OS path includes `012-`; the implemented status was updated in the requested `012-...` file.

## What I Should Understand Before Moving On

This slice proves GOET's data-asset/artifact/publication orchestration shape with tiny field/CDL fixture files. It does not add GOET core parsing for provider templates and does not implement real raster processing.
