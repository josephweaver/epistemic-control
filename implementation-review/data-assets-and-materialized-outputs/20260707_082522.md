## Execution Order Summary

1. `runPythonScript` now loads all work-item args (`python_args`) as before.
2. Bound data assets are materialized first via `materializeDataAssets`, writing `GOET_DATA_ASSETS_JSON` when present.
3. `resolvePythonArgvBindings` scans each `python_args` entry for `${...}` tokens.
4. `${artifact_dir}` resolves to `staging.WorkDir`.
5. `${data.<binding>.local_path}` is looked up from the materialized manifest map.
6. Resolved arguments are passed to `exec.Command(w.pythonExecutable(), append([]string{entrypointPath}, args...)...)`.
7. Existing Python env var path behavior is unchanged except `GOET_ARTIFACT_DIR` is now always set.

## Changed API/Function/Test Surfaces

### `resolvePythonArgvBindings` in `cmd/worker/python_arg_binding.go`

What it does: resolves `${artifact_dir}` and `${data.<binding>.local_path}` tokens while preserving direct argv execution.

Why it exists: supports ergonomic CLI binding for Python scripts that prefer explicit flags over reading `GOET_DATA_ASSETS_JSON`.

Assumption: only explicit `local_path` and `artifact_dir` paths are supported by this OS.

Could break if wrong: any future token grammar extension (shorthand, expression syntax) will require a wider parser.

### `runPythonScript` in `cmd/worker/work_python.go`

What it does: shifts argument interpolation to after materialization and sets `GOET_ARTIFACT_DIR` in env.

Why it exists: argument token resolution depends on the resolved manifest and should happen before subprocess launch.

Could break if wrong: missing/invalid data assets now fail during argument resolution before Python launch as required.

### `cmd/worker/python_arg_binding_test.go`

What it does: validates interpolation (data + artifact), malformed and unsupported token failures, end-to-end argument rendering in Python execution, and failure-before-start behavior.

What it catches: malformed token syntax and unknown binding errors are returned before creating output artifacts.

## Edge Cases Covered

- Unknown binding token names
- Unsupported data token properties
- Malformed token syntax (`${` without `}` and extra `}`)
- Tokenized artifact directory path handling
- Runtime failure path never runs Python on token-resolution errors

## Edge Cases Not Covered

- Token collision/escaping semantics for literal `$`/`{}` in args
- Windows separator normalization for mixed path fragments in all argument combinations
- Large manifest path and performance behavior under many large argument vectors

## Ambiguity and Maintenance Risk

- OS target examples show `${artifact_dir}/...`; with mixed `/`/`\` path fragments, current behavior preserves raw token suffix separators. This is harmless in Go/Python path handling, but docs and tests should remain explicit that tokens are not shell-expanded.
- If a shorthand `${data.<binding>}` is later accepted, this parser must intentionally branch to avoid silent partial-match ambiguity.

## Implementation Quality Verdict

OS objective is implemented with narrow, mechanical code changes and no behavior changes beyond the requested feature boundary.
