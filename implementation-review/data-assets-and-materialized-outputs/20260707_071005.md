## Changed API And Behavior

### Data locations

Files: `internal/model/data_location.go`

What it does: `DataLocation` validates a named `registered_location` and optional access mode. `DataLocationPathTemplate` validates a configured location name plus slash-relative `path_template`.

Why it appears to exist: OS 005 Target State requires named data locations and path templates for provider and publish declarations.

Assumptions: Name characters are limited to letters, digits, `_`, `-`, and `.`. Relative path safety reuses `ValidateArtifactRelativePath`, so template paths also reject backslashes, drive prefixes, `.` segments, `..` segments, empty segments, and trailing slashes.

What could break if wrong: A legitimate future location name with spaces or colons would be rejected. A future directory marker ending in `/` would need a different representation.

Design match: Fully implemented for this slice. No runtime root resolution is implemented, matching the out-of-scope list.

Example trace: `DataLocation{Name:"fixture_data", Type:"registered_location"}.Validate()` accepts the type and validates the name without resolving `root_ref`.

Tests covered: valid registered location; unsafe path template rejection.

Tests not covered: invalid access values and unusual but valid-looking root references.

Risk: The location path validator currently shares artifact wording in error messages through `ValidateArtifactRelativePath`; behavior is correct, but error text says "artifact path".

### Provider templates and binding

Files: `internal/model/data_provider.go`, `internal/model/data_asset.go`

What it does: `DataProviderTemplate` validates `http`, `local_file`, `registered_location`, and `gdrive_rclone` declarations. `Bind` renders `${name}` placeholders with concrete parameter values into a `BoundDataAsset`.

Why it appears to exist: OS 005 requires reusable provider templates and concrete worker-facing bound assets without implementing download, cache, extraction, or publication behavior.

Assumptions: Every name listed in `Parameters` is required at bind time. Template references must be declared in `Parameters`. Rendering uses `fmt.Sprint` and then validates rendered paths and SHA-256 values through the concrete bound asset validation.

What could break if wrong: If a later compiler wants optional parameters or structured parameter rendering rules, the bind method is too simple. If parameter values contain slashes for a path segment, concrete validation rejects the rendered path if it becomes unsafe.

Design match: Fully implemented for model-level binding. Controller workflow parsing is not implemented, matching the out-of-scope list.

Example trace: `cdl_zip` with URL `https://example.invalid/cdl/${year}.zip`, cache key `cdl/${year}/source.zip`, and archive member `${year}_30m_cdls.tif` binds with `year=2024` into URI `https://example.invalid/cdl/2024.zip`, cache key `cdl/2024/source.zip`, and member `2024_30m_cdls.tif`.

Tests covered: all four supported provider types; missing provider name, kind, provider type, URL/path, rclone remote/path; undeclared parameter references; missing bind parameter; JSON round trip; concrete bound asset validation.

Tests not covered: malformed `${...}` syntax outside provider templates, mixed `url_template` plus `uri_template`, and non-string parameter values beyond integer rendering.

Risk: Template parsing is intentionally small and only handles `${name}`. It does not implement escaping, nested templates, or type-aware formatting.

### Integrity and cache policy

Files: `internal/model/data_asset.go`

What it does: Integrity validates SHA-256 shape and non-negative size. Cache validates `worker_cache` or `reference`, validates optional cache keys as safe relative paths, and exposes `EffectiveImmutable()` so omitted `immutable` defaults to true for `worker_cache`.

Why it appears to exist: OS 005 requires representation and validation only; runtime verification is deferred.

Assumptions: Concrete SHA-256 evidence must be lowercase 64-character hex. Template SHA-256 values may be placeholders until binding.

What could break if wrong: Uppercase hash inputs are rejected. If a data source publishes uppercase hashes, the compiler must normalize before validation or the model rule must change.

Design match: Fully implemented for model declarations. No evidence recording or cache reuse behavior is implemented, matching the out-of-scope list.

Example trace: `DataAssetCache{Strategy:"worker_cache"}.EffectiveImmutable()` returns true when `Immutable` is nil.

Tests covered: valid SHA-256 plus size; invalid SHA-256; unsafe cache key; immutable default.

Tests not covered: explicit `immutable:false` behavior and negative size in every struct that carries size.

Risk: Required integrity with only `size_bytes` is accepted. That matches the OS wording allowing SHA-256 or size expectations, but verified immutable reuse will need stronger evidence in later slices.

### Archive declarations

Files: `internal/model/data_archive.go`

What it does: Archive templates and concrete archives validate `zip` and reserved `seven_zip`, safe member paths, safe `as` paths, and `selected_path` requiring exactly one required member.

Why it appears to exist: OS 005 reserves archive selection semantics before worker extraction is implemented.

Assumptions: `Required:nil` defaults to required. Archive `as` paths use the same clean relative path rule as artifacts, so `field_segments_related/` with trailing slash is rejected despite appearing as an example in the Strategic Concept README.

What could break if wrong: If later selected-directory extraction wants directory marker `as` values ending in `/`, this validator will need a representation adjustment.

Design match: Fully implemented for the OS acceptance criteria: valid ZIP selector and valid reserved `seven_zip` declaration validate, unsafe paths are rejected, and no extractor is implemented.

Example trace: A ZIP archive with one required `cdl.tif` selector and `expose:"selected_path"` validates; changing the member to `../cdl.tif` fails before any filesystem operation can be added later.

Tests covered: valid ZIP, valid `seven_zip`, unsafe member template, unsafe `as` path.

Tests not covered: selected_path with zero or multiple required members.

Risk: The selected_path rule is checked by count of required selectors, not by file-versus-directory type, because the model has no file type field.

### Materialized data asset manifests

Files: `internal/model/data_asset.go`

What it does: `MaterializedDataAssetManifest` defaults to schema `goet/materialized-data-assets/v1` and validates non-empty assets. Each asset validates binding name, kind, provider type, local path, cache key safety, optional hashes/sizes, archive type, and archive member evidence.

Why it appears to exist: OS 005 requires a worker-facing manifest shape for assets actually present in the execution environment.

Assumptions: `local_path` is required but not forced to be slash-relative because it is worker-local and may be absolute in the worker environment.

What could break if wrong: If the worker manifest must be portable across OS paths, accepting any non-empty `local_path` may be too loose.

Design match: Fully implemented as a model declaration. Writing `GOET_DATA_ASSETS_JSON` is not implemented, matching the out-of-scope list.

Example trace: A materialized HTTP CDL asset with local path `/worker/cache/cdl.tif`, cache key `cdl/2024/source.zip`, source hash, selected hash, and ZIP member evidence validates.

Tests covered: valid manifest JSON round trip; missing binding name, kind, local path, and provider type.

Tests not covered: invalid archive type inside materialized assets and unsafe archive member path inside materialized manifests.

Risk: Provider type is required even though `ProviderName` is optional; later worker code should preserve provider type when constructing manifests.

### Published data assets

Files: `internal/model/published_data_asset.go`

What it does: `PublishedDataAssetTarget` validates target name, kind, location template, parameter references, and overwrite policy. `Bind` renders a target into a `BoundPublishTarget` with a concrete registered-location path.

Why it appears to exist: OS 005 requires publish targets and publish bindings without runtime copy or registration.

Assumptions: Publish targets always bind to `registered_location` in this slice. `fail_if_exists` and empty default are the only accepted overwrite policies.

What could break if wrong: A future object-store target or hash-preserving replace mode will require adding supported location and overwrite values deliberately.

Design match: Fully implemented for conservative publish declarations. Runtime registration and copying are not implemented, matching the out-of-scope list.

Example trace: Target path `field/year=${year}/tile=${tile}/out.csv` binds with `year=2024`, `tile=tile_001` into `field/year=2024/tile=tile_001/out.csv`.

Tests covered: valid target, concrete publish binding, default overwrite, destructive overwrite rejection, JSON round trip.

Tests not covered: undeclared publish path parameter and missing publish bind parameter.

Risk: `FromArtifact` uses the same name validator as provider names. Artifact names with characters outside that set would be rejected by publish binding validation.

## Overall Acceptance Match

Implemented:

- Named registered locations.
- Provider templates for `http`, `local_file`, `registered_location`, and `gdrive_rclone`.
- Concrete step data bindings and bound data assets.
- Integrity, cache, archive, materialized manifest, publish target, and publish binding model types.
- Safe slash-relative validation for path templates, cache keys, archive members, archive `as`, and publish paths.
- SHA-256 and size validation.
- Conservative overwrite policy validation.
- JSON round trips for the model contracts.
- `go test -count=1 ./internal/model`.

Not implemented by design:

- Controller parsing of top-level data provider or publish target declarations.
- Worker downloading, copying, cache reuse, archive extraction, rclone, `GOET_DATA_ASSETS_JSON`, command interpolation, artifact publication, persistence schema changes, or catalog registration.

Implemented differently or with narrow interpretation:

- `DataAssetIntegrityTemplate.Validate` accepts SHA-256 placeholders and defers final hash-shape validation to bound concrete assets.
- Archive `as` paths reject trailing slash directory markers because the slice says slash-relative safe path rules and the existing validator rejects trailing slashes.
- The OS-required context files for source-control and dependency-aware workflows were found under `docs/concepts/complete/...`, not at the exact non-complete paths named by the OS.

## What I Should Understand Before Moving On

- OS 005 only creates declarations and validation. No bytes move in this slice.
- `Bind` is a model helper, not a workflow compiler. It resolves simple provider templates into concrete declarations but does not integrate with controller admission.
- Path safety is intentionally stricter than normal filesystem convenience; clean slash-relative paths are the contract for authored templates.
- `worker_cache` immutability default exists as a model method, but the actual cache evidence and reuse rules remain future worker materialization work.

## Questions I Should Answer

- Should archive `as` allow directory marker values ending in `/`, or should directory destinations use a clean relative directory path without a trailing slash?
- Should `from_artifact` in publish bindings use the broader artifact name rules instead of the stricter data-name rule?
- Should provider parameter binding support optional parameters, default values, or type-specific formatting in a later compiler slice?
- Should uppercase published SHA-256 values be normalized or rejected at workflow admission?
- Should materialized `local_path` remain only non-empty, or should worker-specific path validation be added in the materialization slice?
