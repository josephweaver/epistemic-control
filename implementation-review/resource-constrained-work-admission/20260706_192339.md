## Implementation In Execution Order

1. The model layer now exposes resource-operator aliases and `ResourceConstraintAllows`.
2. `ResourceConstraintAllows` computes `totalUnits + requestedUnits` through `checkedAdd`, then applies the requested comparison operator.
3. `ResourceConstraintChecksAllow` evaluates a candidate's check rows as an AND and returns true for an empty check list.
4. The persistence record for queued checks now uses `int64` for `TotalUnits`, `RequestedUnits`, and `TargetUnits`.
5. `Store.ListQueuedResourceConstraintChecks` reads rows from `queued_resource_constraint_checks` ordered by `queued_at`, `work_item_id`, and `constraint_index`.
6. Tests prove the model comparison behavior and the store reader behavior.
7. The OS markdown status and `PROJECT_STATE.md` were updated.

## Public API And Behavior Review

### `model.ResourceOperator`

What it does: aliases the existing `WorkItemResourceConstraintOperator` type under the shorter name used by the OS.

Why it exists: the OS target function names `ResourceOperator`; the repository already had the longer work-item-specific operator type.

Assumption: the resource-admission operator set is the same as the persisted work-item constraint operator set.

Risk if wrong: a later resource-admission operator could diverge from work-item declaration validation, and the alias would make the divergence harder to express.

Design match: fully implemented, with naming adapted to existing model types.

### `model.ResourceConstraintAllows`

What it does: returns whether `totalUnits + requestedUnits` satisfies one of `=`, `!=`, `<`, `>`, `<=`, or `>=` against `targetUnits`.

Why it exists: the Strategic Concept says SQL should expose facts and Go should evaluate the predicate.

Assumption: `int64` arithmetic is the correct runtime domain for resource totals.

Risk if wrong: if persisted values can exceed `int64`, scan or evaluation will fail before admission can make a scheduling decision.

Design match: fully implemented.

Concrete trace: with `totalUnits=4`, `requestedUnits=2`, `operator="<="`, and `targetUnits=10`, `checkedAdd` returns `6`; the `<=` branch returns true.

Covered edge cases: each operator true and false, `<=` equality, `>=` equality, overflow, unsupported operator.

Not covered: negative inputs, because current persistence validation rejects negative target units and non-positive requested units, and the view reports sums from persisted requested units.

### `model.ResourceConstraintChecksAllow`

What it does: evaluates a slice of check facts and returns false on the first failed predicate, error on evaluator error, and true when the slice is empty.

Why it exists: the OS states all constraints for one candidate are an AND and zero constraint rows are eligible.

Assumption: the caller groups rows by candidate before passing the slice.

Risk if wrong: passing rows for multiple candidates would incorrectly combine unrelated work items into one AND decision.

Design match: fully implemented for the helper behavior required by OS 004. Claim-path grouping is intentionally out of scope until OS 005.

### `persistence.QueuedResourceConstraintCheckRecord`

What it does: represents one row from the SQL view with work item ID, queued time, constraint index, resource key, total units, requested units, operator, and target units.

Why it exists: the store layer needs a typed reader for candidate-check rows.

Assumption: resource unit quantities should be scanned as `int64` in Go even when SQLite stores integers dynamically.

Risk if wrong: callers that expected `int` fields would need adjustment; current repository usage was limited to this new reader path.

Design match: fully implemented.

### `persistence.Store.ListQueuedResourceConstraintChecks`

What it does: reads every queued resource-check row from `queued_resource_constraint_checks` in deterministic order.

Why it exists: OS 004 requires a persistence reader for rows from the SQL view without changing the claim algorithm.

Assumption: a full-reader method is sufficient for OS 004 and later claim code can either reuse it or add a candidate-specific transaction-scoped helper in OS 005.

Risk if wrong: OS 005 may need a transaction-bound reader to avoid stale reads during claim evaluation; that is a future claim-path concern, not part of this slice.

Design match: fully implemented for OS 004.

Concrete trace: the store test inserts running memory work with requested units 4 and running GPU work with requested units 99. A queued memory row reads `total_units=4`; the unrelated GPU row does not inflate the memory total. The queued GPU row reads `total_units=99`.

Covered edge cases: reader order by queued time, work item ID, and constraint index; same-key running totals; unrelated resource keys do not affect another key's total.

Not covered: database scan failure due to values outside `int64`, because SQLite fixture values are normal bounded integers.

## Ambiguity And Maintenance Risks

- `ResourceConstraintChecksAllow` has no work-item ID field, so the grouping requirement is implicit. OS 005 should make grouping explicit when claim code consumes view rows.
- `ListQueuedResourceConstraintChecks` reads all rows, not rows for one candidate. That matches OS 004 but may be inefficient or too broad for a claim transaction if reused directly.
- The evaluator accepts negative total/request/target values if called directly. Current persistence validation prevents those persisted facts, but direct model callers must preserve the same invariant.

## What I Should Understand Before Moving On

- SQL still does not decide resource eligibility; it only exposes candidate facts.
- The Go evaluator adds requested units to current running total before comparing.
- Zero resource-check rows means resource admission passes.
- OS 004 does not change `/work/next`; resource-aware claim selection belongs to OS 005.
- Overflow is an error, not a failed boolean comparison.

## Questions I Should Answer

- Should OS 005 use the full queued-check reader or add a transaction-scoped candidate reader?
- Should negative direct model inputs be rejected explicitly even though persistence should never produce them?
- Should the store record use `model.ResourceOperator` instead of `string`, or should persistence remain model-agnostic?
