## Changed APIs And Structs

### `workflow.ResourceConstraintDeclaration`

Location: `internal/workflow/fanout.go:27`

What it does: Represents a workflow/raw resource constraint declaration before it is resolved into `model.WorkItemResourceConstraint`.

Why it appears to exist: OS 002 requires workflow-facing declarations with `resource_key`, `requested_units`, `operator`, and `target_units`.

Assumptions encoded:

- JSON declarations use snake-case field names.
- Literal `resource_key` and `operator` are strings.
- Literal `requested_units` and `target_units` are integers.
- Whole-value `${...}` references are enough for resolver-backed integer fields.

What could break if wrong: A workflow author expecting mixed string interpolation such as `target:${controller_config.target}/memory-mib` will not get full resolver interpolation in this slice.

Spec comparison: Partially different from the full variable resolver behavior. It implements the OS examples and whole-value expressions, but not arbitrary string interpolation because the resolver package does not expose typed-expression resolution and OS 002 did not allow editing `internal/variable`.

### `workflow.CompiledFanOutWorkItem`

Location: `internal/workflow/fanout.go:38`

What it does: Carries a compiled `model.WorkItem` plus separately resolved constraints.

Why it appears to exist: OS 002 says not to put constraints into the worker payload unless unavoidable.

Assumptions encoded:

- Resource constraints are admission metadata, not worker execution metadata.
- Existing callers should still be able to receive only `[]model.WorkItem`.

What could break if wrong: If later code accidentally uses only the legacy `CompileFanOutWorkItems` path for admission, constraints would be dropped.

Spec comparison: Fully aligned with the concept decision to carry constraints beside compiled work items.

### `workflow.ResolveResourceConstraints`

Location: `internal/workflow/fanout.go:43`

What it does: Resolves raw wrapper declarations into `model.WorkItemResourceConstraint` values and validates `created_at`.

Why it appears to exist: Raw `/work` submissions do not go through fan-out compilation but still need the same declaration validation.

Assumptions encoded:

- Raw submissions do not have a workflow variable scope unless a future caller supplies one.
- Raw constraints are validated now, persisted in OS 003.

What could break if wrong: Raw constraints with references fail unless the resolver passed to this helper contains those variables.

Spec comparison: Implements the raw wrapper direction from OS 002 without changing `model.WorkItem`.

### `workflow.CompiledWorkItem.ResourceConstraints`

Location: `internal/workflow/workflow.go:16`

What it does: Carries resolved constraints through full workflow compilation.

Why it appears to exist: Existing `CompileWorkflowResult` is the workflow-level admission result.

Assumptions encoded:

- Full-workflow compilation may still be used by tests or compatibility paths.
- Constraints should travel with their owning compiled item.

What could break if wrong: Code that constructs `CompiledWorkItem` manually and omits constraints will behave as an unconstrained item.

Spec comparison: Fully implemented for workflow compilation.

### `workflow.CompileStageWorkItem.ResourceConstraints`

Location: `internal/workflow/compile_stage.go:10`

What it does: Carries resolved constraints through stage-scoped compilation.

Why it appears to exist: Current controller admission compiles dependency-ready stages, not necessarily an entire workflow.

Assumptions encoded:

- Constraint resolution must happen for just-in-time activated stages as well as initial stage 0.

What could break if wrong: If an activation path drops constraints, downstream stages could be queued without resource metadata.

Spec comparison: Fully implemented for in-memory carry-through. Persistence is intentionally not implemented in this slice.

### `cmd/controller.RawWorkSubmission`

Location: `cmd/controller/main.go:67`

What it does: Defines the raw `/work` wrapper shape with `work_item` plus optional `resource_constraints`.

Why it appears to exist: OS 002 explicitly prefers a wrapper rather than changing the worker payload contract.

Assumptions encoded:

- Presence of top-level `work_item` means wrapper mode.
- Absence of top-level `work_item` means legacy direct `model.WorkItem`.

What could break if wrong: A legacy raw work payload with an unrelated top-level `work_item` field would be interpreted as the wrapper.

Spec comparison: Fully implemented.

### `persistenceRecordsFromCompiledStageResults`

Location: `cmd/controller/workflow_stage_queue.go:92`

What it does: Converts compiled stage work into persistence work-item records, queued records, dependency memberships, and now stamped resource constraint records.

Why it appears to exist: This is the controller boundary immediately before work-item insertion and queueing.

Assumptions encoded:

- The persisted constraint `work_item_id` must use the run-scoped ID, not the worker payload's unscoped item ID.
- `created_at` uses the same timestamp as the work item and queue records.

What could break if wrong: If OS 003 persists constraints using unscoped IDs, foreign keys to `work_items` would fail or resource admission would read the wrong records.

Spec comparison: Fully implemented for resolved in-memory records; persistence remains out of scope by OS 002.

## Concrete Trace

Example declaration:

```json
{
  "resource_key": "target:local/memory-mib",
  "requested_units": "${step.memory_allocated_mib}",
  "operator": "<=",
  "target_units": "${controller_config.local_memory_limit_mib}"
}
```

Trace:

1. Workflow JSON decodes the declaration through `ResourceConstraintDeclaration.UnmarshalJSON`.
2. `CompileWorkflowStage` calls `CompileStepItems`.
3. `CompileStepItems` calls `CompileFanOutStepItems`.
4. `CompileFanOutWorkItemResults` resolves a fan-out record.
5. `resolveConstraintValue` maps `${step.memory_allocated_mib}` to `variable.ApplyAccessor(fanOutValue, ".memory_allocated_mib")`.
6. `resolveConstraintValue` maps `${controller_config.local_memory_limit_mib}` through the existing resolver.
7. `validateCompiledResourceConstraint` rejects unsupported operators, empty keys, non-positive requested units, and negative target units.
8. `persistenceRecordsFromCompiledStageResults` rewrites `WorkItemID` from `download-2026` to `run-...:download-2026` and sets `CreatedAt`.
9. The returned constraint record is available beside work-item and queued records. It is not inserted in OS 002.

## Edge Cases Covered By Tests

- Workflow constraints resolve literal strings and integers.
- Workflow constraints resolve whole-value controller-config references.
- Fan-out item fields resolve requested units through `${step.memory_allocated_mib}`.
- JSON declarations accept the OS snake-case shape.
- Invalid operator is rejected.
- Non-integer requested units are rejected.
- Non-positive requested units are rejected.
- Negative target units are rejected.
- Empty resource key is rejected.
- Duplicate resource keys on one work item are rejected.
- Stage persistence conversion stamps run-scoped work-item ID and `created_at`.
- Raw `/work` wrapper is accepted without putting constraints into worker payload JSON.
- Invalid raw wrapper constraints are rejected before queued work is created.
- Existing direct raw `/work` submissions are still covered by existing tests.
- Existing workflows without constraints remain covered by existing workflow and controller tests.

## Edge Cases Not Covered By Tests

- Non-integer `target_units` has no explicit test, although it uses the same resolver path as `requested_units`.
- Invalid raw duplicate resource keys are covered indirectly by workflow duplicate tests, not by a raw handler test.
- Mixed string interpolation in `resource_key` is not implemented or tested.
- Whole-value references with non-step accessors such as `${workflow.step[0]}` are not implemented by the local constraint resolver.
- Just-in-time activation with resource constraints is preserved by code path, but not covered by a dedicated activation test with constraints.

## Ambiguity And Maintenance Risks

- `ResourceConstraintDeclaration` duplicates a narrow part of variable resolution because `internal/variable.Resolver` has no exported method for resolving arbitrary `TypedExpression`. This is a future maintenance risk if resource declarations need full interpolation parity.
- The raw wrapper resolves constraints but discards them after validation because OS 003 owns persistence. This is intentional for OS 002, but it means raw constraints have no runtime effect until OS 003.
- The controller uses blank identifier assignment for the raw wrapper constraints in `submitWorkHandler`. This should be replaced by actual persistence in OS 003.
- Manual construction of `workflow.CompiledWorkItem` or `workflow.CompileStageWorkItem` can omit `ResourceConstraints`. Tests cover the main compiler paths, not every manual fixture.

## What I should understand before moving on

Resource constraints now exist as resolved in-memory admission metadata beside compiled work items. They are not in `worker_payload_json` and are not persisted yet. OS 003 must consume the returned `[]model.WorkItemResourceConstraint` from controller staging and insert it atomically with the work item and queued state.
