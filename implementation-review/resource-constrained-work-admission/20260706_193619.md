## Execution-Order Summary

1. `cmd/controller.(*Controller).nextPersistedWorkHandler` now enters `Controller.claimMu` before calling `Store.ClaimNextWork`.
2. `internal/persistence.(*Store).ClaimNextWork` starts one transaction, lists all queued candidates ordered by `queued_at, work_item_id`, and evaluates candidates one at a time.
3. For each queued candidate, `queuedResourceConstraintsAllow` reads that candidate's rows from `queued_resource_constraint_checks`, converts them to `model.ResourceConstraintCheck`, and calls `model.ResourceConstraintChecksAllow`.
4. If a candidate is blocked, the store continues to the next queued candidate without mutating `work_item_attempts`, `running_work`, or `queued_work`.
5. If a candidate is eligible, `claimQueuedWork` performs the existing mutation sequence: insert `work_item_attempts`, insert `running_work`, delete `queued_work`, commit, and return the claimed record.
6. If no candidate is eligible, the transaction commits without mutations and returns `found=false`.

## Changed API, Function, And Test Review

### `Controller.claimMu`

What it does: Adds a controller-local mutex around persisted claim execution in `nextPersistedWorkHandler`.

Why it appears to exist: OS 005 says to use "controller-local serialization around claim evaluation" and assumes one controller owner for the database.

Assumption encoded: The active deployment has one controller process making claim decisions for this database.

If wrong: Two controller processes could still evaluate resource usage concurrently because `claimMu` is process-local, not database-native.

Spec comparison: Fully implemented for the OS 005 single-controller decision; distributed locking remains out of scope.

### `Store.ClaimNextWork`

What it does: Changes the claim algorithm from oldest-only to resource-aware candidate selection.

Why it appears to exist: The OS target state requires reading queued candidates in deterministic order and claiming the first one whose resolved resource checks pass in Go.

Assumption encoded: `queued_resource_constraint_checks` is the authoritative read model for candidate running totals and persisted constraint facts.

If wrong: A bad view definition or stale `running_work` rows would cause incorrect admission decisions.

Spec comparison: Fully implemented. It preserves empty queue behavior, skips blocked candidates, leaves all-blocked queues unchanged, keeps unconstrained work eligible through zero checks, and performs the claim mutation atomically inside one transaction.

Concrete trace: If `work-older` is queued at `00:00:01`, `work-newer` is queued at `00:00:02`, running memory is `2`, `work-older` requests `2 <= 3`, and `work-newer` requests `1 <= 3`, `ClaimNextWork` evaluates `work-older` as `2 + 2 <= 3` false, then evaluates `work-newer` as `2 + 1 <= 3` true, then inserts `attempt`, inserts `running_work`, deletes only `work-newer` from `queued_work`, and commits.

### `listQueuedWorkForClaim`

What it does: Reads all queued candidates ordered by `queued_work.queued_at, queued_work.work_item_id`.

Why it appears to exist: OS 005 requires deterministic queue order and no head-of-line blocking.

Assumption encoded: Pulling the full queued candidate list into Go is acceptable for the current scale.

If wrong: Large queues could make each claim scan too much data. That is a performance concern, not a current correctness failure.

Spec comparison: Fully implemented for deterministic candidate ordering.

### `listQueuedResourceConstraintChecksForWorkItem`

What it does: Reads one candidate's rows from `queued_resource_constraint_checks`, ordered by `constraint_index`.

Why it appears to exist: OS 005 says each candidate's check rows should be read from the view and evaluated in Go.

Assumption encoded: Candidates with no view rows have no resource constraints.

If wrong: Missing view rows for a constrained work item would make it eligible incorrectly. Earlier slices are responsible for persisting constraints and defining the view.

Spec comparison: Fully implemented for claim-time check reading.

### `queuedResourceConstraintsAllow`

What it does: Converts persistence view rows into `model.ResourceConstraintCheck` values and delegates operator evaluation to `model.ResourceConstraintChecksAllow`.

Why it appears to exist: The Strategic Concept says SQL must expose facts and Go must evaluate operators.

Assumption encoded: The model evaluator is the single authority for all six operators and overflow behavior.

If wrong: A bug in `internal/model/resource_constraint.go` would affect store admission. Existing slice 004 tests cover that evaluator directly.

Spec comparison: Fully implemented.

### `claimQueuedWork`

What it does: Holds the previous oldest-only mutation block as a helper called only after resource eligibility passes.

Why it appears to exist: It keeps the mutation sequence unchanged while allowing the caller to choose an eligible candidate.

Assumption encoded: The prior insert attempt, insert running, delete queued sequence was already correct.

If wrong: Claim atomicity bugs would remain. Existing and new tests assert the specific rows created/deleted.

Spec comparison: Fully implemented.

## Covered Edge Cases

- Empty queue returns no work.
- Unconstrained queued work remains claimable through existing tests and zero-check semantics.
- Oldest constrained item is claimed when eligible.
- Blocked oldest item does not block a later eligible item.
- All queued items blocked returns no work and creates no attempt/running rows.
- Claim mutation inserts `work_item_attempts`, inserts `running_work`, and deletes `queued_work`.
- Second claim sees first claim's running resource usage.
- Completion removes the running row, allowing a later blocked item to be claimed.
- All six operators have pass and block cases at claim time.
- Existing dependency-aware workflow tests in `cmd/controller` still pass.

## Not Covered Or Residual Risk

- No explicit handler-level concurrent request test proves `claimMu` under simultaneous HTTP requests.
- No multi-controller database-locking behavior is implemented; OS 005 explicitly excludes distributed multi-controller locking.
- No performance test covers very large queues; `listQueuedWorkForClaim` scans all queued candidates.
- Failure-path release is covered by existing terminal-path tests for `FailAttempt`, but the new resource-specific release test uses completion, not failure.
- Overflow behavior is covered in the model evaluator from slice 004, not repeated through `ClaimNextWork`.

## Ambiguity Or Maintenance Risk

- `Store.ClaimNextWork` now imports `internal/model`. That is acceptable for this slice because the Strategic Concept makes the Go evaluator authoritative, but it couples persistence admission to model-level operator semantics.
- `listQueuedWorkForClaim` reads all queued candidates. This is simple and matches OS 005, but future high-volume queues may need a bounded scan or indexed candidate window.
- `claimMu` serializes only within one controller process. This matches the OS concept decision but should not be mistaken for a durable database lease.

## What I Should Understand Before Moving On

Resource admission is now a claim-time filter, not a queueing-time mutation. Blocked work stays in `queued_work`; eligible work is the first queued candidate whose view-derived checks all pass through `model.ResourceConstraintChecksAllow`.
