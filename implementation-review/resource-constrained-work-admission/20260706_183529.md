Public/API/Table/Config/Test Changes
- `WorkItemResourceConstraint` and `WorkItemResourceConstraintOperator*` constants in `internal/model/work_item.go`
  - What it does: Defines model fields and legal operators for persisted resource constraints.
  - Why it exists: Directly encodes the OS model row shape and allowed operator set before persistence or claim-time evaluation is added.
  - Assumption encoded: constraint operators are validated at model layer and remain stable across future slices.
  - Failure mode if wrong: later schema writes and DB checks can accept drifted operators, making claim logic impossible to interpret.

- `WorkItemResourceConstraint.Validate()` and `isSupportedWorkItemResourceConstraintOperator()` in `internal/model/work-item.go`
  - What it does: Enforces required fields (`work_item_id`, `resource_key`, numeric bounds) and supported operators.
  - Why it exists: Makes constraints fail before persistence and protects from silent invalid data.
  - Assumption encoded: empty or malformed constraint strings are not valid business inputs.
  - Failure mode if wrong: invalid constraints can be written to SQLite and then break claim behavior at runtime.

- `WorkItemResourceConstraintRecord` and `QueuedResourceConstraintCheckRecord` in `internal/persistence/store.go`
  - What it does: Adds internal typed rows for persisted constraints and derived candidate-checks.
  - Why it exists: Supports future claim-time candidate checks and reduces untyped SQL scanning.
  - Assumption encoded: record layers remain internal and can evolve without API compatibility promises.
  - Failure mode if wrong: future slice adds readers/writers with mismatched field meaning or ordering.

- `SupportedSchemaVersion` change to `5` in `internal/persistence/store.go`
  - What it does: Triggers DB migration/validation updates to include the new table and view.
  - Why it exists: Existing schema evolution requires strict version checking.
  - Assumption encoded: no old runtime opens this DB with implicit assumptions about schema shape.
  - Failure mode if wrong: mixed-version deployments cannot open persisted stores consistently.

- `work_item_resource_constraints` and `queued_resource_constraint_checks` additions in `internal/persistence/db_adapter_sqlite.go`
  - What it does: Creates normalized constraint rows and derived candidate-check rows for each queued work item constraint.
  - Why it exists: Implements the OS requirement that schema and derived facts exist for resource admission.
  - Assumption encoded: view totals are computed as running work candidate totals only by resource key.
  - Failure mode if wrong: claim-time admission can grant/deny based on wrong running totals.

- `TestOpenStoreCreatesSQLiteQueuedResourceConstraintChecksView` in `internal/persistence/db_adapter_sqlite_test.go`
  - What it does: Verifies the view exposes queued constraints only and computes `total_units` including one running item.
  - Why it exists: Locks down this critical shape before moving to claim-time evaluator slice.
  - Assumption encoded: existing `running_work` and constraint sums are sufficient for candidate totals.
  - Failure mode if wrong: scheduling gates become untestable and regression-prone.

- `TestOpenStoreRejectsInvalidWorkItemResourceConstraints` in `internal/persistence/db_adapter_sqlite_test.go`
  - What it does: Verifies schema constraints reject unsupported operators, bad keys, non-positive requests, negative target, negative index, duplicate resource key, and duplicate constraint index.
  - Why it exists: Confirms all SQL invariants from the OS acceptance criteria are active.
  - Assumption encoded: invalid rows should never be inserted by normal pathways.
  - Failure mode if wrong: malformed constraints could persist and cause undefined admission behavior.

- `TestWorkItemResourceConstraintValidate` in `internal/model/work_item_test.go`

  - Why it exists: Adds regression coverage for negative cases in the same layer that composes future insertion behavior.

  - Failure mode if wrong: broken invariants can bypass SQL checks only in corner paths (e.g., null/empty fields).

OS Alignment
- Objective: Implemented in full for persistence and schema. Work-item constraint model and schema are now present; operator set and invariants match requirements.
- View contract: Implemented exactly as requested, including `queued_resource_constraint_checks` columns and no admission decision logic in SQL.
- Schema validation and version bump: implemented.
- Unsupported/invalid constraints: implemented at both schema and model test levels.
- Work items without resource constraints: untouched by this slice and remain supported by default behavior.

What changed from spec
- `target_units` is stored as `INTEGER NOT NULL CHECK (target_units >= 0)` as required.
- `UNIQUE(work_item_id, resource_key)` enforced by schema as required.
- Added record types in `store.go` for internal usage; these are additional internal artifacts, not external API changes.

Concrete Example Through New Path
1) `openTestStore` starts SQLite and initializes schema.
2) `sqliteSchemaStatements` creates `work_item_resource_constraints` and `sqliteViewStatements` creates `queued_resource_constraint_checks`.
3) A queued work item and a running work item are inserted in the test.
4) Constraint rows are inserted for one resource key on queued item and one on running item.
5) Querying `queued_resource_constraint_checks` for the queued item returns only the queued rows, with `total_units` aggregated from running work using matching `resource_key`.

Edge Cases Covered by Tests
- Unsupported operator rejected by SQL CHECK.
- Empty `resource_key` rejected by SQL CHECK.
- Non-positive `requested_units` rejected by SQL CHECK.
- Negative `target_units` rejected by SQL CHECK.
- Negative `constraint_index` rejected by SQL CHECK.
- Duplicate `(work_item_id, constraint_index)` rejected by PRIMARY KEY.
- Duplicate `(work_item_id, resource_key)` rejected by UNIQUE.
- Queue-only visibility of derived constraint view rows.

Edge Cases Not Covered
- Overflow behavior of `SUM(requested_units)` is not tested in this slice.
- Behavior when constraint insertion is attempted through production `Store` APIs is not yet covered because persistence write paths are not yet added by this slice.
- Missing `work_item_id` FK violation insertion path is covered indirectly by table constraints but not explicitly tested in this file.

Ambiguity / Risk
- `QueuedResourceConstraintCheckRecord` is defined and validated by this slice but currently no read accessor exists yet; future slice should keep one canonical reader instead of ad hoc SQL strings.
- The view currently returns totals as plain integers without overflow checks; this can surface as runtime errors in later claim evaluation and must be handled in slice 004.
- The OS states that work items with no constraints are admissible. The current schema change does not change behavior, but no tests in this slice check explicit zero-constraint claim eligibility.

What I should understand before moving on
- The schema now guarantees persistence invariants for constraints even though claim-time evaluation is still outside scope.
- `Constraint` admission policy is intentionally left to future slices; this slice only adds model/schema/state evidence.
- The new operator enum exists, but no operator-evaluation function exists yet.
