## Execution-Order Summary

1. `cmd/controller/main.go` status assembly now reads queued work, running work, active runs, and queued constraint checks before any per-item derivation.
2. For each queued check row in `ListQueuedResourceConstraintChecks`, controller builds an in-memory map of resource checks per queued work item.
3. During queued work traversal, each constrained queued item is evaluated through `resourceConstraintChecksAllow`, incrementing either `QueuedResourceEligibleCount` or `QueuedResourceBlockedCount`. Blocked items additionally contribute to a per-resource blocked candidate map.
4. Controller traverses running work and increments `RunningResourceClaimCount` for each running item that has resolved work-item resource constraints.
5. If there are queued constrained items or running claims, blocked-per-resource summaries are materialized and sorted into `ResourceConstraintSummaries`, clipped to `maxResourceConstraintSummaries = 5`.
6. `internal/model/work_item.go` now carries the new status shape fields and structured per-resource summary type.
7. `cmd/controller/main_test.go` and `internal/model/work_item_test.go` add coverage for status JSON shape and count behavior.
8. `cmd/demo-client/main.go` renders status in compact human-readable form when resource fields are present, with tests in `cmd/demo-client/main_test.go`.
9. `docs/concepts/resource-constrained-work-admission/006-resource-status-and-observability.md` is marked `Implemented` and `PROJECT_STATE.md` records this slice as complete.

## Concrete Trace

- Submit work `resource-running` with constraint `memory-mib <= 1`.
- Submit work `resource-blocked` with the same constraint.
- Call `/work/next` once so one work item enters `running_work`.
- Call `/status`.
- Status path includes `queued` and `running` snapshots, then evaluates each constrained queued item:
  - `resource-running` now running so `resource-blocked` is blocked by `total_units` check.
- Returned status has `queued_resource_eligible_count=0`, `queued_resource_blocked_count=1`, `running_resource_claim_count=1`, and one summary entry for `target:local/memory-mib`.

## Edge Cases Covered by Tests

- JSON serialization of new status fields and slices.
- Blocked-vs-eligible count behavior with one constrained running and one constrained queued item.
- Human output formatting with multi-line resource summary.

## Edge Cases Not Covered

- No test for scenario where queued checks are malformed and evaluator returns error.
- No performance stress test for large queued/running sets and per-item store lookups.
- No test for mixed constrained/unconstrained queued items where all constrained are blocked and unconstrained are still counted as generic pending.
- No test for negative/zero resource units in status path.

## Ambiguity / Risk / Hidden Coupling

- `RunningResourceClaimCount` is implemented as count of constrained running work items, not sum of constrained units. If the intended semantic is total claimed units, this is a mismatch.
- Status computation is O(queue + constrainedRows + runningRows + per-row store calls); for large workloads this is potentially expensive.
- Summary `total_units` is derived from queued check rows only (`check.TotalUnits`), not from active running rows for the same key, so it reflects currently observed queued check snapshots not necessarily current global utilization per key.
- The `resourceConstraintChecksAllow` helper is deterministic only as far as current evaluator semantics; any change in evaluator behavior shifts status semantics immediately.

## What I should understand before moving on

OS 006 extends existing status output without changing claim behavior; resource visibility is derived from the same persisted constraint facts already used at claim time. The key contract is that status now distinguishes resource-ready vs resource-blocked queued work and reports a bounded “pressure” view for top-blocked resources.

## Questions I should answer

1. Should `running_resource_claim_count` represent number of constrained running work items or total constrained units requested by running work?
2. Is it acceptable that summary `total_units` is the queued row’s `total_units` snapshot rather than a recomputed running-capacity sum?
3. Do we need an OS guardrail for status-path cost if queue size grows large?
4. Should unconstrained queued work contribute to any new resource fields, or remain excluded by design?
5. Do we need a log-line counterpart for transitions (e.g., blocked to eligible) to support non-client users of `/status`?
