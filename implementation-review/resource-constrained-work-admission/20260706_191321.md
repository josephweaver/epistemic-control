Operational Slice: `docs/concepts/resource-constrained-work-admission/003-persist-constraints-with-work-items.md`

Diff basis: uncommitted implementation diff on the concept branch against current `HEAD`

## Implementation In Execution Order

1. Work creation still resolves constraints before persistence.
   - Raw `/work` wrappers are decoded by `decodeRawWorkSubmission` in `cmd/controller/main.go`.
   - Workflow stages are compiled by existing workflow compilation paths and converted by `persistenceRecordsFromCompiledStageResults` in `cmd/controller/workflow_stage_queue.go`.

2. Controller admission converts resolved model constraints to persistence records.
   - `persistenceResourceConstraintRecord` maps `model.WorkItemResourceConstraint` to `persistence.WorkItemResourceConstraintRecord`.
   - It preserves `work_item_id`, `constraint_index`, `resource_key`, `requested_units`, `operator`, `target_units`, and `created_at`.

3. Queueable work is inserted through one store transaction.
   - `Store.QueueWorkItems` inserts `work_items`, then `work_item_resource_constraints`, then `queued_work` in one transaction.
   - This matches the OS target insertion order.

4. Raw submission uses the new transaction.
   - `submitRawWorkToStore` now accepts resolved constraints and calls `QueueWorkItems`.

5. Initial workflow stage submission uses the new transaction.
   - `submitWorkflowRunToStore` now keeps the constraint return from `persistenceRecordsFromCompiledStageResults` and calls `QueueWorkItems`.

6. Just-in-time downstream activation uses the new transaction.
   - `activateNextReadyWorkflowStage` now calls `QueueWorkItems` with activated stage work, constraints, and queue rows.

7. The older stage-completion publication request was extended.
   - `CompleteStageRequest` now has `ReadyResourceConstraints`, and `CompleteStageIfReady` inserts them between ready work items and ready queue rows in its existing transaction.

## Changed API And Function Review

### `persistence.QueueWorkItemsRequest`

What it does: groups `[]WorkItemRecord`, `[]WorkItemResourceConstraintRecord`, and `[]QueuedWorkRecord` for one transaction.

Why it exists: OS 003 notes suggest adding a request type if `InsertWorkItems` only accepts `[]WorkItemRecord`; that was the existing state.

Assumption: all queueable work creation paths can provide complete persisted work records, optional resolved constraints, and queue rows together.

If wrong: a future path that creates work without immediate queueing should keep using `InsertWorkItems` or get a separate method; forcing it through `QueueWorkItems` would be semantically wrong.

Status against OS: fully implemented for queueable raw, initial-stage, and activated-stage work insertion.

Concrete trace: raw wrapper constraint for `test-001` becomes a `QueueWorkItemsRequest` with one `WorkItemRecord`, one `WorkItemResourceConstraintRecord`, and one `QueuedWorkRecord`.

Covered tests: `TestStoreQueueWorkItemsPersistsResourceConstraints`, raw controller wrapper test, initial workflow stage test, activation test.

Not covered: mixed batch where one constrained work item and one zero-constraint work item are queued together.

Risk: the method name says "Queue" rather than "InsertAndQueue"; callers need to understand that it also inserts work item rows.

### `Store.QueueWorkItems`

What it does: opens a transaction, inserts work items, inserts constraints, enqueues work, commits.

Why it exists: OS 003 requires `work_items`, `work_item_resource_constraints`, and `queued_work` to mutate atomically.

Assumption: idempotency should match existing work insertion behavior: identical existing rows are accepted, conflicting rows fail.

If wrong: retries after a partial prior failure could surface conflicts instead of completing. The current transaction shape should avoid partial prior failures for these three tables.

Status against OS: fully implemented for the associated work/constraint/queue mutation. It does not change claim behavior, as required by out-of-scope.

Concrete trace: duplicate `resource_key` constraints pass Go validation, first constraint insert succeeds, second hits SQLite uniqueness, transaction rollback removes the already-inserted work item.

Covered tests: rollback test asserts no work item, no queued row, no constraint rows after constraint insert failure.

Not covered: conflict where an identical work item exists but no queued row exists before retry.

Risk: `QueueWorkItemsRequest.validate` does not verify every constraint work-item id is present in `WorkItems`; the foreign key handles missing work items, but a clearer preflight check could improve diagnostics in a later slice.

### `Store.ListWorkItemResourceConstraints`

What it does: reads persisted constraints for one work item ordered by `constraint_index`.

Why it exists: tests and later admission logic need a focused way to inspect persisted resource facts without accessing `Store.db`.

Assumption: reading by work item is sufficient for this slice; resource-aware claim in later slices will likely need the existing view reader instead.

If wrong: later claim code should add a view-specific reader rather than overloading this method.

Status against OS: supporting API, not directly specified, but it reads only resolved facts from the OS table.

Covered tests: controller and persistence tests use it to prove raw, initial-stage, and activated-stage constraints persisted.

Not covered: missing work item id returns an empty list; only empty input validation is tested indirectly if future callers add coverage.

Risk: public store API surface grew for testability; it is still narrow and table-aligned.

### `CompleteStageRequest.ReadyResourceConstraints`

What it does: lets the existing `CompleteStageIfReady` transaction insert constraints for ready work publication.

Why it exists: OS 003 includes a stage-completion transaction shape. This store method already owned that shape for older ready-work publication.

Assumption: callers that use `ReadyWorkItems` can provide matching `ReadyResourceConstraints` when needed.

If wrong: stage completion could publish queued work without constraints through this older API. The current live dependency activation path does not use this method.

Status against OS: implemented for this persistence path.

Covered tests: `TestStoreCompleteStageIfReadyPublishesReadyWork` now asserts the ready work constraint is persisted.

Not covered: rollback from invalid `ReadyResourceConstraints` inside `CompleteStageIfReady`.

Risk: live dependency activation still records membership and marks stage readiness outside the `QueueWorkItems` transaction. That is existing architecture, but it differs from the OS pseudocode that shows predecessor completion and activation inserts in one transaction.

### `persistenceRecordsFromCompiledStageResults`

What it does: now returns persistence constraint records rather than model constraints.

Why it exists: its output is consumed directly by persistence calls, so returning the storage type avoids conversion at each workflow caller.

Assumption: the helper is the controller boundary between compiled workflow facts and persisted work/queue facts.

If wrong: if another caller needs model constraints after this helper, it must use the compiled stage result directly.

Status against OS: fully implemented for initial and activated stage persistence inputs.

Covered tests: existing stamp test still checks work item id and created timestamp on returned constraints.

Not covered: all operator variants at this boundary; operator validation belongs to slice 002/model tests.

Risk: raw submissions use the same converter helper even though they do not pass through `persistenceRecordsFromCompiledStageResults`; this is acceptable because all files are in package `main`.

### `submitRawWorkToStore`

What it does: accepts resolved constraints and queues raw work through `Store.QueueWorkItems`.

Why it exists: OS 003 acceptance criteria require raw wrapper constraints to persist when slice 002 support exists.

Assumption: raw persisted id equals `model.WorkItem.ID`; `persistenceRecordsFromRawWorkItem` confirms this.

If wrong: constraints would point at a non-existent or wrong work item id. The raw test now checks the persisted constraint work item id equals the queued row id.

Status against OS: fully implemented for raw submissions with and without constraints.

Covered tests: raw no-constraint existing test still passes; raw wrapper constraint test asserts stored values.

Not covered: raw duplicate resource key persistence failure mapped to HTTP response. Current invalid raw values are rejected before queue mutation.

Risk: persistence failures from `QueueWorkItems` still return generic `persist work item` HTTP 500 unless detected as conflict; this keeps existing behavior.

### `submitWorkflowRunToStore`

What it does: queues initial-stage work items and constraints through `Store.QueueWorkItems`.

Why it exists: OS 003 acceptance criteria require initial workflow stage submission to persist constraints for compiled constrained items.

Assumption: `persistenceRecordsFromCompiledStageResults` produces one constraint record with the run-scoped work item id for every compiled constraint.

If wrong: claim-time resource checks in later slices would find no facts or facts attached to non-queued ids.

Status against OS: fully implemented for initial stage work items.

Covered tests: `TestSubmitWorkflowHandlerQueuesOnlyInitialSequentialStage` now declares a memory constraint and asserts both initial queued items have it.

Not covered: initial stage with a mix of constrained and unconstrained steps.

Risk: dependency membership recording still happens after queue insertion, as before. OS 003 explicitly says constraints are not dependency-state membership, so the separation is conceptually correct.

### `activateNextReadyWorkflowStage`

What it does: queues newly activated downstream work items and constraints through `Store.QueueWorkItems`.

Why it exists: OS 003 acceptance criteria require just-in-time downstream activation to persist constraints for newly activated work.

Assumption: activated stage compilation has already resolved downstream expressions and resource constraints before persistence.

If wrong: stage activation failure should mark the workflow failed through `failWorkflowStageActivation`, preserving existing failure style.

Status against OS: implemented for resource constraint persistence and queue rollback. Implemented differently from pseudocode because predecessor terminal completion, dependency membership, stage ready state, and queue mutation are not all one database transaction in the live dependency-state architecture.

Covered tests: `TestCompleteWorkHandlerActivatesNextSequentialStage` now gives the second stage a constraint and asserts the activated queued item has that persisted constraint.

Not covered: forced constraint insert failure during live activation. A malformed duplicate resource key would currently be rejected during compilation before persistence, so a DB-level activation rollback test would need a lower-level fixture or fault injection.

Risk: if `QueueWorkItems` succeeds and later membership recording fails, queued work can exist without dependency membership. This risk predates this slice for work items and is not expanded to constraints, but it remains a transactional boundary to revisit if later slices tighten activation atomicity.

## Acceptance Criteria Mapping

- Raw work submission without constraints still persists and queues as before: covered by existing raw no-constraint test and full suite.
- Raw work submission with constraints persists constraints: covered by raw wrapper test.
- Initial workflow stage submission persists constraints: covered by initial sequential-stage test.
- Just-in-time downstream stage activation persists constraints: covered by activation test.
- Constraint insert failures roll back associated work-item/queue mutations: covered by `QueueWorkItems` rollback test.
- Work items with zero constraints are still accepted: covered by existing no-constraint tests and full suite.
- Constraints are not stored in completed/dependency output JSON/evidence: implementation stores constraints only in `work_item_resource_constraints`; tests exercise output paths without adding constraint data to payload structs.
- Existing dependency-aware workflow tests still pass: covered by `go test ./cmd/controller` and `go test ./...`.

## What I Should Understand Before Moving On

- Resource constraints are persisted facts attached to work item ids, not dependency membership state.
- `QueueWorkItems` is now the transaction boundary for queueable work creation.
- Existing `InsertWorkItems` and `EnqueueWorkItems` still exist for older tests and non-queue-bundled operations.
- Claim behavior is intentionally unchanged in this slice; queued constrained work can still be claimed without resource evaluation until later slices.
- Live activation atomicity is narrower than the OS pseudocode: work, constraints, and queue rows are atomic together, but dependency membership and stage-ready transitions remain separate.

## Questions I Should Answer

- Should a later cleanup slice make activation membership, stage readiness, work insertion, constraints, and queueing one store transaction?
- Should `QueueWorkItemsRequest.validate` reject constraints whose `WorkItemID` is not in the request's `WorkItems` before SQLite does?
- Should existing callers of separate `InsertWorkItems` plus `EnqueueWorkItems` be migrated opportunistically only when they create queueable work?
- Should `ListWorkItemResourceConstraints` remain public after the later view-reader slice exists?
- Should controller HTTP errors distinguish constraint persistence failures from generic work persistence failures?

## Recommended Next Test Or Tiny Experiment

Add a fault-injection or lower-level controller test that simulates failure after `QueueWorkItems` succeeds but before dependency membership is recorded, then decide whether that existing activation boundary needs a later transactional store API.
